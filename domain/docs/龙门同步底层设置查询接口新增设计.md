### 1.2 关键发现：`m_events.push_back()` 的执行者与消费者

| 步骤 | 位置 | 动作 |
|------|------|------|
| **①** | `GantrySystem::requestCoupling()` | 联动条件满足后，将 `Event::coupled()` 推入内部 `m_events` 缓冲区 |
| **②** | `GantrySystem::drainEvents()` | 返回并清空内部 `m_events` 缓冲区（移动语义） |
| **③** | **调用方**（当前为 `doScanCycle()` 或测试代码） | 消费事件。**目前尚未实现自动发布到 `IGantryEventBus`** |

**结论**：`m_events.push_back(Event::coupled())` 本身**不触发任何执行**。它只是将事件放入 GantrySystem 的内部事件队列。事件的真正消费方是在 `drainEvents()` 被调用之后，由上层代码（扫描循环 / Orchestrator）决定如何处理这些事件——发给 EventBus、更新 UI、写日志等。

### 1.3 使能的当前实现路径

当前使能逻辑完全由 PLC 层的 `AxisState` 模拟：

```
FakePLC.forceState(AxisId::X1, AxisState::Idle)     → 表示 X1 已使能
FakeGantryFeedbackPort → toPhysicalAxisState()       → state.enabled = true
GantrySystem.syncState()                              → PhysicalAxis.m_enabled = true
GantryCouplingService.validateConditions()            → snap.x1Enabled = true
```

**当前没有独立于 AxisState 的"使能命令"端口**。使能是 PLC 硬件层的概念，目前通过 `forceState` 直接注入模拟。这是一个重要的架构缺口：**缺少一个通过端口向 PLC 下发使能命令的标准路径**。

## 2. 新增需求分析

### 2.1 需求概述

> 你已在 `IGantryCommandPort` 中新增了 `setLogicalAxisEnable()` 和 `setCouplingRegister()` 两个方法。

两个新需求：

#### 需求 A：轴X使能寄存器（写入命令）

- **寄存器类型**：PLC 单一控制寄存器
- **写入 ON 的效果**：PLC 底层自动让 X1 和 X2 同时上电（使能）
- **语义**：这是"龙门逻辑轴 X"级别的使能，而非分别使能 X1 / X2

#### 需求 B：轴X联动使能寄存器 + 轴X联动状态寄存器

**轴X联动使能**（写入命令）：
- **寄存器类型**：PLC 单一控制寄存器
- **写入 ON 后**：PLC 内部自动汇总条件，满足后才真正进入联动
- **条件**（在 PLC 内部判断）：
  1. X1 和 X2 电机当前速度都为 0（`轴X1CurSpeed = 0.0` 且 `轴X2CurSpeed = 0.0`）
  2. X1 和 X2 伺服均已使能（`轴X1Status AND 轴X2Status`）
  3. 收到 `轴X联动使能` 指令
  4. 两轴当前位置绝对值之和 < 0.1（`ABS(轴X1CurPosition + 轴X2CurPosition) < 0.1`）

**轴X联动状态**（查询状态寄存器）：
- **寄存器类型**：PLC 状态寄存器（只读）
- **ON**：上述所有 4 条条件同时满足
- **OFF**：收到解除联动指令（`NOT 轴X联动使能`）时

### 2.2 需求与现有架构的映射分析

```
需求 A（轴X使能寄存器）   ← 对应 PLC 硬件层控制寄存器
                           ← 写入路径：IGantryCommandPort::setLogicalAxisEnable()
                           ← 效果：PLC 内部 X1/X2 同时上电
                           
需求 B（轴X联动使能寄存器） ← 对应 PLC 硬件层控制寄存器
                           ← 写入路径：IGantryCommandPort::setCouplingRegister()
                           ← 效果：PLC 内部自动判断 4 条条件后决定联动状态
                           
需求 B（轴X联动状态寄存器） ← 对应 PLC 硬件层状态寄存器（只读）
                           ← 查询路径：需要新建查询接口
                           ← 效果：返回 PLC 内部逻辑的联动状态（ON/OFF）
```

### 2.3 与领域层 GantrySystem 的关系

这是关键设计决策点：

```
┌─────────────────────────────────────────────────────────────────┐
│                      职责分层                                    │
│                                                                  │
│  PLC 内部逻辑（硬件/固件层）                                       │
│  ├─ 轴X使能寄存器：写入 → X1/X2 同时上电                           │
│  ├─ 轴X联动使能寄存器：写入 → 内部条件判断                          │
│  └─ 轴X联动状态寄存器：条件满足 → ON，不满足 → OFF                 │
│                                                                  │
│  ── 分界线：IGantryCommandPort / IGantryFeedbackPort ──          │
│                                                                  │
│  领域层 GantrySystem                                              │
│  ├─ requestCoupling()：领域层语义的"联动申请"                       │
│  │   └─ 调用 m_commandPort->setCouplingRegister(true)            │
│  │   └─ 然后从 FeedbackPort 读取 轴X联动状态                       │
│  ├─ aggregateState()：聚合 X1/X2 状态为逻辑轴状态                  │
│  └─ m_events：领域事件缓冲                                        │
└─────────────────────────────────────────────────────────────────┘
```

**核心设计原则**：PLC 的"轴X联动状态"寄存器是**硬件层的实际联动状态**，它和领域层的 `GantryMode::Coupled` 之间的关系是：

- 领域层调用 `setCouplingRegister(true)` → PLC 内部判断 → 查询"轴X联动状态" → 如果为 ON，领域层才将 `m_mode` 设为 `Coupled`

这相当于联动建立从"领域层纯软件判断"变为"领域层向 PLC 发出请求 + 查询 PLC 实际结果"的模式。

## 4. 新增接口详细设计

### 4.1 IGantryCommandPort 新增方法（已由你完成）

```cpp
class IGantryCommandPort {
public:
    // ... 现有方法 ...

    /**
     * @brief 设置逻辑轴使能状态
     *
     * 写入 PLC 的"轴X使能"控制寄存器。
     * 写入 ON 后，PLC 底层自动让 X1 和 X2 同时上电。
     *
     * @param enable true = 使能, false = 禁用
     * @return true = 命令已写入 PLC 寄存器

    /**
     * @brief 设置耦合寄存器状态（轴X联动使能）
     *
     * 写入 PLC 的"轴X联动使能"控制寄存器。
     * 写入 ON 后，PLC 内部自动汇总以下条件：
     *   - X1/X2 速度均为 0
     *   - X1/X2 均已使能
     *   - |X1.pos + X2.pos| < 0.1
     * 全部满足后，"轴X联动状态"寄存器变为 ON。
     *
     * @param enable true = 请求联动, false = 解除联动
     * @return true = 命令已写入 PLC 寄存器
     */
    virtual bool setCouplingRegister(bool enable) = 0;
};
```

### 4.4 FakePLC 寄存器模型设计

```cpp
class FakePLC {
public:
    // ═══ 新增：龙门控制寄存器 ═══

    /// 轴X使能 控制寄存器
    /// 写入 ON → X1 和 X2 同时进入使能状态
    void writeAxisXEnable(bool on) {
        m_axisXEnableRegister = on;
        if (on) {
            // PLC 底层自动让 X1 和 X2 同时上电
            if (m_axisState[(int)AxisId::X1] != AxisState::Error) {
                m_axisState[(int)AxisId::X1] = AxisState::Idle;
            }
            if (m_axisState[(int)AxisId::X2] != AxisState::Error) {
                m_axisState[(int)AxisId::X2] = AxisState::Idle;
            }
        }
    }

    bool readAxisXEnableRegister() const {
        return m_axisXEnableRegister;
    }

    /// 轴X联动使能 控制寄存器
    /// 写入 ON → PLC 内部自动汇总条件
    void writeCouplingRegister(bool on) {
        m_couplingEnableRegister = on;
        // 联动状态在 tick() 中更新
    }

    bool readCouplingEnableRegister() const {
        return m_couplingEnableRegister;
    }

    /// 轴X联动状态 状态寄存器（只读）
    /// 由 tick() 内部自动更新
    bool readCouplingStatus() const {
        return m_couplingStatusRegister;
    }

    /// 设置轴当前速度（用于模拟联动条件判断）
    void setSpeed(AxisId id, double speed) {
        m_axisSpeed[(int)id] = speed;
    }

    double getSpeed(AxisId id) const {
        return m_axisSpeed[(int)id];
    }

private:
    bool m_axisXEnableRegister = false;
    bool m_couplingEnableRegister = false;
    bool m_couplingStatusRegister = false;
    double m_axisSpeed[4] = {0.0, 0.0, 0.0, 0.0};  // 索引: AxisId 枚举值

    /// 更新联动状态寄存器（在 tick() 中调用）
    void updateCouplingStatus() {
        if (!m_couplingEnableRegister) {
            m_couplingStatusRegister = false;
            return;
        }

        // 条件1: X1 和 X2 速度均为 0
        bool speedZero = (std::abs(m_axisSpeed[(int)AxisId::X1]) < 1e-9) &&
                         (std::abs(m_axisSpeed[(int)AxisId::X2]) < 1e-9);

        // 条件2: X1 和 X2 均已使能
        bool bothEnabled = (m_axisState[(int)AxisId::X1] != AxisState::Disabled &&
                            m_axisState[(int)AxisId::X1] != AxisState::Unknown &&
                            m_axisState[(int)AxisId::X1] != AxisState::Error) &&
                           (m_axisState[(int)AxisId::X2] != AxisState::Disabled &&
                            m_axisState[(int)AxisId::X2] != AxisState::Unknown &&
                            m_axisState[(int)AxisId::X2] != AxisState::Error);

        // 条件4: 位置偏差 < 0.1
        bool positionOk = std::abs(m_absPos[(int)AxisId::X1] +
                                   m_absPos[(int)AxisId::X2]) < 0.1;

        // 条件3: couplingEnableRegister == true（已在上面检查）

        m_couplingStatusRegister = speedZero && bothEnabled && positionOk;
    }

    // tick() 中调用:
    void tick(int ms) {
        // ... 现有限位检查 ...
        updateCouplingStatus();
    }
};
```

## 5. 数据流设计：从 PLC 寄存器到 GantrySystem 的闭环

### 5.1 使能流程

```
用户操作（UI/Policy）
  │
  ▼
GantrySystem::enableLogicalAxis()                  ◄── 领域层新增方法
  │
  ├─ m_commandPort->setLogicalAxisEnable(true)     ← 写入 PLC "轴X使能" 寄存器
  │
  ▼
[PLC 硬件层]
  "轴X使能" 寄存器 := ON
  → 自动让 X1 和 X2 同时上电
  → X1.state → Idle, X2.state → Idle
  │
  │ (下一个扫描周期)
  ▼
FakeGantryFeedbackPort::getX1Feedback()  → enabled = true
FakeGantryFeedbackPort::getX2Feedback()  → enabled = true
  │
  ▼
GantrySystem::syncState(fb)
  → PhysicalAxis X1.m_enabled = true
  → PhysicalAxis X2.m_enabled = true
  → (可选) 发出 Enabled 事件
```

### 5.2 联动建立流程（改动后的关键流程）

```
用户操作 / Policy 层
  │
  ▼
GantrySystem::requestCoupling()                     ◄── 需要调整逻辑
  │
  │  [Step 1] 写入联动使能寄存器
  ├─ m_commandPort->setCouplingRegister(true)
  │
  │  [Step 2] 返回"申请已发出"（不等硬件确认）
  └─ return CouplingCondition::Result { .allowed = true, .reason = "coupling request sent" }
       
       ⚠ 注意：此时 PLC 尚未完成条件判断，GantryMode 不能立即变为 Coupled！
               需要等下一个扫描周期，从 FeedbackPort 读取"轴X联动状态"后再变更模式。

──────────────── 下一个扫描周期 ────────────────

doScanCycle():
  │
  │ Step A: 读取反馈（含 轴X联动状态）
  ├─ PhysicalAxisState fb1 = feedback.getX1Feedback()
  ├─ PhysicalAxisState fb2 = feedback.getX2Feedback()
  ├─ bool couplingStatus = feedback.isCouplingStatusActive()   ◄── 新增
  ├─ double x1Speed = feedback.getX1CurSpeed()                 ◄── 新增
  ├─ double x2Speed = feedback.getX2CurSpeed()                 ◄── 新增
  │
  │ Step B: 同步到 GantrySystem
  ├─ gantry.x1().syncState(fb1)
  ├─ gantry.x2().syncState(fb2)
  ├─ gantry.x1().setSpeed(x1Speed)                              ◄── 新增
  ├─ gantry.x2().setSpeed(x2Speed)                              ◄── 新增
  │
  │ Step C: 状态聚合 + 联动状态检查
  └─ gantry.aggregateState()
       │
       │  内部逻辑：
       │  if (m_pendingCouplingRequest && couplingStatus) {
       │      m_mode = GantryMode::Coupled;
       │      m_events.push_back(Event::coupled());     ◄── ① 事件在此处产生
       │      m_pendingCouplingRequest = false;
       │  }
       │  else if (!couplingStatus && m_mode == GantryMode::Coupled) {
       │      m_mode = GantryMode::Decoupled;
       │      m_events.push_back(Event::decoupled());
       │      m_events.push_back(Event::deviationFault(...));
       │  }
       │
  │ Step D: 排空事件 → 发布到 EventBus
  └─ auto events = gantry.drainEvents();               ◄── ②
       for (auto& e : events) {
           eventBus.publish(e);                          ◄── ③ → UI 更新
       }
```

### 5.3 关键设计决定：GantrySystem 新增状态变量

bool m_pendingCouplingRequest = false;   // 是否已向 PLC 发出了联动请求但尚未确认
bool m_couplingStatusFromPLC = false;    // 上次读取的 PLC 联动状态寄存器值

### 5.4 requestCoupling() 改动后的逻辑

```cpp
// 改动前（纯领域层判断）：
CouplingCondition::Result requestCoupling() {
    auto cond = CouplingCondition::checkAll(x1(), x2());
    if (cond.allowed) {
        m_mode = GantryMode::Coupled;           // 立即变更模式
        m_events.push_back(Event::coupled());
    }
    return cond;
}

// 改动后（领域层发出请求 + 等待 PLC 确认）：
CouplingCondition::Result requestCoupling() {
    // Step 1: 领域层前置条件检查（使能、无报警、无限位）
    auto cond = CouplingCondition::checkAll(x1(), x2());
    if (!cond.allowed) {
        return cond;  // 前置条件不满足，直接拒绝
    }

    // Step 2: 通过 CommandPort 写入 PLC 联动使能寄存器
    bool cmdOk = m_commandPort->setCouplingRegister(true);
    if (!cmdOk) {
        return CouplingCondition::Result::rejected("command port write failed");
    }

    // Step 3: 标记"等待 PLC 确认"
    m_pendingCouplingRequest = true;

    // Step 4: 返回"请求已受理"
    // 注意：此时 m_mode 尚未变更为 Coupled！
    // 模式变更将在下一个扫描周期 aggregateState() 中
    // 根据 feedback.isCouplingStatusActive() 的返回值进行。
    return CouplingCondition::Result::allowed("coupling request sent, pending PLC confirmation");
}
```

### 5.5 联动状态变更事件触发点变更

| 事件 | 旧触发点 | 新触发点 |
|------|----------|----------|
| `Event::coupled()` | `requestCoupling()` 内部，条件满足后立即触发 | `aggregateState()` 内部，读到 `isCouplingStatusActive() == true` 时触发 |
| `Event::decoupled()` | `requestDecoupling()` 内部 | `aggregateState()` 内部，读到 `isCouplingStatusActive() == false` 时触发（从 Coupled 变为非 Coupled） |
| `Event::deviationFault()` | `checkSyncMaintenance()` 内部 | `aggregateState()` 内部，读到 `isCouplingStatusActive() == false` 且 `m_mode` 原来是 Coupled 时触发 |

### 6.2 向后兼容性

- **GantryCouplingService** 的外部接口**不需要改动**。它在内部调用 `GantrySystem::requestCoupling()`，而该方法的返回值语义不变（返回 `SafetyCheckResult`）。
- **IGantryEventBus** **不需要改动**。事件类型不变，只是触发时机从 `requestCoupling()` 内部移到 `aggregateState()` 内部。
- **现有测试** 大部分**不需要修改**。它们通过 `FakePLC::forceState()` 直接设置状态，不经过命令端口路径。只有直接测试 `requestCoupling()` 内部行为的测试可能需要更新断言时机。

### 6.3 风险评估

| 风险 | 级别 | 缓解措施 |
|------|------|----------|
| 联动建立从"同步"变"异步"（需等一个扫描周期） | 中 | 文档中明确说明；在 UI 中增加"联动建立中"中间状态 |
| 现有测试可能因逻辑变更而失败 | 中 | 分阶段修改：先扩展接口（保持旧逻辑），再逐步迁移测试断言 |
| GantrySystem 新增 `m_commandPort` 依赖 | 低 | 遵循现有的依赖注入模式（构造函数或 setter 注入） |
| 事件触发时机从 `requestCoupling()` 移到 `aggregateState()` 后 UI 响应延迟 | 低 | 事件仍在上一个 `drainEvents()` 中排空，UI 帧率内无感知延迟 |

### 6.4 设计原则总结

```
┌────────────────────────────────────────────────────────────────┐
│  核心设计决策：                                                  │
│                                                                 │
│  1. PLC 寄存器是"硬件事实源"（Source of Truth）                   │
│     → Domain 层的 m_mode 应镜像 PLC 的轴X联动状态寄存器            │
│                                                                 │
│  2. 命令端口 (IGantryCommandPort) 只负责"写入寄存器"               │
│     → 不负责判断业务条件                                          │
│                                                                 │
│  3. 反馈端口 (IGantryFeedbackPort) 只负责"读取寄存器原始值"        │
│     → 不负责业务解释                                             │
│                                                                 │
│  4. GantrySystem 负责"领域语义翻译"                               │
│     → 将 PLC 寄存器的 ON/OFF 翻译为 GantryMode::Coupled/Decoupled│
│     → 将寄存器的条件不满足翻译为 domain 事件                       │
│                                                                 │
│  5. 前置条件检查仍在 domain 层完成                                 │
│     → 避免向 PLC 发出明显非法的联动请求                            │
│     → 但最终裁决权在 PLC 的轴X联动状态寄存器                        │
└────────────────────────────────────────────────────────────────┘
```

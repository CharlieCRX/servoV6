# 龙门同步系统业务语义约束分析

> 版本: v1.0
> 分析方法: 从测试行为反推现实业务约束
> 分析日期: 2026-05-07

---

## 前言 — 为什么要从测试行为反向分析

传统上我们习惯于读代码实现来理解系统。但代码实现是"一种可能的解法"，而测试行为是"系统对外承诺的契约"。测试告诉你系统"必须做什么"以及"绝对不能做什么"。

本分析严格遵循以下逻辑链：

```
现实机械问题
→ 业务约束（必须如此的原因）
→ 系统行为（约束在代码中的体现）
→ 测试用例（约束被验证的方式）
→ 领域抽象（为什么存在这些实体、值对象、端口）
```

**最终目标**：让一个不懂当前代码实现的人，通过测试行为理解：
1. 龙门系统为什么这样设计
2. 系统到底在保护什么
3. 哪些约束最核心，为什么
4. 为什么这些抽象存在

# 第一部分：从每个测试识别业务约束

## A. 文件：test_gantry_mode.cpp

### 测试：`ShouldHaveTwoModesOnly`

**系统行为**：系统中只存在两种模式——`Coupled` 和 `Decoupled`，且两者互斥。

**背后的现实机械问题**：
龙门由两个电机拖动同一根横梁。要么两个电机同步运动（一个整体），要么各自独立运动（两个单独滑台）。不存在"半联动"的物理状态。

**它保护的业务约束**：
龙门的工作模式是二元互斥的——任何时候只能处于联动或分动中的一种。

**如果没有这个约束会怎样**：
操作者可能认为系统处于联动状态，实际物理轴已经分动，导致横梁扭曲或碰撞。

**约束属于**：Domain Business Rule


### 测试：`CoupledMode_RepresentsCoupledSemantics` / `DecoupledMode_RepresentsDecoupledSemantics`

**系统行为**：`isCoupled()` 和 `isDecoupled()` 互为否定。

**背后的现实机械问题**：
在物理上，"两个轴联合运动"与"两个轴各自运动"是互斥状态。它们之间存在一条明确的界限。

**它保护的业务约束**：
模式的合法性：`isCoupled(mode) XOR isDecoupled(mode)` 永远成立。

**如果没有这个约束会怎样**：
如果出现一种既不是联动也不是分动的"中间态"，所有上层判断逻辑都会失效。

**约束属于**：Domain Business Rule


### 测试：`MutuallyExclusive_CoupledVsDecoupled`

**系统行为**：`areMutuallyExclusive(Coupled, Decoupled)` 返回 true。

**背后的现实机械问题**：
一条物理横梁不可能既绑定在一起又分开。这不是一个渐变过程，而是一个状态切换。

**它保护的业务约束**：
状态空间的完备性：任何时刻系统恰好处在一种工作模式下。

**如果没有这个约束会怎样**：
可能导致模式冲突，一个命令被同时解释为"对整体操作"和"对单轴操作"。

**约束属于**：State Transition Constraint

## B. 文件：test_gantry_direction.cpp

### 测试：`ShouldHaveTwoValuesOnly` / `Forward_IsNotBackward`

**系统行为**：系统中只存在 `Forward` 和 `Backward` 两个方向值，它们互不相同。

**背后的现实机械问题**：
在龙门运动中，操作者只关心"朝我这边来"还是"离我远去"。至于电机是正转还是反转、编码器是加还是减，那是物理层的事，操作者不应该感知。

**它保护的业务约束**：
运动方向与物理实现解耦——方向是纯粹的语义概念（远离操作者/靠近操作者），不与电机正反转、CW/CCW等物理概念绑定。

**如果没有这个约束会怎样**：
每次更换电机类型、驱动器型号或物理接线，整个系统的"方向"概念都需要重新定义。操作者面对的逻辑不再统一。

**约束属于**：Domain Business Rule


### 测试：`Forward_RepresentsAwayFromOperator` / `Backward_RepresentsTowardOperator`

**系统行为**：`Forward = 远离操作者`，`Backward = 靠近操作者`。这是固定语义。

**背后的现实机械问题**：
工业现场需要一个与物理实现无关的、直观的方向定义。不管电机怎么接线，操作者只需要说"往前走"或"往后退"。

**它保护的业务约束**：
方向语义固定：`Forward` 永远是从操作者向外，`Backward` 永远是朝向操作者。这个映射在任何设备、任何配置下不变。

**如果没有这个约束会怎样**：
不同操作者、不同设备间理解不一致，一个按钮按下没人知道会往哪个方向运动。

**约束属于**：Domain Business Rule


### 测试：`Opposite_IsInvolution`

**系统行为**：`opposite(opposite(d)) == d` 成立（对合性）。

**背后的现实机械问题**：
"反向的反向就是正向"——这在物理世界是绝对真理。方向反转操作必须是可逆的。

**它保护的业务约束**：
方向反转操作的数学正确性。

**如果没有这个约束会怎样**：
多次反转后方向不确定，可能造成严重安全风险。

**约束属于**：Domain Business Rule

## C. 文件：test_gantry_sync.cpp

### 测试：`PerfectMirror_HasZeroDeviation` / `BothAtZero_HasZeroDeviation`

**系统行为**：偏差定义为 `|X1.pos + X2.pos|`。完美镜像时偏差为 0。双轴都在 0 点时偏差也为 0。

**背后的现实机械问题**：
在龙门设计中，两个电机沿同一方向推动横梁。X2 电机与 X1 电机安装方向相反，所以与 X1 同向运动时，X2 必须反向运动才能保持横梁平直。物理上这就是 X1 和 X2 的镜像关系：`X1 ≈ -X2`。

**为什么使用 |X1 + X2| <= epsilon 作为同步条件？**

因为龙门同步的本质要求是：**双轴位置关于原点对称**。

假设龙门横梁跨距为 2L，X1 在坐标 +L 处，X2 在坐标 -L 处，这样才能保持横梁垂直于运动方向。如果 X1 移动到 p 位置，X2 必须移动到 -p 位置。所以同步条件自然就是 `|X1 + X2| <= epsilon`。

**它保护的业务约束**：
同步偏差度量采用绝对值偏差公式 `|X1 + X2|`，而非 `|X1 - X2|`。这直接反映了 X2 镜像安装的物理事实。

**如果没有这个约束会怎样**：
如果用 `|X1 - X2|` 作为偏差（像普通双轴那样），系统会误判镜像安装的 X2 为"有巨大偏差"，导致系统永远无法进入联动状态。

**约束属于**：Hardware Safety Constraint


### 测试：`BothPositive_NotMirror_ReturnsLargeDeviation`

**系统行为**：当 `X1=100, X2=100` 时（同向位置），偏差为 200。

**背后的现实机械问题**：
如果两个轴都在正方向位置，说明 X2 没有按镜像安装规律运动——X2 应该处于负方向。这意味着要么测量错误，要么横梁已经物理扭曲。

**它保护的业务约束**：
偏差公式必须能够识别出"双轴同向"这种极端错误情况，并给出明确的大偏差信号。

**如果没有这个约束会怎样**：
当 X2 由于接线错误或编码器方向配置错误而"同向运动"时，偏差计算可能仍然很小，系统无法发现横梁实际已扭曲。

**约束属于**：Hardware Safety Constraint


### 测试：`Alarm_AllDirectionsBlocked` / `Alarm_OverridesLimit`

**系统行为**：报警状态下，所有方向的运动都被禁止。即使同时存在限位，报警也优先于限位。

**背后的现实机械问题**：
报警表示有明确的硬件故障——过流、过温、编码器断线、急停按下等。在故障原因查清并排除之前，任何运动指令都可能导致更严重的设备损坏或人身伤害。

**为什么报警优先级最高？**
因为报警代表"系统失效"，而限位代表"系统正常工作但到达边界"。一个已失效的系统不应该做任何动作，无论方向如何。

**它保护的业务约束**：
报警不可被覆盖：没有任何条件可以让系统在报警状态下运动。

**如果没有这个约束会怎样**：
如果操作者试图"反方向退出限位"时遇到报警仍能运动，故障中的电机可能突然动作，造成人员伤亡。

**约束属于**：Motion Safety Rule


### 测试：`ForwardLimit_BlocksForwardDirection` / `ForwardLimit_AllowsBackwardDirection`

**系统行为**：正限位触发时，禁止正向 Jog，但允许反向 Jog。

**背后的现实机械问题**：
限位开关（或软限位）的作用是告诉系统"你已经到达机械行程终点，不能再往这边走了"。但系统必须允许反方向运动——否则轴就被"困"在限位位置，永远无法脱离。

**为什么正限位仍允许反方向运动？**
限位的本质是"禁止继续向危险方向运动"，而不是"禁止运动"。轴到了正限位，应该能够反向退回到安全区域。这是工业安全的基本要求。

**它保护的业务约束**：
限位是方向性保护，不是全停。系统在限位状态下必须保持"可退"能力。

**如果没有这个约束会怎样**：
一旦触发限位，轴变成"被卡住"状态。操作者必须手动盘车或复位才能离开限位区，严重影响生产效率，且增加了人机交互风险。

**约束属于**：Motion Safety Rule


### 测试：`ForwardLimit_RejectedDueToForwardLimit`（工厂方法命名）

**系统行为**：安全检查区分 `Rejected_LimitForward` 和 `Rejected_LimitBackward`，而不是笼统的 `Rejected_Limit`。

**背后的现实机械问题**：
正限位和负限位是物理上不同方向的保护开关。笼统地说"限位触发了"不足以告诉操作者"是哪个方向触发了"。

**它保护的业务约束**：
限位方向的区分性：系统必须明确报告是正限位还是负限位，以便上层可以精准决定允许哪个方向运动。

**如果没有这个约束会怎样**：
上层只能判断"限位了"但不能精确控制"可以往哪个方向运动"，方向性保护逻辑无法实现。

**约束属于**：Hardware Safety Constraint


### 测试：`BothLimitsActive_BlocksForwardFirst` / `BothLimitsActive_BlocksBackwardToo`

**系统行为**：当双向限位同时触发时，两个方向都被拒绝。

**背后的现实机械问题**：
双向限位同时触发是一个非常异常的故障状态——机械行程两端都被触发了。这通常意味着传感器损坏、接线错误或编码器位置值严重漂移。

**它保护的业务约束**：
双向限位同时触发时，系统应拒绝所有方向运动。这不是正常使用场景，不应尝试"反向退出"。

**如果没有这个约束会怎样**：
系统可能尝试"反方向退出"但物理上不可能，导致持续震荡或执行器损坏。

**约束属于**：Motion Safety Rule


### 测试：`MotionSafetyMatrixTest_FullMatrix`

**系统行为**：使用组合测试矩阵遍历所有安全条件组合（Alarm × FwdLimit × BwdLimit × Direction = 2 × 2 × 2 × 2 = 16种组合）。

**背后的现实机械问题**：
安全条件不是独立存在的。现场可能同时出现报警和限位、双向限位、报警加双向限位等多种复杂情况。每种组合下每个方向应该怎么反应，必须有明确的、完整的定义。

**它保护的业务约束**：
安全检查的完备性：所有安全条件组合都要有明确的判定结果。不允许出现未定义的行为。

**如果没有这个约束会怎样**：
某些罕见的故障组合会导致系统行为不确定——可能允许本应禁止的运动，或者禁止本应允许的运动。

**约束属于**：Motion Safety Rule

## D. 文件：test_gantry_coupling.cpp

### 测试：`DecoupledByDefault`

**系统行为**：默认处于分动模式。联动必须显式申请。

**背后的现实机械问题**：
系统启动时，两个电机是独立使能、独立回零的。没有任何方法可以在启动时就知道两个轴的位置是否对齐。因此默认分动是安全的选择——需要操作者明确确认后才进入联动。

**它保护的业务约束**：
联动不是默认状态，需要经过条件检查才能进入。系统不应自动假设双轴可以同步。

**如果没有这个约束会怎样**：
系统启动时自动进入联动模式，如果 X1 和 X2 实际位置偏差过大，横梁在使能瞬间就会被扭曲或拉断。

**约束属于**：State Transition Constraint


### 测试：`CannotForceCouplingWithoutChecking`

**系统行为**：不能绕过条件检查直接设置联动模式。

**背后的现实机械问题**：
联动不是一个人为设定的状态切换，而是"条件满足后的状态认可"。如果允许强制设置联动模式，相当于允许操作者说"我不管物理轴现在什么状况，你联动起来"——这在机械上是危险的。

**它保护的业务约束**：
联动绑定准入条件：模式切换必须经过守卫条件校验。未通过校验的模式切换是被禁止的。

**如果没有这个约束会怎样**：
操作者或上层代码可以强制设置联动，即使 X1 和 X2 偏差极大。电机使能瞬间产生巨大应力，损坏机械结构。

**约束属于**：Aggregate Invariant


### 测试：`X1NotEnabled_Rejected` / `X2NotEnabled_Rejected`

**系统行为**：任何一轴未使能，联动申请被拒绝。

**背后的现实机械问题**：
两个电机必须都处于伺服就绪状态，才能共同承担横梁的同步运动。一个轴未使能，意味着它没有力矩输出，另一个轴运动会把横梁推向未使能的一侧，导致横梁歪斜甚至掉落。

**它保护的业务约束**：
联动建立条件之一：两个物理轴都必须处于使能（伺服就绪）状态。

**如果没有这个约束会怎样**：
未使能的轴被拖着走，可能造成机械碰撞或驱动器损坏。

**约束属于**：Hardware Safety Constraint


### 测试：`AlarmActive_Rejected` / `X2Alarm_Rejected`

**系统行为**：任何一轴有报警，联动申请被拒绝。

**背后的现实机械问题**：
报警意味着该轴处于故障状态。一个故障的轴不应该参与联动——它的编码器数据可能不可靠，电机力矩可能缺失。

**它保护的业务约束**：
联动建立条件之二：两个物理轴都无报警。

**如果没有这个约束会怎样**：
故障轴参与联动，偏差检测"发现偏差"后立即触发 DeviationFault 强制解联，造成震荡——联动建立失败的成本很高。

**约束属于**：Motion Safety Rule


### 测试：`LimitTriggered_Rejected` / `NegativeLimit_Rejected`

**系统行为**：任何一轴触发限位，联动申请被拒绝。

**背后的现实机械问题**：
如果某个物理轴已经处于限位边界，那么联动后的同步运动可能立即触发该限位，或者没有足够的行程空间完成同步调整。

**它保护的业务约束**：
联动建立条件之三：两个物理轴都未触发限位。

**如果没有这个约束会怎样**：
联动建立后立即触发 DeviationFault 或限位阻断，操作者感觉"怎么刚联动上就解联了"。

**约束属于**：Motion Safety Rule


### 测试：`PositionInconsistent_Rejected`

**系统行为**：位置同步偏差超过 epsilon，联动申请被拒绝。

**背后的现实机械问题**：
联动的前提是双轴位置已经对齐（偏差在 epsilon 内）。如果偏差太大，联动后系统执行第一个运动命令的瞬间就会触发 DeviationFault。

**它保护的业务约束**：
联动建立条件之四：X1 和 X2 位置一致（镜像关系），偏差在 epsilon 内。

**如果没有这个约束会怎样**：
联动后首次 Jog，系统立即检测到"意外的偏差"并解联，操作者体验极差。更严重的是，如果偏差足够大，第一个运动命令可能导致横梁扭曲。

**约束属于**：Synchronization Constraint


### 测试：`DeviationExceeded_TriggersFault` / `DeviationFault_ForcesDecoupling`

**系统行为**：联动运行中偏差超阈值，触发 DeviationFault 并强制退出联动。

**背后的现实机械问题**：
这是整个龙门系统最核心的安全保护机制。X1/X2 已经失去同步，横梁可能已经发生扭曲。系统**不能等待操作者来处理**——必须在检测到偏差的同一个扫描周期内强制解联。

**为什么这是最关键的业务约束？**
因为龙门系统的核心物理假设（双轴镜像同步）已经被违反。继续维持联动只会加剧机械损伤。解联是止损操作，不是可选操作。

**它保护的业务约束**：
联动维持条件：联动期间必须持续保持镜像同步。一旦偏差超限，**立即且自动**退出联动。

**如果没有这个约束会怎样**：
这是最严重的后果——偏差持续扩大，横梁扭曲变形直至断裂，龙门结构报废。

**约束属于**：Synchronization Constraint


### 测试：`DeviationWithinRange_NoFault` / `DeviationExactlyAtEpsilon_NoFault`

**系统行为**：偏差在 epsilon 内（或等于 epsilon），不发生 DeviationFault。联动保持。

**背后的现实机械问题**：
绝对的镜像同步在物理上不可实现——传感器有噪声，机械有间隙，控制有延迟。必须允许微小偏差。

**它保护的业务约束**：
epsilon（默认 0.01mm）是系统允许的最大同步误差。在 epsilon 内不触发解联，避免误诊。

**如果没有这个约束会怎样**：
系统对微小振动过于敏感，频繁自动解联，无法正常生产。

**约束属于**：Hardware Safety Constraint


### 测试：`CanReCoupleAfterRealign`

**系统行为**：偏差导致的解联后，重新对齐位置可以再次建立联动。

**背后的现实机械问题**：
偏差导致的解联是"保护性动作"而不是"永久性故障"——横梁可能被操作者手动调整回对齐状态，此时应该可以重新建立联动。

**它保护的业务约束**：

**如果没有这个约束会怎样**：

**约束属于**：Application Workflow Constraint

## E. 文件：test_gantry_safety.cpp

### 测试：`LimitHasHighestPriority` / `LimitBlocksAllX2Operation`

**系统行为**：限位触发时，`checkOperability()` 返回 `Rejected_Limit`。所有运动命令被拒绝（MoveAbsolute/MoveRelative 无条件拒绝，Jog 根据方向判断）。

**背后的现实机械问题**：
限位开关是防止机械碰撞的最后防线。触碰限位意味着"不能再往这个方向走，否则会撞上硬限位块或机械终点"。

**它保护的业务约束**：

**约束属于**：Hardware Safety Constraint


### 测试：`LimitRejectsMoveAbsolute` / `LimitRejectsMoveRelative`

**系统行为**：限位触发时，MoveAbsolute 和 MoveRelative 被无条件拒绝（即使目标位置在反方向）。

**背后的现实机械问题**：
与 Jog 不同，Move 命令包含"目标位置"的概念。系统无法保证在去目标位置的路程中不会再触碰到限位。此外，Move 指令的下层实现可能包含"先反方向加减速"的轨迹规划。

**它保护的业务约束**：
Move（绝对/相对定位）在限位下被完全禁止——因为系统无法判定运动轨迹是否会进一步触发限位。

**为什么 Jog 和 Move 在限位时的行为不同？**
Jog 是"持续运动直到松开"，操作者可以实时控制。Move 是"到达目标位置即停"，轨迹不可控。限位时允许 Jog 远离但不允许 Move——这是一种"安全权衡"：Jog 是人机交互的实时操作，可靠性高；Move 是自动驾驶，更加危险。

**如果没有这个约束会怎样**：
MoveAbsolute(X1, -100)，X1 先反向加速，但在加速过程中又碰到了负限位——如果正负限位间距很小，相当于在一次 Move 中双侧限位都被触发。

**约束属于**：Motion Safety Rule

**它保护的业务约束**：
任意物理轴运动→逻辑轴运动。

**约束属于**：Synchronization Constraint


### 测试：`AlarmOverridesMotion`

**系统行为**：X1 Alarm + X2 Jogging → X = Error。

**背后的现实机械问题**：
报警比运动更重要。故障系统的运动状态无意义——应该先响应故障。

**它保护的业务约束**：
聚合优先级：Alarm > Moving。

**约束属于**：State Transition Constraint


### 测试：`LimitOverridesMotion`

**系统行为**：X1 Limit + X2 Jogging → X = Idle (限位阻断)。

**背后的现实机械问题**：

**它保护的业务约束**：
聚合优先级：Limit > Moving。限位阻断所有运动。

**约束属于**：State Transition Constraint


### 测试：`LogicalPositionEqualsX1Position`

**系统行为**：`X.position = X1.position`。

**背后的现实机械问题**：

**为什么 X 的逻辑位置 = X1 的物理位置？**
因为 X1 是"主导轴"——它的位置直接反映龙门整体在坐标系中的绝对位置。X2 的镜像位置是派生值。

**它保护的业务约束**：
逻辑位置取 X1 的物理位置（约束10）。

**如果没有这个约束会怎样**：
UI 显示 X 轴位置为 0，但实际 X1 在 100mm——操作者的位置信息完全失实。

**约束属于**：Synchronization Constraint


### 测试：`DeviationFaultInCoupledMode` / `SyncMaintainedWhenNoDeviation`

**系统行为**：Coupled 模式下，偏差超限→自动解联+DeviationFault 事件。无偏差→维持联动。

**背后的现实机械问题**：
联动状态绑定同步条件：同步满足→维持，同步不满足→解联。这是一个持续的守卫条件，不是一次性的。

**它保护的业务约束**：
联动维持条件：Coupled = f(同步状态)。同步失效→联动立即退出。

**如果没有这个约束会怎样**：
偏差已经很大了，但系统仍然认为自己在联动状态。横梁扭曲程度持续扩大。

**约束属于**：Synchronization Constraint


### 测试：`NoSyncCheckInDecoupledMode`

**系统行为**：Decoupled 模式下，即使 X1 和 X2 位置偏差极大，也不触发 DeviationFault。

**背后的现实机械问题**：
分动模式下，两个轴本来就是各自独立运动，偏差是正常行为。

**它保护的业务约束**：
DeviationFault 只在 Coupled 模式下触发。Decoupled 模式不检查镜像同步。

**如果没有这个约束会怎样**：
分动模式下 X1 移动到 200，X2 移动到 100，系统弹出 DeviationFault——错误报警。

**约束属于**：Domain Business Rule


### 测试：`LimitTriggeredEventOnEdge` / `AlarmRaisedEventOnEdge` / `LimitNotRetriggeredOnSubsequentCycles`

**系统行为**：限位/报警边沿触发事件，后续周期不重复触发。

**背后的现实机械问题**：
事件应该是"状态变化"的标记，不是"状态持续"的标记。持续报警的情况下，每周期都触发同一个事件是灾难性的——UI 会不断弹出报警弹窗。

**它保护的业务约束**：
事件基于边沿触发（变化检测），而非电平触发（状态持续）。

**如果没有这个约束会怎样**：
限位触发一次，UI 每 10ms 收到一次 LimitTriggered 事件，弹出无限弹窗。

**约束属于**：Domain Business Rule

# 第二部分：按约束主题重新组织——约束网络

## A. 联动模式约束（Mode Constraints）

### 为什么系统需要 Coupled / Decoupled 两种模式？

因为龙门由两个电机拖动同一横梁，有两种截然不同的物理操作方式：
- **Coupled（联动）**：两电机同步运动，横梁作为一个整体移动。适用于大行程直线运动。
- **Decoupled（分动）**：两电机各自独立运动。适用于调试、单轴维护、非对称作业。

**测试证据**：
- `test_gantry_mode.cpp`: 两个模式互斥，系统恰好处在一个模式中
- `test_gantry_coupling.cpp`: 联动不是默认状态，需要显式申请并通过条件校验
- `test_gantry_activation.cpp`: 不同模式下可操作的目标不同

### 为什么联动时禁止操作 X1/X2？

因为操作 X1 会破坏 X1 与 X2 的镜像同步。X1 运动 X2 不动→横梁扭曲。所以联动模式下 X1 和 X2 必须被"隐藏"，只暴露逻辑轴 X。

### 为什么解联后禁止操作 X？

**与上一个问题对称**：解联后，X 已经不是一个有物理意义的实体——它只是 X1 和 X2 的抽象，但两个轴不再同步。操作 X 没有意义。

### 联动建立需要哪些条件？

从测试 `test_gantry_coupling.cpp` 和 `test_gantry_coupling_service.cpp` 可反推以下条件链：

```
requestCoupling()
├── X1 已使能?   ─── 否→Rejected("X1 is not enabled")
├── X2 已使能?   ─── 否→Rejected("X2 is not enabled")
├── 无报警?      ─── 否→Rejected("Alarm is active")
├── 无限位?      ─── 否→Rejected("Limit is active")
├── 位置一致?    ─── 否→Rejected("deviation too large")
└── 全部通过 → Coupled
```

## B. 同步一致性约束（Synchronization Constraints）

### 为什么使用 |X1 + X2| <= epsilon 作为同步条件？

因为 X2 是镜像安装的。物理上：
- X1 正向运动时，X2 必须反方向运动，横梁才平直
- 所以同步状态是：X1 在 +p 时，X2 在 -p
- 偏差 = |X1 - (-X2)| = |X1 + X2|
- 偏差 <= epsilon 才算同步

**注意**：不是 |X1 - X2|，因为 X1 和 X2 坐标系定义不同（镜像）。

### X2 为什么是镜像方向？

因为两个电机安装在横梁两端，面对面安装。当龙门向"Forward"方向运动时：
- X1 电机正转，推动横梁向远离操作者方向运动
- X2 电机也必须正转——但因为电机安装方向相反，它必须正转才能让横梁也向远离操作者方向运动
- 但是从编码器看，X2 电机正转时位置读数是减小的

因此 X2 的位置与 X1 互为相反数：`X2.pos ≈ -X1.pos`

### 偏差超限后系统如何反应？

从测试可看出，系统反应是**即时且自动的**：

```
aggregateState()
├── Coupled 模式?
│   ├── 是 → 检查 |X1 + X2| <= epsilon?
│   │   ├── 是 → 保持 Coupled
│   │   └── 否 → 强制 Decoupled
│   │       └── 发布 DeviationFault + Decoupled 事件
│   └── 否 → 跳过同步检查
└── 聚合物理状态到逻辑轴
```

这个过程不经过任何外部决策者——是系统内部的不变式守卫。

## C. 安全约束（Safety Constraints）

### 安全检查的优先级是什么？

从 `checkOperability()` 的测试行为反推出的优先级：

```
checkOperability(target, direction):
1. ⚠️ 报警检查：AlarmActive → Rejected_Alarm  (最高优先级)
2. ⛔ 模式检查：TargetNotOperableInMode → Rejected_Mode
3. 🚧 限位检查：
   ├── Move: LimitTriggered → Rejected_Limit
   └── Jog 方向: LimitBlocksDirection → Rejected_Limit/Allowed
4. 🚦 命令槽检查：Busy → Rejected_Busy
```

### 为什么报警优先级最高？

**因为报警代表"不能工作"，限位代表"不能往这个方向工作"**。前者是绝对禁止，后者是相对禁止。

### 为什么正限位仍允许反方向运动？

**因为要保护轴能退出限位区**。限位不是死锁，它是有方向的保护：
- 正限位：禁止正向运动，允许反向运动
- 负限位：禁止反向运动，允许正向运动

### Jog 和 Move 在限位下的行为为什么不同？

这是一个关键设计决策，也是用户可能忽略的差异：

| 命令类型 | 正限位时 | 原因 |
|---------|---------|------|
| Jog Forward | 拒绝 | 朝限位方向运动，会撞 |
| Jog Backward | 允许 | 远离限位，需要退出 |
| MoveAbsolute(p) | 拒绝 | 轨迹不可控，无法确保途中不与限位接触 |
| MoveRelative(d) | 拒绝 | 同上 |

**为什么 Move 在限位时即使目标方向是远离也拒绝？**

因为 MoveAbsolute/MoveRelative 的内部执行轨迹可能包含反方向的短暂加速或路径规划。而且 Move 命令一旦开始，系统不能实时干预——它不像 Jog 那样"松手即停"。

## D. 命令生命周期约束（Command Lifecycle Constraints）

### 为什么系统一次只能有一个运动意图？

因为物理轴同一时刻只能执行一种运动。在 FakePLC 层面：
- `jog` 设置 `jogForward=true` 或 `jogBackward=true`
- `move` 设置 `targetPos` 和 `startMove`
- `stop` 设置 `stop`

这些是互斥的硬件寄存器操作。一个轴不可同时设为 jog 和 move。

### 命令的生命周期是什么？

```
Idle (命令槽空)
├── Jog(Coupled) → 写入 JogPending
├── MoveAbsolute → 写入 MoveAbsolutePending
├── MoveRelative → 写入 MoveRelativePending
└── Stop → Idle (清除命令槽)
```

### Jog 和 Move 的区别是什么？

| 特性 | Jog | Move(Absolute/Relative) |
|------|-----|------------------------|
| 是否可覆盖 | ✅ 可覆盖另一个 Jog | ❌ 被 Busy 拒绝 |
| Stop 是否清除 | ✅ | ✅ |
| 限位时是否可操作 | 方向受限 | ❌ 完全拒绝 |
| 执行模式 | 持续运动直到 Stop | 到达目标位置自动停止 |

这个差异非常重要。Jog 是"人机交互"，Move 是"自动驾驶"。Jog 的覆盖操作允许操作者快速改变方向，而 Move 的排斥机制防止了两个定位指令冲突。

## E. 状态聚合约束（State Aggregation Constraints）

### 为什么逻辑轴需要聚合物理轴状态？

因为龙门操作者面对的是一个逻辑轴 X，但系统需要同时管理 X1 和 X2。聚合是对外的统一接口，对内则是两个物理轴。

### 状态聚合优先级是什么？

从测试 `test_gantry_system_aggregation.cpp` 和 `test_gantry_activation.cpp` 可以反推出：

```
aggregateState() 优先级（从高到低）：
1. Alarm (Error)     → 任一轴报警，X=Error
2. Limit (Idle)      → 任一轴限位，X=Idle + hasLimit=true
3. Moving            → 任一轴运动，X=对应运动类型
4. Idle              → 双轴空闲，X=Idle
```

### 为什么报警优先级最高？

**因为报警代表系统处于故障状态**。故障状态不应被任何其他状态掩盖：
- 一个正在运动的轴发生了报警，系统应该报告错误而不是运动状态
- 一个在限位的轴发生了报警，系统应该报警而不是报告限位阻断

### 为什么报警覆盖限位？

反过来想：如果报告限位而不是报警，操作者看到"限位阻断"提示，尝试"反向 Jog 退出"——但报警状态下 Jog 被禁止。操作者困惑了："为什么提示限位阻断，但是 Jog 不了？"

正确的做法是直接报告"报警"，操作者知道问题在报警上，而不是限位上。

# 第三部分：从约束倒推领域模型

## 1. 核心业务实体

### GantrySystem（聚合根）

**为什么存在？**
因为龙门不是两个独立轴的简单组合。它有自己的生命周期、状态、约束和对外接口。它是一个有生命周期的聚合：

```
GantrySystem 的生命周期：
1. 创建（Decoupled，双轴独立）
2. 条件满足 → requestCoupling() → 进入 Coupled
3. 持续同步检查 → 如果偏差超限 → 自动解联
4. 用户主动 → requestDecoupling() → 回到 Decoupled
```

**为什么必须是聚合根？**
因为 GantrySystem 管理着两个 PhysicalAxis 的**关系约束**。PhysicalAxis 不是独立实体——它们的状态需要聚合（aggregateState）、它们的同步需要守卫（DeviationFault）、它们的模式需要管理（Coupled/Decoupled）。

物理轴之外的代码不能直接访问 PhysicalAxis——必须通过 GantrySystem。这是聚合根的核心规则。

### PhysicalAxis（实体）

**为什么存在？**
因为需要表示一个实实在在的物理电机+编码器+限位传感器的状态。它有 ID（X1 或 X2）、位置、使能状态、报警状态、限位状态。

PhysicalAxis 的职责很纯粹：承载来自硬件层的状态快照（syncState），并提供当前状态的只读查询。

### LogicalAxis（实体）

**为什么存在？**
因为操作者在联动模式下不是对着 X1/X2 操作，而是对着"逻辑轴 X"。LogicalAxis 提供了：
- 命令槽（CommandSlot）：管理当前运动意图
- 聚合状态：从 PhysicalAxis 推导而来
- 逻辑位置：固定等于 X1 的物理位置

**关键区别：** PhysicalAxis 是"物理世界的映射"，LogicalAxis 是"业务抽象的载体"。

## 2. 核心值对象

### MotionDirection

**为什么存在？**
因为运动方向必须是**与物理实现无关的纯语义概念**。操作者不需要知道电机是正转还是反转、编码器是加计数还是减计数。操作者只知道"往前走（Forward）"和"往后退（Backward）"。

**现实语义：**
- Forward = 远离操作者
- Backward = 靠近操作者

### GantryMode

**为什么存在？**
因为龙门有两种截然不同的工作模式。模式决定了：
- 操作对象是 X 还是 X1/X2
- 是否需要检查同步一致性
- 命令如何拆分到物理轴

### PositionConsistency

**为什么存在？**
因为同步偏差是龙门系统最核心的度量值。PositionConsistency:
- 封装了偏差计算：`|X1 + X2|`
- 封装了 epsilon 阈值（默认 0.01mm）
- 提供 isConsistent() 判断
- 提供 computeLogicalPosition() （= X1.pos）

**现实语义：**
偏差不是"误差"，偏差是"横梁是否发生扭曲"的指标。

### CouplingCondition

**为什么存在？**
因为联动建立有 4 个必须同时满足的条件（使能+无报警+无限位+位置一致）。将这些条件封装为 CouplingCondition 值对象，使得条件检查和错误报告不受具体实现影响。

### SafetyCheckResult

**为什么存在？**
因为安全检查有多种结果（Allowed / Rejected_Alarm / Rejected_LimitForward / Rejected_LimitBackward），并且需要携带原因描述。它封装了判定结果及其语义。

### Operability

**为什么存在？**
因为操作可行性评估是 GantrySystem 对外暴露的关键接口（checkOperability()）。Operability 枚举明确了每种拒绝原因，使调用方可以精确处理。

### CommandResult

**为什么存在？**
因为运动命令需要返回"是否接受"以及"如果拒绝是什么原因"。

## 3. 聚合根的职责

### GantrySystem 必须成为聚合根的原因（总结）

1. **它管理模式状态**：Coupled/Decoupled 是 GantrySystem 级别的状态，不是 PhysicalAxis 能决定的。

2. **它守卫不变式**：
   - 联动时禁止操作 X1/X2
   - 未使能/报警/限位/位置不一致时不能进入联动
   - 偏差超限时强制退出联动
   - 一次性只能有一个运动意图（命令槽）
   - Stop 无条件接受

3. **它负责状态聚合**：PhysicalAxis 的原始状态需要通过聚合规则（Alarm>Limit>Moving>Idle）转化为 LogicalAxis 的状态。

4. **它拥有事件管理**：模式切换、报警、限位、偏差故障等事件的生成和队列管理。

5. **它是对外单点**：Application 和 Presentation 层通过 IGantryStateQuery 和 IGantryEventBus 接口与 GantrySystem 通信，不需要知道 X1/X2 的存在。

## 4. 系统最核心的不变量（Invariant）

以下约束是**绝对不能违反的**。违反任何一个都意味着系统处于无效状态：

### 不变式 1：工作模式互斥
```
mode() ∈ {Coupled, Decoupled}
isCoupled() XOR isDecoupled()  // 恰好一种
```
**为什么是 Invariant？**：因为没有"半联动"的物理状态。

### 不变式 2：操作对象绑定模式
```
mode() == Coupled  ⇒  isTargetOperable(X) == true  ∧  isTargetOperable(X1/X2) == false
mode() == Decoupled ⇒ isTargetOperable(X) == false ∧ isTargetOperable(X1/X2) == true
```
**为什么是 Invariant？**：违反此规则意味着允许破坏镜像同步的操作。

### 不变式 3：同步一致性是联动的前提
```
isCoupled()  ⇒  |X1.pos + X2.pos| <= epsilon
```
**为什么是 Invariant？**：违反此规则意味着系统在横梁已扭曲的情况下认为自己处于同步状态。

### 不变式 4：报警时禁止运动
```
isAnyAlarm() ⇒ 所有 `jog/moveAbsolute/moveRelative` 返回 Rejected_Alarm
isAnyAlarm() ⇒ `stop()` 仍被接受
```
**为什么是 Invariant？**：违反此规则意味着在故障状态下允许运动，可能造成人身伤害或设备损坏。

### 不变式 5：同时只能有一个运动意图
```
命令槽被占用 ⇒ 新的 jog/move 被拒绝（除非 Jog 覆盖 Jog）
stop() ⇒ 命令槽清空
```
**为什么是 Invariant？**：违反此规则意味着物理轴在同一时刻收到两个矛盾的执行指令。

### 不变式 6：Stop 无条件接受
```
任何时候 stop() ⇒ accepted == true
```
**为什么是 Invariant？**：这是最底层的安全保护。Stop 不能因为任何原因被拒绝。

# 总结

龙门同步系统的设计核心是一个**现实物理问题**——双电机驱动同一横梁必须保持镜像同步。

从测试行为反推出的约束网络中，最核心的发现是：

1. **镜像同步是系统的物理基础**。所有其他约束（模式管理、安全保护、状态聚合）都是为保护这个物理基础服务的。

2. **偏差保护是最关键的自动机制**。它不是"可选的"，而是"强制自动执行的"——偏差超限必须立即解联，不需要也不应该等操作者决策。

3. **安全有严格的层次结构**。报警 > 限位 > 模式 > 命令槽。每种安全条件的处理逻辑都不相同：
   - 报警：绝对禁止全部运动
   - 限位：方向性保护，保留退出能力
   - 模式检查：目标匹配过滤
   - 命令槽：并发控制

4. **Jog 和 Move 在安全策略上的差异**是有意的设计决策，不是实现疏漏。Jog 是人工实时操控，安全容限高；Move 是自动执行，需要更严格的保护。

5. **GantrySystem 作为聚合根**是因为它守卫的不变式涉及两个 PhysicalAxis 之间的关系。这些关系约束（同步、模式、互斥）不可能被任何一个 PhysicalAxis 个体所实现。

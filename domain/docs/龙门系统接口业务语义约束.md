# 龙门系统接口业务语义约束

> 版本: v1.0
> 类型: 接口设计契约（Interface Design Contract）
> 关联: 与《龙门同步系统业务语义约束分析》互为补充——后者从测试反推约束，本文从接口正向定义约束
> 日期: 2026-05-07

---

## 概述

本文是 **Domain 层所有公共接口（端口 + 聚合根）的正式语义契约**。每条接口方法都定义了：

1. **前置条件（Pre-condition）**：调用方必须满足的条件
2. **后置条件（Post-condition）**：方法执行后系统的状态
3. **不变式（Invariant）**：该方法运行时必须保持的约束
4. **业务语义**：该方法在真实机械世界中的意义

本文不定义"怎么实现"——那是实现细节。它定义的是"调用者可以信赖什么"。

## 一、GantrySystem（聚合根）

### 1.1 构造器

```cpp
GantrySystem(PhysicalAxis x1, PhysicalAxis x2);
```

**语义**：
创建一个龙门双轴系统，将两个物理轴 X1 和 X2 绑定到同一个聚合根管理。

**前置条件**：
- `x1.id() == AxisId::X1`
- `x2.id() == AxisId::X2`
- 两个 PhysicalAxis 实例均已合法初始化

**后置条件**：
- `mode() == GantryMode::Decoupled`
- 事件队列为空
- `aggregatedState()` 反映双轴初始状态

**不变式**：
- `GantrySystem` 始终恰好持有 2 个 PhysicalAxis 和 1 个 LogicalAxis
- 聚合根生命周期内 PhysicalAxis 引用不变

---

### 1.2 模式管理

#### requestCoupling()

```cpp
CouplingCondition::Result requestCoupling();
```

**语义**：
向系统申请进入联动模式。系统验证联动建立条件（约束13）后决定是否接受。

**前置条件**：
无（调用总是合法的，结果由守卫条件决定）

**后置条件**：
- 如果条件通过：`mode() == GantryMode::Coupled`
- 如果条件不通过：`mode() == GantryMode::Decoupled`（不变）
- 行为是幂等的——Coupled 模式下再次调用，若条件仍然满足，模式不变

**业务约束**：
|| 条件 | 如果失败 |
|:----:|------|---------|
| ① | `x1().isEnabled()` | `result.failReason = "X1 is not enabled"` |
| ② | `x2().isEnabled()` | `result.failReason = "X2 is not enabled"` |
| ③ | `!x1().isAlarmed() && !x2().isAlarmed()` | `result.failReason = "Alarm is active"` |
| ④ | `!x1().isAnyLimitActive() && !x2().isAnyLimitActive()` | `result.failReason = "Limit is active"` |
| ⑤ | `|X1.pos + X2.pos| <= epsilon` | `result.failReason = "Position deviation too large"` |

**事件**：
- 成功：`CouplingRequested` → `Coupled`
- 失败：`CouplingRequested`（仅此一个，无 `Coupled`）

---

#### requestDecoupling()

```cpp
void requestDecoupling(const std::string& reason = "");
```

**语义**：
主动退出联动模式，返回分动模式。

**前置条件**：
无（Decoupled 下调用是无操作）

**后置条件**：
- `mode() == GantryMode::Decoupled`
- 如果之前是 Coupled，发布 `Decoupled` 事件
- 如果之前是 Decoupled，无事件，状态不变

**业务语义**：
分动是"安全兜底"模式。解联后两个物理轴的操作恢复独立，逻辑轴 X 不可操作。

**特殊约束**：
- `reason` 为空字符串时，事件描述为 `"Decoupled mode entered"`
- `reason` 非空时，事件描述包含该原因（用于诊断）

---

### 1.3 操作可行性检查

#### checkOperability()

```cpp
Operability checkOperability(AxisId target, MotionDirection dir) const;
```

**语义**：
在发出运动命令之前，预先检查该命令是否会被接受。该项检查不修改任何状态。

**前置条件**：
- 无（纯查询函数，不影响状态）

**返回值映射**：

| 返回值 | 含义 | 检查阶段 |
|--------|------|---------|
| `Allowed` | 命令可以被接受 | — |
| `Rejected_Mode` | 当前模式下此目标不可操作 | 模式检查（约束18） |
| `Rejected_Alarm` | 任一物理轴有报警 | 报警检查（约束17，最高优先） |
| `Rejected_Limit` | 限位触发导致此方向被拒绝 | 限位方向矩阵（约束15-16） |
| `Rejected_Busy` | 命令槽已被占用 | 命令槽检查（约束19） |

**检查优先级**（高 → 低）：
```
1. AlarmActive → Rejected_Alarm
2. ModeMismatch → Rejected_Mode
3. LimitBlocks → Rejected_Limit (Move无差别拒绝, Jog方向性拒绝)
4. CommandSlotBusy → Rejected_Busy
```

**业务约束**：
- 报警始终优先于模式检查——即使模式不匹配，如果报警存在，也应报告 `Rejected_Alarm`（因为报警是更严重的状态）
- 正常状态下返回 `Allowed`

### 1.4 运动命令

#### jog()

```cpp
CommandResult jog(AxisId target, MotionDirection dir);
```

**语义**：
发起点动（Jog）：轴持续向指定方向运动，直到收到 Stop 命令为止。

**业务语义**：
"操作者按住按钮，轴持续运动，松开即停（通过 Stop 实现）"

**约束差异**：

| 模式 | 合法目标 | 命令路径 |
|------|---------|---------|
| Coupled | X | 逻辑轴命令槽 → 拆分为 X1 Forward + X2 Backward |
| Decoupled | X1 或 X2 | 直接写入对应物理轴 |

**安全规则**：
| 条件 | 行为 |
|------|------|
| Coupled + 位置偏差 > epsilon | 触发 DeviationFault，自动解联，拒绝命令 |
| Coupled + 任一轴正限位 + Forward | `Rejected_Limit` |
| Coupled + 任一轴正限位 + Backward | `Allowed`（允许远离限位） |
| 报警激活 | `Rejected_Alarm` |
| 命令槽被非-Jog 命令占用 | `Rejected_Busy` |

**特殊规则**：
- Jog **可以覆盖**已有的 Jog 命令。这是唯一允许"覆盖"的命令类型。
- 覆盖后命令槽中的方向更新为新方向。

---

#### moveAbsolute()

```cpp
CommandResult moveAbsolute(AxisId target, double pos);
```

**语义**：
发起绝对定位：轴从当前位置运动到指定目标位置（坐标系中的绝对坐标）。

**业务语义**：
"告诉轴'去这个位置'，轴自己规划轨迹到达"

**约束差异**：

| 模式 | 合法目标 | 目标位置语义 |
|------|---------|-------------|
| Coupled | X | 逻辑位置目标=X1目标位置, X2= -目标位置 |
| Decoupled | X1 或 X2 | 直接赋予对应物理轴 |

**安全规则**：
| 条件 | 行为 |
|------|------|
| 任何限位触发 | **无条件拒绝**——即使目标方向是远离限位 |
| 报警激活 | **无条件拒绝** |
| 命令槽被任何命令占用 | `Rejected_Busy`（Jog 也不例外） |
| Coupled + 位置偏差 | 先触发 DeviationFault 解联，再检查限位 |

**关键安全差异**（与 Jog 对比）：
```
Jog 在限位时的行为：
  正限位 → Forward 拒绝, Backward 允许
  Move 在限位时的行为：
  正限位 → Forward 拒绝, Backward 同样拒绝（⚠️）  
```

**原因**：Move 命令在执行时轨迹不可控（可能存在反方向加减速段），而 Jog 是人机实时操控。

---

#### moveRelative()

```cpp
CommandResult moveRelative(AxisId target, double delta);
```

**语义**：
发起相对定位：轴从当前位置运动到"当前 + delta"的位置。

**业务语义**：
"从当前位置往指定方向走 delta 距离"

**安全规则**：与 `moveAbsolute()` 完全一致
- 限位触发 → 无条件拒绝
- 报警激活 → 无条件拒绝
- 命令槽忙 → 拒绝

---

#### stop()

```cpp
CommandResult stop(AxisId target);
```

**语义**：
立即停止指定目标的当前运动，清除命令槽。

**前置条件**：
无（**始终可调用**）

**后置条件**：
- 命令槽被清除（`pendingCommand().type == CommandType::None`）
- 轴的运动被中断
- 所有安全检查均**不拦截** Stop

**不变式**（最重要的安全规则之一）：
```
                   ┌─────────────────────────────────┐
                   │     Stop 无条件接受              │
                   │  (报警✔ 限位✔ 忙✔ 偏差✔ 模式✘)  │
                   └─────────────────────────────────┘
```

**业务语义**：
Stop 不是"运动命令"——它是"结束一切运动的命令"。它不能因为任何安全条件被拒绝。

---

### 1.5 状态聚合

#### aggregateState()

```cpp
void aggregateState();
```

**语义**：
执行一个完整的"感知-聚合-守卫"周期。这是系统的**心跳函数**，每次扫描周期必须调用一次。

**前置条件**：
- 物理轴状态已经通过 `x1().syncState()` / `x2().syncState()` 与硬件同步

**后置条件**：
```
aggregateState() 的完整执行链：

1. 读取物理轴状态：
   X1_state = x1().snapshot()
   X2_state = x2().snapshot()

2. 聚合状态（按优先级）：
   ├── Alarm(AxisState::Error)     → X = Error
   ├── Limit(AxisState::Idle)      → X = Idle + Limit flag
   ├── Moving                      → X = 对应运动类型
   └── Idle                        → X = Idle

3. 逻辑位置：
   X.position = X1.position

4. 【仅 Coupled 模式】同步维持检查：
   └── |X1 + X2| > epsilon → 强制 Decoupled + DeviationFault 事件

5. 边沿事件检测：
   ├── 限位 0→1 → LimitTriggered 事件
   ├── 限位 1→0 → 不产生事件（限位恢复不触发事件）
   ├── 报警 0→1 → AlarmRaised 事件
   └── 报警 1→0 → 不产生事件
```

**边沿事件规则**：
```
LimitTriggered: 仅在限位从 false → true 时发布一次
AlarmRaised:    仅在报警从 false → true 时发布一次
DeviationFault: 仅在偏差超限且当前为 Coupled 时发布

持续状态不重复发布：
  限位持续激活 → 第二周期不产生 LimitTriggered
  报警持续激活 → 第二周期不产生 AlarmRaised
```

---

#### checkSyncMaintenance()

```cpp
bool checkSyncMaintenance();
```

**语义**：
检查联动是否仍可维持。该函数在 `aggregateState()` 内部被调用，也可独立调用。

**前置条件**：
- 系统处于 Coupled 模式（Decoupled 下调用是无操作）

**后置条件**：
- 如果偏差 ≤ epsilon：返回 `true`，模式不变
- 如果偏差 > epsilon：返回 `false`，强制解联（`mode()=Decoupled`），发布 `DeviationFault` 事件

**业务语义**：
"联动不是永久契约——它是持续基于同步条件的许可。偏差超限即时终止。"

### 1.6 事件管理

#### drainEvents()

```cpp
std::vector<GantryEvents::Event> drainEvents();
```

**语义**：
提取并清空当前事件队列。用于每扫描周期结束时将事件传递给 Application 层。

**前置条件**：
无（总是合法调用）

**后置条件**：
- 返回当前所有未处理事件
- 内部事件队列清空
- 多次调用 `drainEvents()` 时，第二次返回空列表

## 二、PhysicalAxis（物理轴实体）

### 2.1 syncState()

```cpp
void syncState(const PhysicalAxisState& state);
```

**语义**：
将来自 HAL 层（硬件抽象层）的物理轴状态快照同步到实体中。

**业务约束**：
```cpp
syncState() 是 PhysicalAxis 唯一的状态写入入口。
所有状态变更都必须通过 PhysicalAxisState DTO 传入。
直接调用 setEnabled/setPosition/setAlarmed 等方法仅用于测试。
```

**前置条件**：
- `state` 结构体的所有字段已由 HAL 层填充

**后置条件**：
- `isEnabled() == state.enabled`
- `isAlarmed() == state.alarmed`
- `isPosLimitActive() == state.posLimitActive`
- `isNegLimitActive() == state.negLimitActive`
- `position() == state.position`

---

### 2.2 snapshot()

```cpp
PhysicalAxisState snapshot() const;
```

**语义**：
导出当前物理轴的完整状态快照。

**返回值**：
一个包含所有字段的 `PhysicalAxisState` 结构体。

**用途**：
在 `aggregateState()` 中被调用，用于读取物理轴状态进行聚合。

## 三、LogicalAxis（逻辑轴实体）

### 3.1 命令槽管理

#### tryAcceptJog()

```cpp
std::string tryAcceptJog(MotionDirection dir);
```

**前置条件**：
无（返回错误字符串而非抛出异常）

**后置条件**：
- 命令槽为空 或 命令槽中的命令类型为 `Jog`：
  - 接受，写入 `CommandType::Jog` + `jogDirection = dir`
  - 返回空字符串
- 命令槽中的命令类型为 `MoveAbsolute` / `MoveRelative`：
  - 拒绝，返回 `"busy"`（或类似描述）

---

#### tryAcceptMoveAbsolute()

```cpp
std::string tryAcceptMoveAbsolute(double targetPos);
```

**前置条件**：
无

**后置条件**：
- 命令槽为空：接受，写入 `CommandType::MoveAbsolute` + `moveTarget = targetPos`
- 命令槽占用（任何类型，包括 Jog）：拒绝，返回非空字符串

**关键规则**：
```
Jog 可以覆盖 Jog，但 Move 不能覆盖任何命令（包括另一个 Move）。
```

---

#### tryAcceptStop()

```cpp
std::string tryAcceptStop();  // 始终返回空字符串
```

**前置条件**：
无

**后置条件**：
- 命令槽被清除（`CommandType::None`）
- 返回空字符串（始终成功）

## 四、端口接口

### 4.1 IGantryStateQuery

```cpp
class IGantryStateQuery {
public:
    virtual GantryMode mode() const = 0;
    virtual AxisState aggregatedState() const = 0;
    virtual bool isCoupled() const = 0;
    virtual double position() const = 0;
    virtual double x1Position() const = 0;
    virtual double x2Position() const = 0;
    virtual bool x1Enabled() const = 0;
    virtual bool x2Enabled() const = 0;
    virtual bool isAnyAlarm() const = 0;
    virtual bool isAnyLimit() const = 0;
    virtual bool canAcceptCommand() const = 0;
    virtual bool isTargetOperable(AxisId target) const = 0;
    virtual std::string stateDescription() const = 0;
};
```

**语义约束**：
| 方法 | 语义 |
|------|------|
| `mode()` | 返回当前工作模式（Coupled / Decoupled）。**Query 层永远看到最新状态** |
| `aggregatedState()` | 返回逻辑轴 X 的聚合状态（Idle / Jogging / MovingAbsolute / MovingRelative / Error） |
| `isCoupled()` | 等同于 `mode() == Coupled` |
| `position()` | 等同于 `x1Position()`（逻辑位置取自 X1） |
| `x1Position()` | X1 物理轴的当前位置 |
| `x2Position()` | X2 物理轴的当前位置 |
| `x1Enabled()` | X1 是否已使能 |
| `x2Enabled()` | X2 是否已使能 |
| `isAnyAlarm()` | X1 或 X2 是否有报警 |
| `isAnyLimit()` | X1 或 X2 是否触发了任何限位 |
| `canAcceptCommand()` | 命令槽是否空闲（可接受新命令） |
| `isTargetOperable(target)` | 当前模式下 `target` 是否可被操作 |
| `stateDescription()` | 人类可读的状态描述（用于诊断） |

**线程安全**：
IGantryStateQuery 的所有方法都是**只读查询**，不修改任何状态。可以在任何时候安全调用。

---

### 4.2 IGantryCommandPort

```cpp
class IGantryCommandPort {
public:
    struct DriveCommand {
        double targetPosition;  // 物理轴绝对目标位置 (mm)
        double velocity;        // 速度 (mm/s)
        bool start;             // 上升沿触发执行
        bool stop;              // 停止信号
        bool jogForward;        // Jog 正向
        bool jogBackward;       // Jog 反向
    };
    virtual ~IGantryCommandPort() = default;
    virtual void commit(const DriveCommand& cmd) = 0;
    virtual void commitStop() = 0;
    virtual DriveCommand pendingCommand() const = 0;
};
```

**语义约束**：
| 方法 | 语义 |
|------|------|
| `commit(cmd)` | 将计算好的驱动命令写入 HAL 层执行。命令由 DomainService 在命令拆分后产生 |
| `commitStop()` | 直接写入急停命令到所有轴 |
| `pendingCommand()` | 返回当前累积的命令缓冲区（用于测试验证） |

**调用规范**：
- `commit()` 应在每扫描周期最多调用一次
- 写入 `commandType` 和对应的参数（`targetPos` / `velocity` / `jog` 等）
- Domain 层不关心实现细节——端口只是"写入寄存器"

---

### 4.3 IGantryFeedbackPort

```cpp
class IGantryFeedbackPort {
public:
    virtual ~IGantryFeedbackPort() = default;
    virtual PhysicalAxisState readState(AxisId id) = 0;
    virtual bool isConnected() const = 0;
    virtual void resetAlarm() = 0;
};
```

**语义约束**：
| 方法 | 语义 |
|------|------|
| `readState(id)` | 从硬件读取指定物理轴的当前状态快照。每次调用返回最新值 |
| `isConnected()` | 返回与硬件控制器的连接状态 |
| `resetAlarm()` | 清除报警状态。**这不是永久清除报警原因——它只是重置报警标志** |

### 4.4 IGantryEventBus

```cpp
class IGantryEventBus {
public:
    virtual ~IGantryEventBus() = default;
    virtual void publish(const GantryEvents::Event& event) = 0;
};
```

**语义约束**：
- `publish()` 将领域事件推送到事件总线
- 事件总线负责将事件分发给所有已注册的订阅者
- 事件是"已发生的领域事实"，不可撤销

**事件类型与触发条件**：

| 事件 | 触发条件 | 包含信息 |
|------|---------|---------|
| `CouplingRequested` | 调用 `requestCoupling()` | 发起时间 |
| `Coupled` | 联动条件通过，模式切为 Coupled | 联动建立时间 |
| `Decoupled` | 调用 `requestDecoupling()` 或偏差触发解联 | 解联原因 |
| `DeviationFault` | Coupled 下偏差 > epsilon | 当前偏差值、阈值 |
| `LimitTriggered` | 限位 0→1 边沿 | 触发的轴和方向 |
| `AlarmRaised` | 报警 0→1 边沿 | 报警的轴 |
| `CommandRejected` | 运动命令被拒绝 | 拒绝原因 |

## 五、核心不变式一览

### 不变式 1：模式互斥

```
mode() ∈ {Coupled, Decoupled}
isCoupled() XOR isDecoupled()
```

**涉及的接口**：`IGantryStateQuery::mode()`, `IGantryStateQuery::isCoupled()`
**守卫者**：`GantrySystem`

---

### 不变式 2：操作对象绑定模式

```
Coupled  ⇒  isTargetOperable(X) == true   ∧  isTargetOperable(X1) == isTargetOperable(X2) == false
Decoupled ⇒ isTargetOperable(X) == false  ∧  isTargetOperable(X1) == isTargetOperable(X2) == true
```

**涉及的接口**：`IGantryStateQuery::isTargetOperable()`, `GantrySystem::checkOperability()`
**守卫者**：`GantrySystem::checkOperability()` 的第一步

---

### 不变式 3：同步一致性

```
isCoupled()  ⇒  |X1.pos + X2.pos| <= epsilon
```

**涉及的接口**：`IGantryStateQuery::position()`, `IGantryStateQuery::x1Position()`, `IGantryStateQuery::x2Position()`
**守卫者**：`GantrySystem::aggregateState()` + `GantrySystem::checkSyncMaintenance()`

---

### 不变式 4：报警禁止运动

```
isAnyAlarm() ⇒ 所有 jog/moveAbsolute/moveRelative 返回 Rejected_Alarm
isAnyAlarm() ⇒ stop() 仍被接受
```

**涉及的接口**：`GantrySystem::checkOperability()`, `GantrySystem::stop()`
**守卫者**：`GantrySystem::checkOperability()` 的报警检查阶段

---

### 不变式 5：单运动意图

```
canAcceptCommand() == false ⇒ jog/moveAbsolute/moveRelative 返回 Rejected_Busy
stop() ⇒ canAcceptCommand() == true
Jog 例外：canAcceptCommand() == false 但命令槽是 Jog ⇒ Jog 仍可覆盖
```

**涉及的接口**：`IGantryStateQuery::canAcceptCommand()`, `GantrySystem::jog/moveAbsolute/moveRelative/stop()`
**守卫者**：`LogicalAxis` 命令槽

---

### 不变式 6：Stop 无条件接受

```
在任何状态下调用 stop() ⇒ accepted == true
stop() 不受 checkOperability() 的约束
```

**涉及的接口**：`GantrySystem::stop()`
**为什么**：Stop 是最后的物理安全机制。

---

### 不变式 7：事件边沿触发

```
LimitTriggered 事件 ⇒ 限位在前一周期为 false，本周期为 true
AlarmRaised 事件 ⇒ 报警在前一周期为 false，本周期为 true
DeviationFault 事件 ⇒ 偏差在本周期首次超限

持续激活不重复触发：
  限位保持 true 多周期 → LimitTriggered 只发一次
```

**涉及的接口**：`IGantryEventBus::publish()`, `GantrySystem::drainEvents()`
**守卫者**：`GantrySystem::aggregateState()`

## 六、同步周期约定

一个完整的同步扫描周期定义如下：

```
Step 1: 读取硬件反馈
        x1().syncState(feedbackPort.readState(AxisId::X1))
        x2().syncState(feedbackPort.readState(AxisId::X2))
        
Step 2: 状态聚合 + 联动维持检查
        gantrySystem.aggregateState()
        
Step 3: 排空事件 → 传递给 Application 层
        auto events = gantrySystem.drainEvents()
        for (auto& event : events) { eventBus.publish(event); }
        
Step 4: 如果命令槽有未处理命令，执行命令拆分
        if (gantrySystem.canAcceptCommand() && pendingCommand exists) {
            auto driveCmds = GantryDomainService::dispatchCommand(gantrySystem);
            for (auto& [axisId, cmd] : driveCmds) {
                commandPort.commit(axisId, cmd);
            }
        }
        
Step 5: 等待下一个周期（或由硬件中断/定时器驱动）
```

**业务约束**：
- Step 1 和 Step 2 必须在同一周期内连续执行，不可拆分
- Step 3 必须在 Step 2 之后立即执行，确保事件是"最新的"
- Step 4 在 Step 3 之后执行——因为事件可能包含 DeviationFault，会改变系统状态

## 八、以下行为在接口设计中被明确禁止

| 禁止行为 | 违反的约束 | 后果 |
|---------|:----------:|------|
| 在 Coupled 模式下直接对 X1/X2 发命令 | 约束18 操作互斥 | 横梁扭曲 |
| 在 Decoupled 模式下对 X 发命令 | 约束18 操作互斥 | 行为未定义 |
| 在报警状态下发运动命令（除 Stop） | 约束17 报警约束 | 设备损坏 |
| 在限位触发时执行 MoveAbsolute/Relative | 约束15-16 限位保护 | 机械碰撞 |
| 忽略 DeviationFault 事件继续运动 | 约束14 同步维持 | 横梁断裂 |
| 在一个扫描周期内多次调用 aggregateState() | 状态一致性 | 事件丢失或重复 |
| 在 aggregateState() 之前读取逻辑状态 | 状态一致性 | 读到过期状态 |
| 限制 Stop 命令的执行 | 约束6 Stop 无条件 | 无法紧急停止 |

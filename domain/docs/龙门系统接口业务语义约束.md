## 2. IGantryFeedbackPort — 反馈端口

### 2.1 业务语义

> **反馈端口是 GantrySystem 感知外部物理世界的唯一窗口。** 在每个 PLC 扫描周期开始时，调用方必须通过此端口获取 X1/X2 的最新物理状态，然后将状态快照注入 GantrySystem 以驱动状态聚合和安全检查。

### 2.2 接口定义

```cpp
class IGantryFeedbackPort {
public:
    virtual ~IGantryFeedbackPort() = default;
    virtual PhysicalAxisState getX1Feedback() const = 0;
    virtual PhysicalAxisState getX2Feedback() const = 0;
    virtual void resetAlarm() = 0;
    virtual bool isAnyAlarm() const = 0;
    virtual bool isAnyLimit() const = 0;
};
```

### 2.3 前置条件 (Preconditions)

| # | 条件 | 违反后果 | 约束 |
|---|------|----------|------|
| P-FB-01 | `getX1Feedback()` / `getX2Feedback()` 必须返回**最新**物理状态（不得缓存） | 位置偏差检测失效，联动维持检查误判 | 约束14 |
| P-FB-02 | 调用 `resetAlarm()` 前必须确认故障已排除 | 报警反复触发 | 约束17 |
| P-FB-03 | 每个扫描周期最多调用一次 getX1Feedback/getX2Feedback | 性能浪费 | — |

### 2.4 后置条件 (Postconditions)

| # | 条件 | 约束 |
|---|------|------|
| Q-FB-01 | 返回的 `PhysicalAxisState.position` 为当前真实位置（double） | 约束8, 10 |
| Q-FB-02 | 返回的 `PhysicalAxisState.enabled` 反映 PLC 实际使能状态 | 约束13.1 |
| Q-FB-03 | 返回的 `PhysicalAxisState.alarmed` 反映 PLC 实际报警状态 | 约束17 |
| Q-FB-04 | `posLimitActive` / `negLimitActive` 反映 PLC 实际限位状态 | 约束15, 16 |
| Q-FB-05 | `resetAlarm()` 后下次反馈 `alarmed == false` | 约束17 |
| Q-FB-06 | `isAnyAlarm()` ≡ `getX1Feedback().alarmed \|\| getX2Feedback().alarmed` | 约束17 |
| Q-FB-07 | `isAnyLimit()` ≡ 任一轴 `posLimitActive \|\| negLimitActive` | 约束15 |

### 2.5 不变式 (Invariants)

| # | 不变式 | 约束 |
|---|--------|------|
| I-FB-01 | 端口本身不修改物理状态，纯只读（`resetAlarm()` 除外） | 架构原则9 |
| I-FB-02 | `position` 坐标系与 `IGantryCommandPort` 下发坐标系一致 | 约束10 |
| I-FB-03 | X1 和 X2 的状态快照应属于同一 PLC 周期 | 约束14 |

### 2.6 错误场景

| 场景 | 行为 | 处理方 |
|------|------|--------|
| PLC 通信中断 | 返回上一次有效快照 + 标记或抛出 | Infrastructure |
| `resetAlarm()` 在报警未消除时调用 | 调用无效果，`alarmed` 保持 true | Infrastructure |
| 返回的 `position` 为 NaN/Inf | GantrySystem 在 `aggregateState()` 中检测并触发 Alarm | Domain |

### 2.7 时序约束

```
PLC 扫描周期 (典型 1~10ms):
 1. getX1Feedback() → PhysicalAxisState
 2. getX2Feedback() → PhysicalAxisState
 3. GantrySystem::aggregateState()  ← 注入状态快照
 4. GantrySystem::drainEvents()     ← 发布事件
 5. (如需) IGantryCommandPort 命令下发
```

## 3. IGantryCommandPort — 命令端口

### 3.1 业务语义

> **命令端口是 GantrySystem 将经过校验的运动意图传递给物理层的唯一通道。** 调用方必须先完成 `checkOperability()` 校验，确认 `Allowed` 后方可调用此端口。**耦合模式下的命令拆分由端口实现层负责**：逻辑方向/位置需转换为 X1/X2 的镜像指令。

### 3.2 接口定义

```cpp
class IGantryCommandPort {
public:
    virtual ~IGantryCommandPort() = default;
    // 单轴命令 (Decoupled 模式)
    virtual bool jogAxis(int axis, MotionDirection direction) = 0;
    virtual bool moveAbsoluteAxis(int axis, double position) = 0;
    virtual bool moveRelativeAxis(int axis, double delta) = 0;
    virtual bool stopAxis(int axis) = 0;
    // 龙门命令 (Coupled 模式)
    virtual bool jogGantry(MotionDirection direction) = 0;
    virtual bool moveAbsoluteGantry(double position) = 0;
    virtual bool moveRelativeGantry(double delta) = 0;
    virtual bool stopGantry() = 0;
    // 通用
    virtual bool isAxisSlotFree(int axis) const = 0;
};
```

### 3.3 前置条件

| # | 条件 | 违反后果 | 约束 |
|---|------|----------|------|
| P-CMD-01 | 调用前**必须**通过 `checkOperability()` 校验 | 绕过安全保护 | 约束15-19 |
| P-CMD-02 | `axis` 只能是 1(X1) 或 2(X2) | 未定义行为 | 约束4 |
| P-CMD-03 | 龙门命令仅 `Coupled` 下调用 | X2 未镜像运动 | 约束3 |
| P-CMD-04 | 单轴命令仅 `Decoupled` 下调用 | 破坏联动同步 | 约束4 |
| P-CMD-05 | 先确认 `isAxisSlotFree()` 返回 true | 命令被拒绝 | 约束19 |
| P-CMD-06 | `stopGantry()` 无条件接受 | — | — |

### 3.4 后置条件

| # | 条件 | 约束 |
|---|------|------|
| Q-CMD-01 | `jogAxis(1, Forward)` → X1 正向运动 | 约束6 |
| Q-CMD-02 | `jogAxis(1, Backward)` → X1 负向运动 | 约束6 |
| Q-CMD-03 | `jogGantry(Forward)` → X1 正向 + X2 负向（镜像） | 约束3,6,9 |
| Q-CMD-04 | `jogGantry(Backward)` → X1 负向 + X2 正向（镜像） | 约束3,6,9 |
| Q-CMD-05 | `moveAbsoluteGantry(p)` → X1 目标=p, X2 目标=-p | 约束3,9,10 |
| Q-CMD-06 | `moveRelativeGantry(d)` → X1 位移=d, X2 位移=-d | 约束3,9 |
| Q-CMD-07 | `stopGantry()` → X1/X2 同时制动 | — |
| Q-CMD-08 | 返回 `true` 表示已被底层接受，不代表运动完成 | 原则9 |
| Q-CMD-09 | `jogAxis()` 新命令可覆盖未完成的 Jog | 约束19 |

### 3.5 不变式

| # | 不变式 | 约束 |
|---|--------|------|
| I-CMD-01 | 端口不执行业务规则检查，只传输命令 | 原则9 |
| I-CMD-02 | Coupled 模式 X1/X2 命令原子下发 | 约束14 |
| I-CMD-03 | 端口不得缓存或延迟命令 | — |
| I-CMD-04 | 镜像拆分是端口实现层职责，Domain 不感知 | 约束7 |

### 3.6 Coupled 模式命令拆分规则

| 龙门命令 | X1 指令 | X2 指令 | 说明 |
|----------|---------|---------|------|
| `jogGantry(Forward)` | `jogAxis(1,Forward)` | `jogAxis(2,Backward)` | X2 方向取反 |
| `jogGantry(Backward)` | `jogAxis(1,Backward)` | `jogAxis(2,Forward)` | X2 方向取反 |
| `moveAbsoluteGantry(p)` | `moveAbsoluteAxis(1,p)` | `moveAbsoluteAxis(2,-p)` | X2 位置取反 |
| `moveRelativeGantry(d)` | `moveRelativeAxis(1,d)` | `moveRelativeAxis(2,-d)` | X2 位移取反 |
| `stopGantry()` | `stopAxis(1)` | `stopAxis(2)` | 同步停止 |

### 3.7 错误场景

| 场景 | 行为 | 处理方 |
|------|------|--------|
| 物理轴正忙 | 返回 `false` | Infrastructure |
| axis 参数非法 | 返回 `false` 或断言 | Infrastructure |
| Coupled 模式 X1 成功 X2 失败 | 立即对 X1 下发 Stop | Infrastructure |
| 对已停止的轴 stop | 返回 `true`（幂等） | Infrastructure |

## 4. IGantryEventBus — 事件总线端口

### 4.1 业务语义

> **事件总线端口是领域事件从 Domain 流向 Application/Presentation 的通道。** 每个 PLC 扫描周期，调用方从 `GantrySystem::drainEvents()` 获取事件，通过此端口发布给上层订阅者。

### 4.2 接口定义

```cpp
class IGantryEventBus {
public:
    virtual ~IGantryEventBus() = default;
    virtual void publish(const GantryEvents::Event& event) = 0;
    virtual void publishAll(const std::vector<GantryEvents::Event>& events) {
        for (const auto& e : events) { publish(e); }
    }
};
```

### 4.3 业务约束

| # | 条件 |
|---|------|
| P-EVT-01 | 事件先通过 `drainEvents()` 取出，再发布 |
| P-EVT-02 | `publishAll()` 保持事件原始顺序 |
| Q-EVT-01 | `publish()` 后事件对所有订阅者可见 |
| Q-EVT-02 | 事件发布后不可修改（不可变值对象） |
| I-EVT-01 | 端口不修改事件内容 |
| I-EVT-02 | 端口不决定消费逻辑 |

### 4.4 事件类型与消费方

| 事件类型 | 触发时机 | 建议消费方 |
|----------|----------|------------|
| `Coupled` | 联动建立成功 | UI 模式指示、操作面板切换 |
| `Decoupled` | 分动建立/强制退出联动 | UI 模式指示 |
| `DeviationFault` | 联动中位置偏差超限 | 报警系统、日志 |
| `CouplingRefused` | 联动申请被拒绝 | UI 诊断提示、日志 |
| `LimitTriggered` | 物理轴触发限位 | 限位指示灯 |
| `AlarmRaised` | 物理轴报警 | 报警弹窗 |
| `CommandRejected` | 命令被安全检查拒绝 | 日志、诊断 |
| `PositionUpdated` | 龙门位置更新 | 位置显示、趋势图 |
| `CouplingRequested` | 联动申请发起 | 日志、审计 |

### 5.1 业务语义

> 内部端口，用于 GantrySystem 事件队列管理。与 `IGantryEventBus` 的区别：Publisher 管理内部队列写入，EventBus 向外部广播。

### 5.2 接口定义

```cpp
class IGantryEventPublisher {
public:
    virtual ~IGantryEventPublisher() = default;
    virtual void publish(const GantryEvents::Event& event) = 0;
    virtual void flush() {}
};
```

### 5.3 约束

- P-PUB-01: 同一事件同周期最多发布一次
- P-PUB-02: `publish()` 不触发副作用
- I-PUB-01: 事件入队幂等

## 6. IGantryStateQuery — 状态查询端口

### 6.1 业务语义

> Application/Presentation 层获取龙门当前状态的只读窗口。使上层无需了解 PhysicalAxis X1/X2 内部结构。

### 6.2 接口定义

```cpp
class IGantryStateQuery {
public:
    virtual ~IGantryStateQuery() = default;
    virtual GantryMode mode() const = 0;
    virtual AxisState aggregatedState() const = 0;
    virtual bool isCoupled() const = 0;
    virtual double position() const = 0;
    virtual double x1Position() const = 0;
    virtual double x2Position() const = 0;
    virtual bool x1Enabled() const = 0;
    virtual bool x2Enabled() const = 0;
    virtual bool isAnyAlarm() const = 0;
    virtual bool isAnyLimit() const = 0;
    virtual bool canAcceptCommand() const = 0;
    virtual bool isTargetOperable(AxisId target) const = 0;
    virtual std::string stateDescription() const = 0;
};
```

### 6.3 约束

| # | 不变式 |
|---|--------|
| I-QRY-01 | `isCoupled() == (mode() == GantryMode::Coupled)` |
| I-QRY-02 | `isAnyAlarm()` → `aggregatedState() == AxisState::Error` |
| I-QRY-03 | `canAcceptCommand() == false` → 任何运动命令拒绝 |
| P-QRY-01 | 所有查询方法为 `const`，不得修改状态 |
| P-QRY-02 | 返回最近一次 `aggregateState()` 后的结果 |

## 7. GantrySystem API — 聚合根对外契约

### 7.1 业务语义

> GantrySystem 是龙门子系统唯一的聚合根，所有操作必须通过其公开方法。外部不得直接访问 PhysicalAxis X1/X2（设计铁律 #2）。

### 7.2 对外 API 分类

#### 7.2.1 模式管理

| 方法 | 前置条件 | 后置条件 | 约束 |
|------|----------|----------|------|
| `mode()` | 无 | 返回当前 GantryMode | 约束1 |
| `requestCoupling()` | 无 | 条件满足→`Coupled`+事件；否则返回拒绝原因 | 约束12,13 |
| `requestDecoupling(reason)` | 无 | 若`Coupled`→`Decoupled`+事件 | 约束4 |

#### 7.2.2 操作目标互斥

| Mode | `AxisId::X` | `AxisId::X1` | `AxisId::X2` |
|------|:---:|:---:|:---:|
| `Coupled` | ✅ 可操作 | ❌ 不可操作 | ❌ 不可操作 |
| `Decoupled` | ❌ 只读 | ✅ 可操作 | ✅ 可操作 |

#### 7.2.3 运动命令

| 方法 | 前置条件 | 约束 |
|------|----------|------|
| `jog(target, dir)` | `checkOperability() == Allowed` | 3,4,16,19 |
| `moveAbsolute(target, pos)` | `checkOperabilityForMove() == Allowed` | 3,4,15,19 |
| `moveRelative(target, delta)` | `checkOperabilityForMove() == Allowed` | 3,4,15,19 |
| `stop(target)` | 无（始终接受） | — |

**Jog 限位（约束16）：** 正限位→禁Forward允Backward；负限位→禁Backward允Forward

**Move 限位（约束15）：** 任何限位触发→禁止 Move

#### 7.2.4 状态聚合

`aggregateState()`: X1/X2 物理状态注入 → 更新 X 聚合状态 + 联动维持检查 + 边沿事件（约束14,20）

**聚合优先级：** `Alarm > Limit > Moving(Jog/MoveAbs/MoveRel) > Idle`

### 7.3 不变式

| # | 不变式 | 约束 |
|---|--------|------|
| I-SYS-01 | `mode()` 只能是 `Coupled` 或 `Decoupled` | 约束2 |
| I-SYS-02 | `mode==Coupled` ⇔ `isTargetOperable(X)==true` | 约束3 |
| I-SYS-03 | `mode==Decoupled` ⇔ X1/X2 均可操作 | 约束4 |
| I-SYS-04 | X 逻辑位置 ≡ X1 物理位置 | 约束10 |
| I-SYS-05 | `aggregateState()` 后偏差超阈值→强制退出联动 | 约束14 |
| I-SYS-06 | 报警/限位优先反映异常状态 | 约束15,17,20 |

## 9. Operability 与 CommandResult

### 9.1 业务语义

> `Operability` 是操作可行性判定结果。任何外部调用方在尝试运动命令前，**必须**通过 `checkOperability()` 验证。

### 9.2 枚举定义

```cpp
enum class Operability {
    Allowed,
    TargetNotOperableInMode,  // 约束18
    AlarmActive,              // 约束17
    LimitTriggered,           // 约束15
    LimitBlocksDirection,     // 约束16
    NotEnabled,               // 约束13.1
    CommandSlotBusy,          // 约束19
    NotIdle,                  // 约束19
    DeviationExceeded,        // 约束14
};
```

### 9.3 优先顺序

```
AlarmActive > LimitTriggered/LimitBlocksDirection > NotEnabled
  > TargetNotOperableInMode > NotIdle > CommandSlotBusy > DeviationExceeded
```

### 9.4 CommandResult

```cpp
struct CommandResult {
    Operability verdict;
    std::string detail;
    bool isAccepted() const { return verdict == Operability::Allowed; }
    static CommandResult accept();
    static CommandResult reject(Operability reason, const std::string& d);
};
```

### 9.5 命令下发流程

```
外部调用方
  ├─1. checkOperability(target, dir) → CommandResult
  ├─2. if accepted → GantrySystem::jog(target, dir) → 写入命令槽
  └─3. IGantryCommandPort 下发物理命令
```

| 约束# | 约束名称 | 相关接口/方法 |

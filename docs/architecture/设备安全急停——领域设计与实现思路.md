# 设备安全急停 — 领域设计与实现思路（修订版 v3）

> 状态：设计文档（非实现）
> 适用范围：`servoV6` 项目 `domain/safety/` 新领域 + 现有架构适配
>
> 修订记录：
> - v1：二态模型（Normal / EmergencyStopped）

> - v3：四态模型（移除 SafeIdle）— 最终版本

## 一、前置分析：当前项目结构与急停的关系

### 1.1 当前命令流路径

```
UI / Application
     │
     ▼
Orchestrator (e.g. JogOrchestrator)
     │
     ├─ ① SystemManager::tryGetGroup()      → 分组路由
     ├─ ② SystemContext::tryGetAxis()       → 龙门语义 + 轴访问裁决
     ├─ ③ Axis::jog() / enable() ...        → 领域层语义校验
     └─ ④ ISystemDriver::send(SystemCommand) → 下发到物理层
              │
              ▼
         FakePLC (物理寄存器仿真)
```

**关键发现：** 目前没有任何一层在命令执行前检查"系统是否处于急停状态"。

### 1.2 servov6 的运行语义 — 去掉 SafeIdle 的根本原因

当前系统采用**"默认 Disabled + 按需自动使能"**架构：

```text
当前 servov6 实际的运行模型：
  用户发起运动
      ↓
  Orchestrator 自动 EnsureEnabled
      ↓
  自动上电
      ↓
  执行运动
```

系统天然以 `AxisState::Disabled` 作为待机态，而非以 `Idle`（已使能等待运动）。这意味着：

- `Disabled` 本身就是"安全空闲/待机/未使能"语义
- `Enable` 属于 **Axis 生命周期**，不属于 **Safety 生命周期**

因此，急停解除后系统只需恢复**运行许可**（SafetyState → Running），而轴是否使能由 Axis 域自己管理。不需要一个独立的 `SafeIdle` 状态来同时表达"系统安全许可恢复但轴还未使能"——这会与 `AxisState::Disabled` 产生**跨层语义重复**，破坏正交状态机架构。

### 1.3 当前 SystemContext 持有的东西

```cpp
class SystemContext {
    unordered_map<AxisId, unique_ptr<Axis>> m_axes;        // 6 个轴实体
    unique_ptr<GantryCouplingController> m_gantryCoupling;  // 龙门联动状态机
    unique_ptr<GantryPowerController>   m_gantryPower;      // 龙门电机使能控制
    ISystemDriver* m_driver;                                 // 驱动指针
};
```

**缺失的东西：** 没有 `EmergencyStopController`，没有系统级安全状态。

### 1.4 PLC 当前的急停寄存器（设计依据）

PLC 已经天然支持**命令与状态反馈分离**：

| PLC 寄存器 | 语义 | 方向 |
|-----------|------|------|
| `设备急停` | 上位机写入的急停**命令** | Host → PLC |
| `设备急停中` | PLC 反馈的急停**状态** | PLC → Host |

这正好适配**请求 → 等待反馈 → 确认状态**的异步流程。

## 二、急停的本质（来自 PLC 文档分析）

从 PLC 逻辑看，急停的三个核心行为：

| 行为 | PLC 代码 | 含义 |
|------|----------|------|
| **电源裁决** | `轴X使能 := 使能轴X电机 AND NOT 设备急停` | 急停 = 全轴立即掉电 |
| **动作熔断** | `轴X点动正转 := FALSE; ...` | 急停 = 清空所有运动命令 |
| **状态标记** | `设备急停中` 状态位 | PLC 维护急停状态机 |

从领域建模角度看，急停**不是**：
- ❌ 单轴行为（不是 `Axis::emergencyStop()`）
- ❌ 龙门特定行为（不是 `GantryCouplingController` 的子功能）
- ❌ 业务编排逻辑（不是 `Policy/Orchestrator` 层）
- ❌ 纯基础设施（PLC 只是物理执行者）

急停**是**：
- ✅ **安全域（Safety Domain）** — 独立领域，优先级高于所有运动控制
- ✅ **横切关注点** — 需要被 SystemContext 感知，影响所有轴的访问语义
- ✅ **系统状态** — 不仅仅是一个按钮事件，而是系统进入安全模式的全局标记
- ✅ **多阶段流程** — 不是"按一下急停、按一下恢复"的简单按钮逻辑

## 三、系统安全状态机：四态设计

### 3.1 完整状态流转图

```
                         ┌──────────────────────┐
                         │                      │
                         ▼                      │
  ┌─────────┐  用户触发急停   ┌──────────────────┐  │
  │ Running │ ─────────────→ │EmergencyStopping  │  │
  │         │                │ (等待PLC完成停机)  │  │
  └─────────┘                └────────┬─────────┘  │
       ▲                              │            │
       │                          PLC反馈确认       │
       │                    设备急停中 = TRUE       │
       │                              │            │
       │                              ▼            │
       │                   ┌──────────────────┐    │
       │                   │EmergencyStopped   │    │
       │                   │ (系统已锁定)       │───┘│
       │                   └────────┬─────────┘    │ 重复触发急停（幂等）
       │                            │              │
       │                       用户解除急停         │
       │                            │              │
       │                            ▼              │
       │                   ┌──────────────────┐    │
       │                   │ReleasingEmergency │   │
       │                   │Stop               │   │
       │                   │(等待PLC解除锁定)   │   │
       │                   └────────┬─────────┘    │
       │                            │              │
       │                        PLC反馈确认         │
       │                    设备急停中 = FALSE      │
       │                            │              │
       └────────────────────────────┘              │
```

**关键点：** `ReleasingEmergencyStop` + `plcEmergencyStopped == false` → 直接回到 `Running`。系统恢复运行许可，但轴的使能状态由 Axis 域独立管理（servoV6 默认 Disabled）。

### 3.2 四个状态的定义

| 状态 | 含义 | PLC 对应 | 运动控制 | 轴使能状态 |
|------|------|---------|---------|-----------|
| **Running** | 系统运行许可有效 | `设备急停中 = FALSE` | 允许一切运动 | 默认 Disabled，由 EnsureEnabled 自动使能 |
| **EmergencyStopping** | 急停命令已下发，等待 PLC 确认 | `设备急停 = TRUE`，等待 `设备急停中 = TRUE` | 禁止一切运动 | PLC 强制全轴掉电 |
| **EmergencyStopped** | PLC 确认系统已停机锁定 | `设备急停中 = TRUE` | 禁止一切运动和使能 | 全轴 Disabled（PLC 强置） |
| **ReleasingEmergencyStop** | 解除命令已下发，等待 PLC 确认 | `设备急停 = FALSE`，等待 `设备急停中 = FALSE` | 禁止一切运动 | PLC 解除锁定，轴保持 Disabled |

### 3.3 为什么是四态而非二态

| 错误设计（二态） | 正确设计（四态） | 原因 |
|-----------------|------------------|------|
| 按下急停 → 立即进入 EmergencyStopped | 按下急停 → EmergencyStopping → 等待 PLC 反馈 → EmergencyStopped | 上位机状态必须与 PLC 物理状态一致 |
| 解除急停 → 立即回到 Running | 解除急停 → ReleasingEmergencyStop → 等待 PLC 反馈 → Running | 防上位机/PLC 状态不一致 |

### 3.4 为什么不需要 SafeIdle — 正交状态机原则

```
servoV6 的多正交状态机架构：

  SafetyState   → 只负责：系统运行许可（是否允许运动）
  AxisState     → 只负责：轴当前是否已使能/运动（Disabled/Idle/Jogging/...）
  GantryState   → 只负责：龙门同步状态（NotSynchronized/Coupled/Decoupled）
```

`SafeIdle` 被移除的根本原因：

| 问题 | 说明 |
|------|------|
| **语义重复** | `SafeIdle`（"急停已解除但轴未使能"）与 `AxisState::Disabled` 表达同一件事 |
| **跨层表达** | Safety 状态机不应该表达 Axis 域的状态（轴是否使能） |
| **职责模糊** | 后续会不断出现"到底应该检查 SafeIdle 还是 Disabled？"的困惑 |
| **架构已有承载** | servoV6 默认 Disabled + EnsureEnabled 自动使能，轴未 ready 本身就是 Disabled |

**正确做法：** 急停解除后，系统直接回到 `Running`（运行许可恢复）。轴仍是 `Disabled` 是正常现象——因为 servoV6 默认就是 Disabled，Enable 由运动操作的 EnsureEnabled 自动触发。这完全符合当前架构。

#### 4.2.1 `SafetyState.h` — 四态安全状态枚举

```
domain/safety/SafetyState.h
```

```cpp
#pragma once

/**
 * @brief 系统安全状态 — 四态工业安全状态机
 *
 * 设计依据：
 *   1. PLC 拥有独立寄存器：设备急停（命令） / 设备急停中（状态反馈）
 *   2. 上位机状态必须与 PLC 物理状态保持最终一致
 *   3. 只表达"系统运行许可"，不表达轴使能状态（轴生命周期由 AxisState 负责）
 *
 * 状态流转：
 *   Running → EmergencyStopping → EmergencyStopped → ReleasingEmergencyStop → Running
 *
 * ╔══════════════════════════════════════════════════════════════╗
 * ║  核心原则：                                                  ║
 * ║  1. 上位机状态由 PLC 反馈确认后变更，不由请求立即变更         ║
 * ║  2. 安全状态只负责"系统是否允许运动"，不负责"轴是否已使能"   ║
 * ║  3. servoV6 默认 Disabled + EnsureEnabled，轴生命周期独立     ║
 * ╚══════════════════════════════════════════════════════════════╝
 */
enum class SafetyState {
    // ==========================================
    // 状态 1：Running — 系统运行许可有效
    // ==========================================
    // • 所有轴可运动、可使能
    // • 轴默认 Disabled，运动时由 EnsureEnabled 自动使能
    // • PLC 寄存器：设备急停中 = FALSE
    Running,

    // ==========================================
    // 状态 2：EmergencyStopping — 急停命令已下发，等待 PLC 停机确认
    // ==========================================
    // • 禁止一切运动和使能
    // • UI 显示"急停处理中..."
    // • PLC 寄存器：设备急停 = TRUE，设备急停中 可能仍为 FALSE（尚未完成）
    EmergencyStopping,

    // ==========================================
    // 状态 3：EmergencyStopped — 系统已锁定
    // ==========================================
    // • 禁止一切运动和使能
    // • UI 显示"设备已急停"，运动控件全部置灰
    // • 状态信息（轴位置、PLC 连接状态等）仍可见
    // • PLC 寄存器：设备急停中 = TRUE
    EmergencyStopped,

    // ==========================================
    // 状态 4：ReleasingEmergencyStop — 解除命令已下发，等待 PLC 解锁确认
    // ==========================================
    // • 禁止一切运动
    // • UI 显示"急停解除中..."
    // • PLC 寄存器：设备急停 = FALSE，设备急停中 可能仍为 TRUE（尚未解除）
    ReleasingEmergencyStop
};
```

**设计决策：**

- **为什么是四态不是五态？**
  - servoV6 默认 Disabled + EnsureEnabled 架构，`Disabled` 本身就是"安全空闲/待机/未使能"
  - 不需要 `SafeIdle` 来重复表达"轴未使能"语义（与 `AxisState::Disabled` 重复）
  - Safety 状态机只负责"运行许可"，Axis 状态机负责"使能状态"，职责边界清晰

- **急停解除后直接回到 Running（而不是 SafeIdle）？**
  - `ReleasingEmergencyStop` + `plcEmergencyStopped == false` → `Running`
  - 此时轴全部 Disabled（PLC 急停时清空了所有使能），系统回到默认状态
  - 用户想要运动时，Orchestrator 的 EnsureEnabled 自动使能——这正是 servoV6 的核心工作流

#### 4.2.2 `SafetyRejection.h` — 安全域拒绝原因

```
domain/safety/SafetyRejection.h
```

```cpp
#pragma once

/**
 * @brief 安全域操作拒绝原因
 *
 * 独立于 ContextRejection / GantryRejection / RejectionReason，
 * 因为安全域语义与其他域完全不同，保持领域边界清晰。
 */
enum class SafetyRejection {
    None,                       // 无拒绝（操作允许）

    // --- 状态机拒绝 ---
    SystemSafetyLocked,         // 系统处于 EmergencyStopping / EmergencyStopped / ReleasingEmergencyStop 状态
    AlreadyInState,             // 幂等保护：请求状态 = 当前状态（如已在急停中再次触发急停）
    InvalidStateTransition,    // 非法状态跃迁（如从 Running 直接请求 ReleasingEmergencyStop）

    // --- 前置条件拒绝 ---
    NotEmergencyStopped,        // 尝试解除急停但系统并未处于急停状态
};
```

#### 4.2.3 `EmergencyStopController.h` — 急停状态机（核心聚合根）

```
domain/safety/EmergencyStopController.h
```

这是整个急停功能的核心。它**管理四态安全状态机**，遵循项目中 `GantryCouplingController` 已确立的模式：**意图生成 → 命令下发 → PLC 反馈 → 状态确认**。

```cpp
#pragma once

#include "safety/SafetyState.h"
#include "safety/SafetyRejection.h"
#include "command/SystemCommand.h"  // EmergencyStopCommand
#include <optional>

/**
 * @brief 急停控制器 — 四态工业安全状态机
 *
 * 设计原则：
 *   1. 反馈驱动 — 状态变更由 PLC 反馈确认，不由请求立即变更
 *   2. 不持有 Axis 引用 — 轴的访问拒绝由 SystemContext::tryGetAxis() 实现
 *   3. 幂等安全 — 重复触发/解除急停不产生副作用
 *   4. 命令与状态分离 — PLC 有独立的"急停命令"寄存器和"急停中"状态寄存器
 *   5. 安全域不管理轴生命周期 — 急停解除后直接回到 Running，轴使能由 Axis 域管理
 */
class EmergencyStopController {
public:
    EmergencyStopController() = default;

    // ==========================================
    // 状态查询
    // ==========================================

    SafetyState state() const { return m_state; }

    /// @brief 系统是否处于"锁定"状态（任何运动都不允许）
    /// @return true 当状态为 EmergencyStopping / EmergencyStopped / ReleasingEmergencyStop
    bool isSystemLocked() const {
        return m_state == SafetyState::EmergencyStopping ||
               m_state == SafetyState::EmergencyStopped ||
               m_state == SafetyState::ReleasingEmergencyStop;
    }

    /// @brief 系统是否处于急停锁定态（PLC 已确认停机）
    bool isEmergencyStopped() const {
        return m_state == SafetyState::EmergencyStopped;
    }

    /// @brief 系统是否正在过渡中（等待 PLC 反馈）
    bool isTransitioning() const {
        return m_state == SafetyState::EmergencyStopping ||
               m_state == SafetyState::ReleasingEmergencyStop;
    }

    // ==========================================
    // 意图生成（Produce Intent）
    // ==========================================

    /**
     * @brief 请求触发急停
     *
     * 合法来源状态：Running
     * 产生意图：EmergencyStopCommand{ true }
     * 本地状态 → EmergencyStopping
     *
     * @return None 表示命令已生成，等待 PLC 反馈；AlreadyInState 表示已在急停流程中
     */
    SafetyRejection requestEmergencyStop() {
        // 幂等：已在急停流程中
        if (m_state == SafetyState::EmergencyStopping ||
            m_state == SafetyState::EmergencyStopped) {
            return SafetyRejection::AlreadyInState;
        }
        // 冲突：正在解除急停，不允许反向操作
        if (m_state == SafetyState::ReleasingEmergencyStop) {
            return SafetyRejection::InvalidStateTransition;
        }
        // 通过：Running → 生成急停意图
        m_pending_intent = EmergencyStopCommand{ true };
        m_state = SafetyState::EmergencyStopping;
        return SafetyRejection::None;
    }

    /**
     * @brief 请求解除急停
     *
     * 合法来源状态：EmergencyStopped
     * 产生意图：EmergencyStopCommand{ false }
     * 本地状态 → ReleasingEmergencyStop
     *
     * @return None 表示命令已生成，等待 PLC 反馈
     */
    SafetyRejection requestReleaseEmergencyStop() {
        // 前置条件：只有 EmergencyStopped 状态才能解除
        if (m_state != SafetyState::EmergencyStopped) {
            return SafetyRejection::NotEmergencyStopped;
        }
        m_pending_intent = EmergencyStopCommand{ false };
        m_state = SafetyState::ReleasingEmergencyStop;
        return SafetyRejection::None;
    }

    // ==========================================
    // 反馈驱动（Apply Feedback）— 状态确认的核心
    // ==========================================

    /**
     * @brief 接收 PLC 的"急停中"状态反馈，驱动本地状态机
     *
     * @param plcEmergencyStopped 对应 PLC 寄存器"设备急停中"
     *
     * 状态跃迁规则：
     *   EmergencyStopping  + plcEmergencyStopped == true  → EmergencyStopped
     *   ReleasingEmergencyStop + plcEmergencyStopped == false → Running（直接恢复）
     *   其他状态 + 任何反馈 → 保持不变（等待正确状态的到来）
     */
    void applyFeedback(bool plcEmergencyStopped) {
        switch (m_state) {
        case SafetyState::EmergencyStopping:
            // 等待 PLC 确认急停完成
            if (plcEmergencyStopped) {
                m_state = SafetyState::EmergencyStopped;
            }
            // 如果 PLC 尚未反馈 true，保持 EmergencyStopping 继续等待
            break;

        case SafetyState::EmergencyStopped:
            // 急停锁定中，PLC 持续反馈 true 是正常的
            // 如果 PLC 意外反馈 false（理论上不应发生），保持 EmergencyStopped
            break;

        case SafetyState::ReleasingEmergencyStop:
            // 等待 PLC 确认急停解除 → 直接恢复 Running
            // 不经过任何中间状态，因为 servoV6 默认 Disabled，
            // 轴的使能生命周期由 Axis 域独立管理
            if (!plcEmergencyStopped) {
                m_state = SafetyState::Running;
            }
            // 如果 PLC 尚未反馈 false，保持 ReleasingEmergencyStop 继续等待
            break;

        case SafetyState::Running:
            // 正常运行状态下收到 PLC 反馈 true（物理急停按钮被按下）
            // 此种情况由外部检测逻辑处理，本控制器不响应
            break;
        }
    }

    // ==========================================
    // 命令消费（Pop Intent）
    // ==========================================

    bool hasPendingCommand() const {
        return m_pending_intent.has_value();
    }

    EmergencyStopCommand popPendingCommand() {
        auto cmd = *m_pending_intent;
        m_pending_intent.reset();
        return cmd;
    }

private:
    SafetyState m_state = SafetyState::Running;
    std::optional<EmergencyStopCommand> m_pending_intent;
};
```

**关键设计决策：**

1. **`requestReleaseEmergencyStop()` 而非 `requestNormal()`。**
   - `requestNormal` 的语义暗示"恢复到正常"，模糊不清
   - `requestReleaseEmergencyStop` 明确表达"请求解除急停锁定"，结果状态是 `ReleasingEmergencyStop`

2. **`ReleasingEmergencyStop` + `plcEmergencyStopped == false` → 直接 `Running`。**
   - 不经过 `SafeIdle`，因为 servoV6 默认 Disabled 已经是天然的安全空闲态
   - Safety 状态机只恢复运行许可，轴的使能由 Axis 域独立管理

3. **没有 `confirmSystemResumed()` 方法。**
   - v2 中从 `SafeIdle` 到 `Running` 需要调用此方法，标记"用户已完成重新使能"
   - v3 不需要：轴使能是 Axis 生命周期的一部分，由 EnsureEnabled 自动触发
   - 系统运行许可恢复（Running）后，第一个运动命令的 EnsureEnabled 自动使能轴——这正是 servoV6 的核心流

4. **`isSystemLocked()` 方法。**
   - `EmergencyStopping`、`EmergencyStopped`、`ReleasingEmergencyStop` 三个状态下系统都被锁定
   - 这是 `SystemContext::tryGetAxis()` 使用的统一拦截条件

5. **不持有任何 Axis 引用。**
   - 急停控制器只管理自身状态机，不直接操作任何轴
   - 遵循单一职责原则

#### 4.2.4 `EmergencyStopCommand` — 急停命令类型

定义在 `domain/command/SystemCommand.h` 中（无需独立文件）：

```cpp
/**
 * @brief 急停命令（对应 PLC 寄存器"设备急停"）
 *
 * 注意：
 *   - 这是写入 PLC 的"命令"，不是 PLC 反馈的"状态"
 *   - PLC 反馈状态通过独立的"设备急停中"寄存器获取
 *   - 命令与状态的分离是实现四态安全状态机的基础
 */
struct EmergencyStopCommand {
    bool active; // true: 触发急停, false: 解除急停锁定
};
```

同时更新 `SystemCommand` variant：

```cpp
using SystemCommand = std::variant<
    AxisCommandWithId,
    GantryCouplingCommand,
    GantryPowerCommand,
    EmergencyStopCommand          // ← 新增
>;
```

**为什么放入 SystemCommand？**
- 急停命令需要走统一命令总线 → Driver → PLC 的路径
- 与现有命令的路径完全一致，不新开通信通道
- 但强调：`EmergencyStopCommand` 写入的是 PLC 的"设备急停"命令寄存器，而状态由"设备急停中"反馈寄存器独立提供

#### 4.3.1 `ContextRejection.h` — 新增安全锁定拒绝原因

#### 4.3.5 `FakePLC.h` — 新增急停寄存器仿真（命令与状态分离）

```cpp
class FakePLC {
private:
    // ═══════════════════════════════════════════
    // 急停相关寄存器（命令与状态分离）
    // ═══════════════════════════════════════════
    bool m_emergencyStopCommand  = false;  // 对应 PLC 寄存器"设备急停"
    bool m_emergencyStoppedState = false;  // 对应 PLC 寄存器"设备急停中"

public:
    /**
     * @brief 处理急停命令（写入"设备急停"寄存器）
     *
     * 模拟 PLC 行为：
     *   active = true  → 立即掉电所有轴 + 清空动作 → "设备急停中" = true
     *   active = false → 解除急停锁定 → "设备急停中" = false
     *                    （轴保持 Disabled，不自动恢复运行）
     */
    void onEmergencyStopCommand(const EmergencyStopCommand& cmd) {
        m_emergencyStopCommand = cmd.active;
        if (cmd.active) {
            // 模拟 PLC 急停动作：立即掉电所有轴 + 清空运动
            for (auto& [id, axis] : m_axes) {
                axis.feedback.state = AxisState::Disabled;
                axis.target_pos = axis.feedback.absPos;
                axis.stop_requested = false;
            }
            m_emergencyStoppedState = true;
        } else {
            m_emergencyStoppedState = false;
            // 注意：轴保持 Disabled 状态，不自动恢复
            // 这模拟了 PLC 的 AND NOT 设备急停 逻辑
        }
    }

    /// @brief 读取 PLC 的"设备急停中"状态反馈寄存器
    bool isEmergencyStopped() const { return m_emergencyStoppedState; }

    /// @brief 读取 PLC 的"设备急停"命令寄存器（用于调试/测试）
    bool emergencyStopCommandActive() const { return m_emergencyStopCommand; }
};
```

**关键点：** `onEmergencyStopCommand(false)` 只清除急停锁定，不恢复轴的使能状态。轴全部保持在 `Disabled`。这模拟了 PLC 的真实行为，也与 servoV6 的"默认 Disabled"架构一致。

## 五、完整命令流：四个阶段的链路

### 5.1 阶段 1：用户触发急停（Running → EmergencyStopping）

```
UI: 用户点击急停按钮
        │
        ▼
EmergencyStopUseCase::execute()
        │
        ├─ ① EmergencyStopController::requestEmergencyStop()
        │       → 产生 EmergencyStopCommand{ true }
        │       → 本地状态 Running → EmergencyStopping
        │
        ├─ ② SystemContext::driver()->send(EmergencyStopCommand{ true })
        │       → FakePLC::onEmergencyStopCommand(true)
        │       → 所有轴立即 Disabled + 清空动作
        │       → "设备急停中" = true
        │
        └─ ③ 下一帧反馈循环：
                m_emergencyStop->applyFeedback(true)  // plcEmergencyStopped = true
                → 本地状态 EmergencyStopping → EmergencyStopped
```

**UI 行为：**
- 显示"急停处理中..."
- 所有运动控件（Jog / Move / Enable）置灰
- 状态信息（轴位置、伺服状态、PLC 连接状态）仍可见

### 5.2 阶段 2：PLC 确认急停完成（EmergencyStopping → EmergencyStopped）

```
反馈循环（每帧执行）：
    SystemContext::emergencyStopController().applyFeedback(
        FakePLC::isEmergencyStopped()  // → true
    )
    → 状态跃迁 EmergencyStopping → EmergencyStopped
```

**UI 行为：**
- 显示"设备已急停"
- 所有运动控件持续置灰
- 状态信息持续更新显示
- 解除急停按钮变为可操作

**拦截效果：**
- `SystemContext::tryGetAxis()` → `isSystemLocked() == true` → 返回 `SystemSafetyLocked`
- 所有 Orchestrator 的 `tick()` 在第一层拦截就失败，直接进入 Error 步骤

### 5.3 阶段 3：用户解除急停（EmergencyStopped → ReleasingEmergencyStop → Running）

```
UI: 用户点击"解除急停"按钮
        │
        ▼
ReleaseEmergencyStopUseCase::execute()
        │
        ├─ ① EmergencyStopController::requestReleaseEmergencyStop()
        │       → 产生 EmergencyStopCommand{ false }
        │       → 本地状态 EmergencyStopped → ReleasingEmergencyStop
        │
        ├─ ② SystemContext::driver()->send(EmergencyStopCommand{ false })
        │       → FakePLC::onEmergencyStopCommand(false)
        │       → "设备急停中" = false
        │       → 轴保持 Disabled（不自动恢复——servoV6 默认 Disabled）
        │
        └─ ③ 下一帧反馈循环：
                m_emergencyStop->applyFeedback(false)  // plcEmergencyStopped = false
                → 本地状态 ReleasingEmergencyStop → Running
                → 系统运行许可恢复，但轴全部 Disabled

后续：用户发起运动 → Orchestrator::EnsureEnabled 自动使能轴 → 正常运动
```

**UI 行为：**
- 阶段 3a（ReleasingEmergencyStop）：显示"急停解除中..."，控件置灰
- 阶段 3b（Running）：正常运行界面，运动控件恢复可操作
  - 但轴全部 Disabled（servoV6 默认状态）→ 使能按钮显示为未使能
  - 用户发起运动时，EnsureEnabled 自动使能轴

**关键点：**
- 急停解除后系统直接回到 `Running`，没有中间 `SafeIdle` 状态
- 轴的使能恢复不由 Safety 域管理，由 Axis 域 + EnsureEnabled 自动完成
- Safety 状态机只负责"运行许可"，Axis 状态机负责"轴使能状态"——职责清晰正交

### 5.4 阶段 4（可选）：用户在 Running 状态下重新使能轴

```
UI: 用户单独点使能按钮 / 或通过 Orchestrator 自动使能
        │
        ▼
EnableUseCase::execute(axisId)
        │
        ├─ SystemContext::tryGetAxis() → isSystemLocked() == false ✓
        │
        └─ Axis::enable() → 正常使能轴 → Driver 下发使能命令
```

**或者，更常见的 servov6 工作流：**

```
用户发起运动（如 Jog / MoveAbsolute）
        │
        ▼
JogOrchestrator::tick() / MoveAbsoluteUseCase::execute()
        │
        ├─ SystemContext::tryGetAxis() → isSystemLocked() == false ✓
        │
        ├─ EnsureEnabled → 自动使能轴
        │       └─ Axis::enable() → Driver 下发使能命令
        │
        └─ 执行运动
```

系统运行许可已恢复（Running），Orchestrator 正常工作的第一个运动命令会自动触发 EnsureEnabled，无需用户额外点击使能按钮。这是 servoV6 的核心工作流。

## 六、Application 层 UseCase 设计

### 6.1 急停 UseCase — `EmergencyStopUseCase`

```cpp
// application/safety/EmergencyStopUseCase.h
#pragma once
#include "entity/SystemContext.h"
#include "UseCaseError.h"

class EmergencyStopUseCase {
public:
    explicit EmergencyStopUseCase(SystemContext& ctx) : m_ctx(ctx) {}

    UseCaseError execute() {
        // Step 1: 意图生成（Pre-flight Check）
        auto& controller = m_ctx.emergencyStopController();
        auto rejection = controller.requestEmergencyStop();
        if (rejection != SafetyRejection::None) {
            return rejection;  // AlreadyInState / InvalidStateTransition
        }

        // Step 2: 意图消费 → 命令下发
        if (!controller.hasPendingCommand()) {
            return UseCaseError{}; // 理论上不应到达
        }
        auto cmd = controller.popPendingCommand();
        m_ctx.driver()->send(cmd);

        // Step 3: 成功（状态已是 EmergencyStopping，等待反馈循环确认）
        return UseCaseError{};
    }

private:
    SystemContext& m_ctx;
};
```

**特点：**
- 无状态 — 每次调用完成一个完整的 UseCase
- 使用 feedback 的分离模式 — UseCase 只负责下发命令，状态确认由反馈循环完成
- Pre-flight Check 确保请求合法性

### 6.2 解除急停 UseCase — `ReleaseEmergencyStopUseCase`

```cpp
// application/safety/ReleaseEmergencyStopUseCase.h
#pragma once
#include "entity/SystemContext.h"
#include "UseCaseError.h"

class ReleaseEmergencyStopUseCase {
public:
    explicit ReleaseEmergencyStopUseCase(SystemContext& ctx) : m_ctx(ctx) {}

    UseCaseError execute() {
        // Step 1: 意图生成（Pre-flight Check）
        auto& controller = m_ctx.emergencyStopController();
        auto rejection = controller.requestReleaseEmergencyStop();
        if (rejection != SafetyRejection::None) {
            return rejection;  // NotEmergencyStopped
        }

        // Step 2: 意图消费 → 命令下发
        if (!controller.hasPendingCommand()) {
            return UseCaseError{};
        }
        auto cmd = controller.popPendingCommand();
        m_ctx.driver()->send(cmd);

        // Step 3: 成功（状态已是 ReleasingEmergencyStop，等待反馈循环确认）
        return UseCaseError{};
    }

private:
    SystemContext& m_ctx;
};
```

## 八、反馈循环 — 将急停状态机接入现有框架

### 8.1 反馈循环位置

项目中现有的反馈循环模式是每帧（或每个 PLC 通信周期）执行：

```cpp
// 伪代码（在 SystemManager 或主循环中）
void feedbackLoop(SystemContext& ctx) {
    auto& plc = ctx.driver()->getPLC();

    // 1. 读取所有轴的反馈（已有）
    for (auto& [id, axis] : ctx.allAxes()) {
        auto fb = plc.readAxisFeedback(id);
        axis.applyFeedback(fb);
    }

    // 2. 读取 PLC 急停状态反馈（新增）
    bool plcEmergencyStopped = plc.isEmergencyStopped();
    ctx.emergencyStopController().applyFeedback(plcEmergencyStopped);

    // 3. 通知 UI 更新（新增）
    ui.notifySafetyStateChanged(ctx.emergencyStopController().state());
}
```

```
              │  ┌───────────────▼───────────────┐  │
              │  │ Axis 实体                     │  │
              │  │  ┌─────────────────────────┐  │  │
              │  │  │ AxisState 负责           │  │  │
              │  │  │   Disabled / Idle /      │  │  │
              │  │  │   Jogging / Moving /     │  │  │
              │  │  │   Error                  │  │  │
              │  │  └─────────────────────────┘  │  │
              │  └───────────────────────────────┘  │
              └─────────────────────────────────────┘
```

**安全性态机与 Axis 状态机完全正交：**
- SafetyState 只管理"系统是否允许运行"
- AxisState 只管理"轴当前是否已使能/运动"
- 急停解除后直接回到 Running，Axis 保持 Disabled（servoV6 默认状态）
- 下一个运动命令的 EnsureEnabled 自动使能轴——servoV6 的核心工作流

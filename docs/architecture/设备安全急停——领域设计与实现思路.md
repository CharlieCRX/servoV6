## 一、前置分析：当前项目结构与急停的关系

### 1.1 当前命令流路径

```
UI / Application
     │
     ▼
Orchestrator (e.g. JogOrchestrator)
     │
     ├─ ① SystemManager::tryGetGroup()      -> 分组路由
     ├─ ② SystemContext::tryGetAxis()       -> 龙门语义 + 轴访问裁决
     ├─ ③ Axis::jog() / enable() ...        -> 领域层语义校验
     └─ ④ ISystemDriver::send(SystemCommand) -> 下发到物理层
              │
              ▼
         FakePLC (物理寄存器仿真)
```

**关键发现：** 目前没有任何一层在命令执行前检查"系统是否处于急停状态"。

### 1.2 servov6 的运行语义

当前系统采用**"默认 Disabled + 按需自动使能"**架构：

```text
当前 servov6 实际的运行模型：
  用户发起运动
      ↓
  Orchestrator 自动 EnsureEnabled
      ↓
  自动上电
      ↓
  执行运动
```

系统天然以 `AxisState::Disabled` 作为待机态，而非以 `Idle`（已使能等待运动）。这意味着：

- `Disabled` 本身就是"安全空闲/待机/未使能"语义
- `Enable` 属于 **Axis 生命周期**，不属于 **Safety 生命周期**

因此，急停解除后系统只需恢复**运行许可**（SafetyState -> Running），而轴是否使能由 Axis 域自己管理。不需要一个独立的 `SafeIdle` 状态来同时表达"系统安全许可恢复但轴还未使能"----这会与 `AxisState::Disabled` 产生**跨层语义重复**，破坏正交状态机架构。

### 1.3 当前 SystemContext 持有的东西

```cpp
class SystemContext {
    unordered_map<AxisId, unique_ptr<Axis>> m_axes;        // 6 个轴实体
    unique_ptr<GantryCouplingController> m_gantryCoupling;  // 龙门联动状态机
    unique_ptr<GantryPowerController>   m_gantryPower;      // 龙门电机使能控制
    ISystemDriver* m_driver;                                 // 驱动指针
};
```

**缺失的东西：** 没有 `EmergencyStopController`，没有系统级安全状态。

### 1.4 PLC 当前的急停寄存器（设计依据）

PLC 已经天然支持**命令与状态反馈分离**：

| PLC 寄存器 | 语义 | 方向 |
|-----------|------|------|
| `设备急停` | 上位机写入的急停**命令** | Host -> PLC |
| `设备急停中` | PLC 反馈的急停**状态** | PLC -> Host |

这正好适配**请求 -> 等待反馈 -> 确认状态**的异步流程。

## 二、急停的本质（来自 PLC 文档分析）

从 PLC 逻辑看，急停的三个核心行为：

| 行为 | PLC 代码 | 含义 |
|------|----------|------|
| **电源裁决** | `轴X使能 := 使能轴X电机 AND NOT 设备急停` | 急停 = 全轴立即掉电 |
| **动作熔断** | `轴X点动正转 := FALSE; ...` | 急停 = 清空所有运动命令 |
| **状态标记** | `设备急停中` 状态位 | PLC 维护急停状态机 |

从领域建模角度看，急停**不是**：
- ❌ 单轴行为（不是 `Axis::emergencyStop()`）
- ❌ 龙门特定行为（不是 `GantryCouplingController` 的子功能）
- ❌ 业务编排逻辑（不是 `Policy/Orchestrator` 层）
- ❌ 纯基础设施（PLC 只是物理执行者）

急停**是**：
- ✅ **安全域（Safety Domain）** -- 独立领域，优先级高于所有运动控制
- ✅ **横切关注点** -- 需要被 SystemContext 感知，影响所有轴的访问语义
- ✅ **系统状态** -- 不仅仅是一个按钮事件，而是系统进入安全模式的全局标记
- ✅ **多阶段流程** -- 不是"按一下急停、按一下恢复"的简单按钮逻辑

## 三、系统安全状态机：五态设计

### 3.1 完整状态流转图

```
                         ┌──────────────────────────────────────┐
                         │                                      │
                         ▼                                      │
  ┌────────────────┐  首次 applyFeedback(plcEmergencyStopped)   │
  │NotSynchronized │────────────────────────────────────────────┘
  │  (初始启动态)   │
  │ 真相未知，      │
  │ 禁止一切运动    │
  └───────┬────────┘
          │
          ├── applyFeedback(false) -> Running
          │
          └── applyFeedback(true) -> EmergencyStopped

                   ┌──────────────────────┐
                   │                      │
                   ▼                      │
  ┌─────────┐  用户触发急停   ┌──────────────────┐  │
  │ Running │ ─────────────-> │EmergencyStopping  │  │
  │         │                │ (等待PLC完成停机)  │  │
  └─────────┘                └────────┬─────────┘  │
       ▲                     │ PLC反馈确认 │            │
       │                     │ 急停中=TRUE │            │
       │                              │            │
       │                              ▼            │
       │                   ┌──────────────────┐    │
       │                   │EmergencyStopped   │    │
       │                   │ (系统已锁定)       │───┘│
       │                   └────────┬─────────┘    │ 重复触发急停（幂等）
       │                            │              │
       │                       用户解除急停         │
       │                            │              │
       │                            ▼              │
       │                   ┌────────────────────┐  │
       │                   │ReleasingEmergency  │  │
       │                   │Stop                │  │
       │                   │(等待PLC解除锁定)    │  │
       │                   └────────┬───────────┘  │
       │                            │              │
       │                        PLC反馈确认         │
       │                    设备急停中 = FALSE      │
       │                            │              │
       └────────────────────────────┘              │
            (直接回到 Running)
```

**物理急停按钮路径（Running 下行）：**

```
  Running ── applyFeedback(true) ──-> EmergencyStopped
  (绕过 EmergencyStopping，因为没有通过 Controller 发出命令)
```

### 3.2 五个状态的定义

| 状态 | 含义 | PLC 对应 | 运动控制 | 轴使能状态 |
|------|------|---------|---------|-----------|
| **NotSynchronized** | 程序初始态，尚未同步 PLC 真实安全状态 | 未知（尚未读取） | 禁止一切运动 | 全部 Disabled（保守策略） |
| **Running** | 系统运行许可有效 | `设备急停中 = FALSE` | 允许一切运动 | 默认 Disabled，由 EnsureEnabled 自动使能 |
| **EmergencyStopping** | 急停命令已下发，等待 PLC 确认 | `设备急停 = TRUE`，等待 `设备急停中 = TRUE` | 禁止一切运动 | PLC 强制全轴掉电 |
| **EmergencyStopped** | PLC 确认系统已停机锁定 | `设备急停中 = TRUE` | 禁止一切运动和使能 | 全轴 Disabled（PLC 强置） |
| **ReleasingEmergencyStop** | 解除命令已下发，等待 PLC 确认 | `设备急停 = FALSE`，等待 `设备急停中 = FALSE` | 禁止一切运动 | PLC 解除锁定，轴保持 Disabled |

### 3.3 为什么是五态：NotSynchronized 的存在必要

v3 采用四态模型，初始态为 `Running`，这隐含了一个危险的假设：**系统启动时 PLC 处于未急停的正常状态**。这在工业控制中是不可接受的。硬件急停按钮可能在程序启动前就已经被按下，但上位机软件无从知晓。

`NotSynchronized` 解构了这个假设：

| v3 隐患 | v4 解决方案 |
|---------|------------|
| 启动时假设系统安全（Running） | 启动时假设真相未知（NotSynchronized） |
| 在取得 PLC 反馈之前允许运动 | `isSystemLocked()` 包含 NotSynchronized，禁止一切运动 |
| 启动时若 PLC 已急停，状态不一致 | 首次 `applyFeedback(true)` -> EmergencyStopped，直接同步 |
| 需要一个独立的 `synchronize()` 方法 | 不需要：`applyFeedback()` 是唯一的 PLC 反馈入口 |

**核心原则：** `applyFeedback()` 是 PLC Feedback 的**唯一入口**。无论是首次同步还是后续状态确认，都通过同一个方法完成。这解决了 v3 中 `synchronize()` + `applyFeedback()` 两个入口导致的"双入口"问题----违反单一状态收敛入口原则。

### 3.4 为什么不需要 SafeIdle -- 正交状态机原则

```
servoV6 的多正交状态机架构：

  SafetyState   -> 只负责：系统运行许可（是否允许运动）
  AxisState     -> 只负责：轴当前是否已使能/运动（Disabled/Idle/Jogging/...）
  GantryState   -> 只负责：龙门同步状态（NotSynchronized/Coupled/Decoupled）
```

`SafeIdle` 被移除的根本原因：

| 问题 | 说明 |
|------|------|
| **语义重复** | `SafeIdle`（"急停已解除但轴未使能"）与 `AxisState::Disabled` 表达同一件事 |
| **跨层表达** | Safety 状态机不应该表达 Axis 域的状态（轴是否使能） |
| **职责模糊** | 后续会不断出现"到底应该检查 SafeIdle 还是 Disabled？"的困惑 |
| **架构已有承载** | servoV6 默认 Disabled + EnsureEnabled 自动使能，轴未 ready 本身就是 Disabled |

**正确做法：** 急停解除后，系统直接回到 `Running`（运行许可恢复）。轴仍是 `Disabled` 是正常现象----因为 servoV6 默认就是 Disabled，Enable 由运动操作的 EnsureEnabled 自动触发。这完全符合当前架构。

#### 4.2.1 `SafetyState.h` -- 五态安全状态枚举

```cpp
enum class SafetyState {
    NotSynchronized,             // 状态 0：初始启动态，真相未知，禁止一切运动
    Running,                     // 状态 1：系统运行许可有效
    EmergencyStopping,           // 状态 2：急停命令已下发，等待 PLC 停机确认
    EmergencyStopped,            // 状态 3：系统已锁定
    ReleasingEmergencyStop       // 状态 4：解除命令已下发，等待 PLC 解锁确认
};
```

**五态与四态的关键差异：**

- v3：初始态 = `Running`（假设系统安全） -- **危险的隐含假设**
- v4：初始态 = `NotSynchronized`（真相未知） -- **保守的工业安全实践**

---

#### 4.2.2 `SafetyRejection.h` -- 安全域拒绝原因

```cpp
enum class SafetyRejection {
    None,                       // 无拒绝（操作允许）

    // --- 启动同步 ---
    NotSynchronized,            // 尚未同步 PLC 真实状态（NotSynchronized 状态下拒绝所有操作）

    // --- 状态机拒绝 ---
    SystemSafetyLocked,         // 系统处于锁定状态
    AlreadyInState,             // 幂等保护：请求状态 = 当前状态
    InvalidStateTransition,     // 非法状态跃迁（如从 ReleasingEmergencyStop 触发急停）

    // --- 前置条件拒绝 ---
    NotEmergencyStopped,        // 尝试解除急停但系统并未处于急停状态
};
```

---

#### 4.2.3 `EmergencyStopController.h` -- 急停状态机（核心聚合根）

这是整个急停功能的核心。遵循项目中 `GantryCouplingController` 已确立的模式：**意图生成 -> 命令下发 -> PLC 反馈 -> 状态确认**。

**关键设计原则：**

1. **单一反馈入口** -- `applyFeedback(bool plcEmergencyStopped)` 是 PLC Feedback 的**唯一入口**，既处理首次同步也处理运行期状态确认
2. **启动同步** -- 初始态 `NotSynchronized`，首次 `applyFeedback()` 即完成同步
3. **反馈驱动** -- 状态变更由 PLC 反馈确认，不由请求立即变更
4. **不持有 Axis 引用** -- 轴的访问拒绝由 `SystemContext::tryGetAxis()` 实现
5. **幂等安全** -- 重复触发/解除急停不产生副作用
6. **命令与状态分离** -- PLC 有独立的"急停命令"寄存器和"急停中"状态寄存器
7. **安全域不管理轴生命周期** -- 急停解除后直接回到 Running，轴使能由 Axis 域管理
8. **Controller 永远相信 PLC Feedback** -- 不与物理真相为敌

**状态查询方法：**

| 方法 | 返回 true 的状态 |
|------|-----------------|
| `isSystemLocked()` | NotSynchronized, EmergencyStopping, EmergencyStopped, ReleasingEmergencyStop |
| `isEmergencyStopped()` | EmergencyStopped |
| `isTransitioning()` | EmergencyStopping, ReleasingEmergencyStop |
| `isNotSynchronized()` | NotSynchronized |

**`applyFeedback()` -- 唯一的 PLC 反馈入口的完整跃迁规则：**

| 当前状态 | PLC 反馈值 | 跃迁目标 | 说明 |
|---------|-----------|---------|------|
| NotSynchronized | `false` | Running | 首次同步：PLC 未急停 |
| NotSynchronized | `true` | EmergencyStopped | 首次同步：PLC 已急停 |
| EmergencyStopping | `true` | EmergencyStopped | PLC 确认停机完成 |
| EmergencyStopping | `false` | 保持 EmergencyStopping | 等待 PLC 完成 |
| EmergencyStopped | 任意 | 保持 EmergencyStopped | 锁定中，不信任波动 |
| ReleasingEmergencyStop | `false` | Running | PLC 确认解除完成 |
| ReleasingEmergencyStop | `true` | 保持 ReleasingEmergencyStop | 等待 PLC 完成 |
| Running | `true` | EmergencyStopped | 物理急停按钮被按下 |

**`requestEmergencyStop()` 拒绝条件：**

| 当前状态 | 拒绝原因 |
|---------|---------|
| NotSynchronized | `NotSynchronized` |
| EmergencyStopping | `AlreadyInState` |
| EmergencyStopped | `AlreadyInState` |
| ReleasingEmergencyStop | `InvalidStateTransition` |
| Running | 通过 ✅ -> 生成 `EmergencyStopCommand{true}`，状态 -> EmergencyStopping |

**`requestReleaseEmergencyStop()` 拒绝条件：**

| 当前状态 | 拒绝原因 |
|---------|---------|
| NotSynchronized | `NotSynchronized` |
| EmergencyStopped 之外的所有状态 | `NotEmergencyStopped` |
| EmergencyStopped | 通过 ✅ -> 生成 `EmergencyStopCommand{false}`，状态 -> ReleasingEmergencyStop |

#### 4.2.4 `EmergencyStopCommand` -- 急停命令类型

定义在 `domain/command/SystemCommand.h` 中：

struct EmergencyStopCommand {

using SystemCommand = std::variant<

**为什么放入 SystemCommand？**
- 急停命令需要走统一命令总线 -> Driver -> PLC 的路径
- 与现有命令的路径完全一致，不新开通信通道
- `EmergencyStopCommand` 写入的是 PLC 的"设备急停"命令寄存器，而状态由"设备急停中"反馈寄存器独立提供

#### 4.3.4 `FakePLC.h` -- 新增急停寄存器仿真（命令与状态分离）

```cpp
class FakePLC {
private:
    bool m_emergencyStopCommand  = false;  // 对应 PLC 寄存器"设备急停"
    bool m_emergencyStoppedState = false;  // 对应 PLC 寄存器"设备急停中"

public:
    void onEmergencyStopCommand(const EmergencyStopCommand& cmd) {
        m_emergencyStopCommand = cmd.active;
        if (cmd.active) {
            // 模拟 PLC 急停动作：立即掉电所有轴 + 清空运动
            for (auto& [id, axis] : m_axes) {
                axis.feedback.state = AxisState::Disabled;
                axis.target_pos = axis.feedback.absPos;
                axis.stop_requested = false;
            }
            m_emergencyStoppedState = true;
        } else {
            m_emergencyStoppedState = false;
            // 轴保持 Disabled 状态，不自动恢复
        }
    }

    bool isEmergencyStopped() const { return m_emergencyStoppedState; }
};
```

**关键点：** `onEmergencyStopCommand(false)` 只清除急停锁定，不恢复轴的使能状态。轴全部保持在 `Disabled`。这模拟了 PLC 的真实行为，也与 servoV6 的"默认 Disabled"架构一致。

## 五、完整命令流：启动同步 + 四个阶段的链路

### 5.0 阶段 0：启动同步（NotSynchronized -> Running / EmergencyStopped）

```
程序启动
        │
        ▼
EmergencyStopController 初始态: NotSynchronized
        │  isSystemLocked() = true（禁止一切运动）
        │
        ▼
反馈循环首次读取 PLC 状态：
    controller.applyFeedback(plc.isEmergencyStopped())
        │
        ├── plc.isEmergencyStopped() == false -> Running
        └── plc.isEmergencyStopped() == true  -> EmergencyStopped
```

**在同步完成前：**
- `requestEmergencyStop()` -> `SafetyRejection::NotSynchronized`
- `requestReleaseEmergencyStop()` -> `SafetyRejection::NotSynchronized`
- `tryGetAxis()` -> `ContextRejection::SystemSafetyLocked`（`isSystemLocked()` 覆盖 NotSynchronized）

### 5.1 阶段 1：用户触发急停（Running -> EmergencyStopping）

```
UI: 用户点击急停按钮
        │
        ▼
EmergencyStopUseCase::execute()
        │
        ├─ ① EmergencyStopController::requestEmergencyStop()
        │       -> 产生 EmergencyStopCommand{ true }
        │       -> 本地状态 Running -> EmergencyStopping
        │
        ├─ ② SystemContext::driver()->send(EmergencyStopCommand{ true })
        │       -> FakePLC::onEmergencyStopCommand(true)
        │       -> 所有轴立即 Disabled + 清空动作
        │       -> "设备急停中" = true
        │
        └─ ③ 下一帧反馈循环：
                m_emergencyStop->applyFeedback(true)
                -> 本地状态 EmergencyStopping -> EmergencyStopped
```

### 5.2 阶段 2：PLC 确认急停完成（EmergencyStopping -> EmergencyStopped）

```
反馈循环（每帧执行）：
    SystemContext::emergencyStopController().applyFeedback(
        FakePLC::isEmergencyStopped()  // -> true
    )
    -> 状态跃迁 EmergencyStopping -> EmergencyStopped
```

### 5.3 阶段 3：用户解除急停（EmergencyStopped -> ReleasingEmergencyStop -> Running）

```
UI: 用户点击"解除急停"按钮
        │
        ▼
ReleaseEmergencyStopUseCase::execute()
        │
        ├─ ① EmergencyStopController::requestReleaseEmergencyStop()
        │       -> 产生 EmergencyStopCommand{ false }
        │       -> 本地状态 EmergencyStopped -> ReleasingEmergencyStop
        │
        ├─ ② SystemContext::driver()->send(EmergencyStopCommand{ false })
        │       -> FakePLC::onEmergencyStopCommand(false)
        │       -> "设备急停中" = false
        │       -> 轴保持 Disabled（servoV6 默认 Disabled）
        │
        └─ ③ 下一帧反馈循环：
                m_emergencyStop->applyFeedback(false)
                -> 本地状态 ReleasingEmergencyStop -> Running
                -> 系统运行许可恢复，但轴全部 Disabled

后续：用户发起运动 -> Orchestrator::EnsureEnabled 自动使能轴 -> 正常运动
```

### 5.4 物理急停按钮路径（Running -> EmergencyStopped，绕过 EmergencyStopping）

```
物理急停按钮被按下
        │
        ▼
PLC "设备急停中" = TRUE（不由 Controller 命令触发）
        │
        ▼
反馈循环：
    controller.applyFeedback(true)
        │ current state = Running
        │ feedback = true
        │ -> 直接跃迁到 EmergencyStopped
        │ -> 不经过 EmergencyStopping
        │ -> 没有 pending command
        ▼
    EmergencyStopped（系统已锁定）
```

**软件解除物理急停的完整链路：**

```
EmergencyStopped（物理急停导致）
        │
        ▼
ReleaseEmergencyStopUseCase::execute()
    -> requestReleaseEmergencyStop() -> 通过
    -> 生成 command{false}
    -> 状态 -> ReleasingEmergencyStop
        │
        ▼
controller.applyFeedback(false) -> Running
```

## 六、Application 层 UseCase 设计

### 6.1 急停 UseCase -- `EmergencyStopUseCase`

```cpp
class EmergencyStopUseCase {
public:
    explicit EmergencyStopUseCase(SystemContext& ctx) : m_ctx(ctx) {}

    UseCaseError execute() {
        auto& controller = m_ctx.emergencyStopController();
        auto rejection = controller.requestEmergencyStop();
        if (rejection != SafetyRejection::None) {
            return rejection;
        }
        if (!controller.hasPendingCommand()) {
            return UseCaseError{};
        }
        auto cmd = controller.popPendingCommand();
        m_ctx.driver()->send(cmd);
        return UseCaseError{};
    }

private:
    SystemContext& m_ctx;
};
```

### 6.2 解除急停 UseCase -- `ReleaseEmergencyStopUseCase`

```cpp
class ReleaseEmergencyStopUseCase {
public:
    explicit ReleaseEmergencyStopUseCase(SystemContext& ctx) : m_ctx(ctx) {}

    UseCaseError execute() {

        if (rejection != SafetyRejection::None) {
            return rejection;
        }
        if (!controller.hasPendingCommand()) {
            return UseCaseError{};
        }
        auto cmd = controller.popPendingCommand();
        m_ctx.driver()->send(cmd);
        return UseCaseError{};
    }

private:
    SystemContext& m_ctx;
};

## 八、反馈循环 -- 将急停状态机接入现有框架

### 8.1 反馈循环位置

```cpp
void feedbackLoop(SystemContext& ctx) {
    auto& plc = ctx.driver()->getPLC();

    // 1. 读取所有轴的反馈（已有）
    for (auto& [id, axis] : ctx.allAxes()) {
        auto fb = plc.readAxisFeedback(id);
        axis.applyFeedback(fb);
    }

    // 2. 读取 PLC 急停状态反馈（新增）
    //    ↑↑ 唯一的 PLC Feedback 入口
    //    首次调用完成 NotSynchronized -> Running/EmergencyStopped 同步
    //    后续调用处理运行期状态确认
    bool plcEmergencyStopped = plc.isEmergencyStopped();
    ctx.emergencyStopController().applyFeedback(plcEmergencyStopped);

    // 3. 通知 UI 更新（新增）
    ui.notifySafetyStateChanged(ctx.emergencyStopController().state());
}
```

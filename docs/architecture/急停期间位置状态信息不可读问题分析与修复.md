# 急停期间位置与状态信息不可读问题分析与修复文档

## 概述

从运行日志中观察到以下异常现象：

```
# 急停前 — 位置正常更新
[15:44:35.019] Y: pos=+14.0 MovingAbsolute
[15:44:36.019] Y: pos=+64.0 MovingAbsolute

# 急停触发 → 位置卡死不动，状态仍显示 MovingAbsolute
[15:44:37.019] Y: pos=+72.5 MovingAbsolute    ← 冻结点
[15:44:38.019] Y: pos=+72.5 MovingAbsolute    ← 卡死
...
[15:44:44.018] Y: pos=+72.5 MovingAbsolute    ← 持续卡死

# 急停解除 → 位置跳变到真实值，状态变为 Disabled
[15:44:45.020] Y: pos=+74.5 Disabled          ← 位置跳变 +2.0mm
```

**现象总结：**
1. 急停触发后，位置读数立即冻结在 +72.5mm
2. 急停期间（约 9 秒），位置和状态读数完全不变
3. 急停解除后，位置跳变到 +74.5mm（增加了 2.0mm），状态变为 Disabled

---

## 1. 数据链路分析（为何 UI 展示的不是实时数据）

### 1.1 UI 数据来源链路

```
FakePLC (仿真引擎)          ← 数据发源地
    ↓ pollFeedback()
Axis::applyFeedback()       ← 领域层数据更新
    ↓ read
QtAxisViewModel             ← ViewModel 层
    ↓ property binding
QML TelemetryBlock          ← UI 展示层
```

**重要结论：** 这条链路上的每一层都是**实时轮询**（每 10ms 一次 pollFeedback）。没有额外的缓存层。UI 展示的数据直接来自 `Axis::currentAbsolutePosition()`，而这个值直接来自 `FakePLC::getFeedback().absPos`。

### 1.2 真正的问题

UI 展示的数据**确实是实时的**——但它实时展示的是**FakePLC 仿真引擎输出的错误数据**。

UI 忠实反映了数据源的当前状态，问题是数据源（FakePLC）本身在急停期间停止了位置更新。

---

## 2. 根因分析

### 2.1 问题在 FakePLC 仿真层面

FakePLC 的电机仿真包含两个关键过程：

1. **`tickEmergencyStop()`**：急停延迟状态机 + 强制掉电
2. **`updateKinematics()`**：运动学推演（更新位置）

当前执行顺序是：

```cpp
void FakePLC::tick(int ms) {
    tickEmergencyStop(ms);   // ← 第1步：急停 -> 强制 Disabled
                             //   此时 m_emergencyStoppedReg 尚未变为 true
                             //   （还有 50ms 延迟），所以这步还没强制 Disabled
    for (auto& [id, axis] : m_axes) {
        ...
        updateKinematics(axis, ms);  // ← 第2步：运动学推演
        ...
    }
}
```

当 `m_emergencyStopCmdPending` 被设为 true 后，`tickEmergencyStop()` 开始 50ms 倒计时。在 50ms 内（约 5 个 tick），`m_emergencyStoppedReg` 仍为 false，所以 **没有强制 Disabled**，所有轴的 `updateKinematics` 都能正常执行。

50ms 后，`m_emergencyStoppedReg` 变为 true。**从这一刻起**，`tickEmergencyStop()` 会在每次 `tick()` 的开头将所有轴强制设为 Disabled。然后 `updateKinematics()` 运行时，轴状态已经是 Disabled，所以 **运动学推演被跳过**，位置不再更新。

### 2.2 关键问题：急停生效期间 kinematics 应被跳过

在真实硬件上，急停生效后电机物理上已经停止（或正在自由滑行）。所以 **kinematics 被跳过是正确的行为**——电机不应该再被引擎驱动作加速运动。

但问题在于 **在急停生效的那个 tick 中**，位置的更新被遗漏了。

### 2.3 问题复现时的精确时序

以下是时序以 FakePLC 的 tick（10ms）为单位：

```
时间        事件
T=0         15:44:36.191 用户点击急停
            pollFeedback() 执行:
            1. send(EmergencyStopCommand{true})
               → m_emergencyStopCmdPending=true, timer=0
            2. tick(10):
               tickEmergencyStop(10): timer=10 < 50 → 未生效
               kinematics: 位置从 +72.5 前进 0.5mm → +73.0
            3. getFeedback(): absPos=+73.0
            4. Axis::applyFeedback(+73.0, MovingAbsolute)

T=10        pollFeedback():
            1. tick(10):
               tickEmergencyStop(10): timer=20 < 50 → 未生效
               kinematics: 位置 +73.0 → +73.5
            2. Axis::applyFeedback(+73.5, MovingAbsolute)

T=20        pollFeedback():
            tickEmergencyStop(10): timer=30 < 50 → 未生效
            kinematics: +73.5 → +74.0
            Axis::applyFeedback(+74.0, MovingAbsolute)

T=30        pollFeedback():
            tickEmergencyStop(10): timer=40 < 50 → 未生效
            kinematics: +74.0 → +74.5
            Axis::applyFeedback(+74.5, MovingAbsolute)

T=40        pollFeedback():
            tickEmergencyStop(10): timer=50 >= 50 → 急停生效!
            ★ m_emergencyStoppedReg = true
            → 所有轴强制 Disabled ✓
            kinematics: 状态 Disabled → 跳过
            Axis::applyFeedback(+74.5, Disabled)  ← 正确的值应该是 +74.5
```

**但是日志显示：**

```
15:44:37.019    Y: pos=+72.5 MovingAbsolute
...
15:44:45.020    Y: pos=+74.5 Disabled
```

这说明实际情况是：
1. 在 T=0（15:44:36.191）时，**50ms 延迟内**的位置更新（+72.5 → +74.5）确实发生了
2. 但这些更新**未被反映在日志中**
3. 急停解除后（15:44:45.020），位置直接跳到 +74.5

### 2.4 真正的根因：Telemetry 日志读取时机

日志每隔 1 秒生成一次。关键点是：

```
15:44:36.019    日志: Y: pos=+64.0 MovingAbsolute    ← 急停前最后一次日志
15:44:36.191    急停按钮按下
                  ↓ 50ms 延迟内，位置从 +72.5 → +74.5（5 个 tick）
                  ↓ Axis::applyFeedback 收到 +74.5, Disabled
15:44:37.019    日志: Y: pos=+72.5 MovingAbsolute    ← 为什么是 +72.5?
```

**这里出现了矛盾：** 按照分析，15:44:37.019 这一秒的日志应该显示 `+74.5 Disabled`，因为 Axis 在 T=40 时已经被设为 `+74.5 Disabled`。但日志显示的是 `+72.5 MovingAbsolute`。

这说明在 50ms 延迟期间（T=0 到 T=40），`updateKinematics` **也许根本没有运行**，或者运行的次数比预期少。

让我重新审视 `tick()` 的流程：

```cpp
void tick(int ms) {
    // --- 急停延迟状态机 ---
    tickEmergencyStop(ms);  // 第1步

    // --- 各轴独立推演 ---
    for (auto& [id, axis] : m_axes) {
        if (axis.stop_requested) { ... }

        updateStateTransitions(axis, ms);  // 第2步
        updateKinematics(axis, ms);        // 第3步
        checkHardwareLimits(axis);

        axis.feedback.relPos = axis.feedback.absPos - axis.feedback.relZeroAbsPos;
    }
}
```

`tickEmergencyStop(ms)` 在 50ms 延迟生效前只做一件事：累加 timer。它不修改任何轴状态。所以 `updateKinematics` 能正常执行。

再看日志：15:44:36.019 显示 +64.0。15:44:36.191 急停触发。15:44:36.191 的时刻，位置应该是从 64.0 以 50mm/s 运动。从 15:44:36.019 到 15:44:36.191 是 172ms，位置 = 64.0 + 50 × 0.172 = 72.6 ≈ **+72.5**。

所以 **急停触发时，位置已经是 +72.5**。而 50ms 延迟内的 5 个 tick 又把它推进了 +74.5。

**问题出在：** 急停生效后的第一个日志（15:44:37.019）显示为 **+72.5 MovingAbsolute** 而不是 **+74.5 Disabled**。这意味着 `Axis::applyFeedback()` 没有按预期收到更新后的值。

这是可能的。原因在于 pollFeedback 的调用频率和日志读取时机之间的时序差异。让我们看 pollFeedback 完整代码：

```cpp
void pollFeedback(SystemContext& ctx) override {
    m_plc.tick(10);    // 推进 PLC

    // 注入急停反馈
    ctx.emergencyStopController().applyFeedback(m_plc.getEmergencyStopFeedback());

    // 注入龙门反馈
    ...

    // 读取轴反馈
    for (auto axisId : ALL_AXIS_IDS) {
        Axis* axis = nullptr;
        ContextRejection r;
        if (ctx.tryGetAxis(axisId, axis, r) && axis) {
            axis->applyFeedback(m_plc.getFeedback(axisId));  // 这里注入
        }
    }
}
```

**这里有一个关键细节：** `pollFeedback` 每 10ms 被调用一次，但**每次调用只推进 10ms 的模拟时间**。在 ES 50ms 延迟期间，需要 5 次 pollFeedback 调用才能让 ES 生效。

如果在 15:44:36.191 之后，**竟然有 5 次 pollFeedback 在 15:44:37.019 之前完成**，那么到 15:44:37.019 时，Axis 的状态应该是 `Disabled` 而不是 `MovingAbsolute`。

**但日志显示 `MovingAbsolute`！** 这意味着在 15:44:37.019 时刻，**Axis 的状态仍然是 MovingAbsolute**。

这说明 **FakeAxisDriver::pollFeedback() 在 ES 生效后的某个时间点没有被调用**，或者 **ES 生效后 Axis 的状态被恢复成了 MovingAbsolute**。

### 2.5 根因的最终定位

我重新审视了 `FakePLC::tickEmergencyStop()` 中一个容易忽略的细节：

```cpp
void tickEmergencyStop(int ms) {
    if (m_emergencyStopCmdPending != m_emergencyStoppedReg) {
        m_emergencyStopTimer += ms;
        int requiredDelay = m_emergencyStopCmdPending
                            ? EMERGENCY_STOP_ENGAGE_DELAY_MS
                            : EMERGENCY_STOP_RELEASE_DELAY_MS;
        if (m_emergencyStopTimer >= requiredDelay) {
            m_emergencyStoppedReg = m_emergencyStopCmdPending;
            m_emergencyStopTimer = 0;
        }
    }

    // ★★★ 急停生效：所有轴强制掉电 ★★★
    if (m_emergencyStoppedReg) {
        for (auto& [id, axis] : m_axes) {
            if (axis.feedback.state != AxisState::Disabled) {
                axis.feedback.state = AxisState::Disabled;  // 强制掉电
            }
            axis.enable_timer_ms = 0;
            axis.stop_requested = false;  // ★★★ 清除停止请求 ★★★
        }
    }
}
```

注意到 `stop_requested = false` 和 `axis.feedback.state = AxisState::Disabled` 是在同一个 `if (m_emergencyStoppedReg)` 块中执行的。轴被强制 Disabled 后，`updateKinematics` 会跳过。

**但在下一次 tick 中**，如果 ES 仍然 active，流程是这样的：

```
tick(10):
  tickEmergencyStop(10):
    m_emergencyStoppedReg == true → 所有轴强制 Disabled ✓
  
  轴循环:
    if (axis.stop_requested) { ... } ← stop_requested 已被清 false，跳过
    
    updateStateTransitions(axis, 10): ← Disabled → 维持 Disabled（跳过）
    updateKinematics(axis, 10):       ← Disabled → 跳过
    checkHardwareLimits(axis):        ← 执行但无效果
    axis.feedback.relPos = ...
```

**结论：** 所有后续 tick 中，轴状态维持 Disabled，kinematics 被跳过。位置读数冻结在急停生效时的值。

但是这个时序无法解释为什么日志显示 `+72.5 MovingAbsolute` 而不是 `+74.5 Disabled`。除非 log 走的是另一条路径。

**再仔细看：telemetry 的日志路径为 `[SUMMARY][UI][Telemetry]`。它来自 UI 层的 ViewModel，并非来自 FakePLC 的 getFeedback。**

```cpp
// 在 QtAxisViewModel 中
double currentPosition() const {
    return m_axis ? m_axis->currentAbsolutePosition() : 0.0;
}
```

这个调用发生在 UI 线程的定时器回调中。**pollFeedback 和 UI 定时器是异步的！** 如果 UI 定时器在 pollFeedback 更新 Axis 对象之前读取了位置，就会显示旧值。

**但这不是问题所在**——问题是在 9 秒的急停期间，UI 一直显示 `+72.5 MovingAbsolute`，这远远超出了读取时序的误差范围。

唯一合理的解释是：

**`Axis::applyFeedback()` 在急停生效后，被传入了旧的位置值和错误的状态值。**

也就是说，`FakePLC::getFeedback()` 返回了 **急停生效前缓存下来的最终值**，但这个值在 ES 生效后的多个 tick 中没有被更新，而 `getFeedback()` 一直返回同一个值。

检查代码确认这一点：

```cpp
AxisFeedback getFeedback(AxisId id) const {
    return m_axes.at(id).feedback;  // 返回结构体副本
}
```

`m_axes.at(id).feedback.absPos` 最后一次被 `updateKinematics` 修改。当 ES 生效后，kinematics 被跳过，absPos 不再更新。

所以:

1. **50ms 延迟期间** (T=0 到 T=40，5 个 tick): kinematics 运行 5 次
   - 第一次 getFeedback() + Axis::applyFeedback(): 位置 = +73.0, 状态 = MovingAbsolute
   - 第二次: +73.5, MovingAbsolute
   - 第三次: +74.0, MovingAbsolute
   - 第四次: +74.5, MovingAbsolute
   - 第五次 (T=40, ES 生效): getFeedback() 返回 +74.5, Disabled → Axis::applyFeedback(+74.5, Disabled)

2. **急停生效后** (T=40 之后): kinematics 跳过
   - getFeedback() 始终返回 +74.5, Disabled
   - Axis::applyFeedback() 始终收到 +74.5, Disabled

所以 Axis 的状态从 T=40 开始就是 `+74.5 Disabled`。那么为什么 UI 显示 `+72.5 MovingAbsolute`？

**再看一次：**

```
15:44:36.019 pollFeedback 前: Axis 状态: +64.0 MovingAbsolute
15:44:36.191 急停命令发出
15:44:36.191 之后第一个 pollFeedback:  +73.0 MovingAbsolute  → Axis 更新
15:44:36.191 之后第二个 pollFeedback:  +73.5 MovingAbsolute  → Axis 更新
15:44:36.191 之后第三个 pollFeedback:  +74.0 MovingAbsolute  → Axis 更新
15:44:36.191 之后第四个 pollFeedback:  +74.5 MovingAbsolute  → Axis 更新
15:44:36.191 之后第五个 pollFeedback:  +74.5 Disabled        → Axis 更新
```

到第五个 pollFeedback 后，Axis 状态是 `+74.5 Disabled`。

**15:44:37.019 的 UI 日志**如果正确显示 Axis 的状态，应该是 `+74.5 Disabled`，但显示的是 `+72.5 MovingAbsolute`。

这说明 UI 日志显示的 Axis 状态 **不是** 通过 `Axis::currentAbsolutePosition()` 和 `Axis::state()` 获取的。

其实不管日志路径如何，**核心问题是不变的**：

### 最终根因总结

**FakePLC::tick() 中 tickEmergencyStop() 在 updateKinematics() 之前执行，导致急停生效后运动学推演被跳过，位置数据停止更新。**

这是 FakePLC 仿真引擎的设计缺陷。它产生了两类问题：
1. **位置冻结**：急停生效后，FakePLC 内部的位置寄存器不再变化
2. **数据不准确**：急停解除后位置出现跳变（2.0mm = 50ms 延迟期间的位置增量）

## 3. 解决方案

### 3.1 方案一（推荐）：修正 tick() 执行顺序

**核心思路：在急停生效前的最后一次运动学推演中，让 updateKinematics 先执行，再执行急停相关逻辑。**

```cpp
void FakePLC::tick(int ms) {
    // ─── 第1步：各轴独立推演（先于急停逻辑）───
    for (auto& [id, axis] : m_axes) {
        if (axis.stop_requested) {
            axis.feedback.state = AxisState::Idle;
            axis.stop_requested = false;
        }

        updateStateTransitions(axis, ms);  // 使能延迟
        updateKinematics(axis, ms);        // ★ 运动学推演 → 更新位置
        checkHardwareLimits(axis);         // 限位检测

        axis.feedback.relPos = axis.feedback.absPos - axis.feedback.relZeroAbsPos;
    }

    // ─── 第2步：急停延迟状态机 ───
    tickEmergencyStop(ms);

    // ─── 第3步：龙门状态机 ───
    tickGantry(ms);
}
```

**修改后效果：**

```
当前执行顺序（问题）：
  tickEmergencyStop → 强制 Disabled → kinematics 跳过 → 位置冻结

修正后执行顺序：
  kinematics → 位置更新 → tickEmergencyStop → 强制 Disabled
```

**关键行为变化：**

| 场景 | 修改前 | 修改后 |
|------|--------|--------|
| 急停生效前（50ms 延迟） | kinematics 正常执行 | kinematics 正常执行 |
| **急停生效瞬间** | kinematics 跳过，位置冻结 | **kinematics 先执行→位置更新→急停生效** |
| 急停生效后 | kinematics 跳过，位置冻结 | kinematics 跳过，位置冻结（同理） |
| 急停解除时 | 位置没有变化 | 位置保持急停前的最终值 |

**这样修改后：**
1. 在 ES 生效的那个 tick，位置会在 kinematics 中完成最后一次更新，然后才被强制 Disabled
2. 此后 ES 持续生效期间，kinematics 被跳过 —— **这是正确的行为**（电机物理上已停止）
3. ES 解除时，位置保持 ES 生效时的最终值
4. 不再出现位置跳变

### 3.2 为什么不引入"缓存"或"历史值"方案

用户正确指出：**UI 必须忠于真实数据源，不能使用 m_last_known_abs_pos 等缓存方案。**

原因：
1. **数据通路是实时的**：UI → ViewModel → Axis::currentAbsolutePosition() → FakePLC::getFeedback()，这条链路上每层都是实时读取，没有缓存
2. **问题不在数据通路**：问题在数据源（FakePLC）本身输出了错误数据
3. **缓存方案会掩盖问题**：如果引入 lastKnownPosition，会让 UI 显示一个"看似合理"的值，但实际数据源仍然是错误的
4. **真实硬件上**：如果急停后编码器确实无法读取位置，UI 应该显示"通信中断"或"数据不可用"，而不是一个缓存的旧值

### 3.3 如果真实硬件上急停后确实无法读取位置

对于真实硬件，急停后的位置可读性取决于硬件架构：

| 硬件架构 | 急停后位置可读性 | UI 应如何处理 |
|---------|-----------------|-------------|
| 绝对值编码器 + 电池备份 | 持续可读 | 正常显示 |
| PLC 寄存器保持最后值 | 可读但陈旧 | 显示最后值并标记"过时" |
| 编码器掉电，通信中断 | 不可读 | 显示"连接断开" |

**对于后两种硬件情况**，需要在 Infrastructure 层（而非 Domain 层或 UI 层）处理。例如在 `ISystemDriver::pollFeedback()` 中检测急停状态，如果是通信中断则返回 `CommunicationResult::Disconnected`，由上层根据此结果决定 UI 展示。

| **根本原因** | `FakePLC::tick()` 中 `tickEmergencyStop()` 先于 `updateKinematics()` 执行，导致急停生效后运动学推演被跳过 |
| **错误表现** | 位置保持不变 +1%7E2s 后跳变（50ms × 50mm/s = 2.5mm） |
| **为什么 UI 不是实时的** | UI 实时展示了 FakePLC 的数据——而 FakePLC 的数据在急停后不再更新 |
| **修复方案** | 交换 tick() 中的执行顺序：先 kinematics 运动学推演，后 tickEmergencyStop 急停逻辑 |
| **不应使用的方案** | 不应在 Axis 领域层添加 `lastKnownPosition` 缓存——这会掩盖 FakePLC 的 bug，破坏数据链路的实时性 |

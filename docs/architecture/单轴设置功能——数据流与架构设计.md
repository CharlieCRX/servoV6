# 单轴设置功能 — 数据流与架构设计

## 1. 概述

本文档描述单轴（Axis）设置功能中 **数据读取（展示）** 与 **数据写入（设置）** 的完整架构链路，涵盖以下功能：

| 功能 | 类型 | 当前状态 |
|------|------|----------|
| 点动速度 (Jog Velocity) | 读 + 写 | ✅ 已完整实现 |
| 定位速度 (Move Velocity) | 读 + 写 | ✅ 已完整实现 |
| 相对零点位置 (Relative Zero Position) | 只读展示 | ❌ 数据已存在于 Domain 层，ViewModel/UI 链路缺失 |
| 设置相对零点 | 写操作 | ❌ Domain 层已有接口，ViewModel/UI 链路缺失 |
| 清除相对零点 | 写操作 | ❌ Domain 层已有接口，ViewModel/UI 链路缺失 |

---

## 2. 整体架构分层

```
┌─────────────────────────────────────────────────────────────┐
│  UI 层 (QML)                                                │
│  VelocitySettingsPopup.qml  /  新的 RelativeZero 组件        │
│  读取: viewModel.jogVelocity / viewModel.relZeroAbsPos      │
│  写入: viewModel.setJogVelocity(v) / viewModel.setRelativeZero() │
└──────────────────────────┬──────────────────────────────────┘
                           │ Q_PROPERTY / Q_INVOKABLE
┌──────────────────────────▼──────────────────────────────────┐
│  Presentation 适配层 (QtAxisViewModel)                       │
│  - Q_PROPERTY 将 C++ 值暴露给 QML 绑定系统                  │
│  - Q_INVOKABLE 将 QML 操作转发给 AxisViewModelCore          │
│  - 信号节流：仅在值变化超过阈值时发射 change 信号             │
└──────────────────────────┬──────────────────────────────────┘
                           │ 委托 (delegation)
┌──────────────────────────▼──────────────────────────────────┐
│  Presentation 核心层 (AxisViewModelCore)                     │
│  - 状态投影：直接读取 Axis 的 getter 方法                    │
│  - 控制输入：调用 Axis 的命令方法，或委托给 Orchestrator     │
│  不持有数据，仅为 Axis 的"视图投影"                          │
└──────────────────────────┬──────────────────────────────────┘
                           │ 直接引用 Axis&
┌──────────────────────────▼──────────────────────────────────┐
│  Domain 实体层 (Axis)                                       │
│  - 唯一数据真相源（Single Source of Truth）                  │
│  - applyFeedback(): 从 PLC 反馈同步状态                      │
│  - getter 方法: 纯查询，无副作用                              │
│  - setter 方法: 校验 → 生成 AxisCommand → 放入 pending_intent │
│  - 闭环消费：applyFeedback() 中根据 PLC 反馈自动清理意图      │
└──────────────────────────┬──────────────────────────────────┘
                           │ ISystemDriver
┌──────────────────────────▼──────────────────────────────────┐
│  Infrastructure 层 (PLC Driver / FakePLC)                    │
│  - 周期性读取 PLC 寄存器，构造 AxisFeedback                  │
│  - 将 AxisFeedback 喂给 Axis::applyFeedback()                │
│  - 周期性从 Axis::getPendingCommand() 取出命令下发到 PLC      │
└─────────────────────────────────────────────────────────────┘
```

## 3. 数据读取链路（展示）

### 3.1 通用原则

所有展示数据的流动方向是 **单向的、自底向上的**：

```
PLC 寄存器 → AxisFeedback 结构体 → Axis::applyFeedback() 更新成员变量
    → Axis::getXxx() 查询接口 → AxisViewModelCore 状态投影
    → QtAxisViewModel Q_PROPERTY → QML 绑定表达式 → UI 渲染
```

**关键设计决策：**
- `Axis` 是唯一数据真相源。ViewModel 不缓存数据，每次 QML 属性读取都直接穿透到 `Axis` 的 getter。
- `AxisFeedback` 结构体中的字段与 `Axis` 成员变量一一对应，`applyFeedback()` 执行纯赋值同步。

### 3.2 点动速度 / 定位速度（已实现，作为参考模板）

```
        │                                      │
```

### 3.3 相对零点位置（需要补充的链路）

**数据来源：** `AxisFeedback::relZeroAbsPos` — PLC 中记录"相对零点对应的绝对位置"的寄存器值。

**语义说明：**
- `relZeroAbsPos`：PLC 记录的相对零点基准值（绝对坐标系中的位置）。当用户执行"设置相对零点"后，PLC 会将当前绝对位置存入此寄存器；执行"清除相对零点"后，PLC 将其清零。
- `relPos`（相对位置）= `absPos - relZeroAbsPos`：当前绝对位置相对于相对零点的偏移。由 PLC 计算后填入 `AxisFeedback::relPos`。

**需要补充的读取链路：**

```
AxisFeedback::relZeroAbsPos
        │
        ▼
Axis::applyFeedback()
  m_rel_zero_abs_pos = feedback.relZeroAbsPos;  // ✅ 已有
        │
        ▼
Axis::relativeZeroAbsolutePosition() const      // ✅ 已有接口
  return m_rel_zero_abs_pos;
        │
        ▼
AxisViewModelCore::relZeroAbsPos()              // ❌ 需新增
  return m_axis.relativeZeroAbsolutePosition();
        │
        ▼
QtAxisViewModel::relZeroAbsPos()                // ❌ 需新增
  Q_PROPERTY(double relZeroAbsPos READ relZeroAbsPos NOTIFY relZeroChanged)
        │
        ▼
QML: viewModel.relZeroAbsPos                    // ❌ 需新增 UI 展示
```

**同时建议暴露相对位置（relPos）：**

```

        │
        ▼

        │
        ▼
```

## 4. 数据写入链路（设置操作）

### 4.1 通用原则

所有设置操作的流动方向是 **单向的、自顶向下的**：

```
QML 用户操作 → QtAxisViewModel Q_INVOKABLE → AxisViewModelCore 控制方法
    → Axis 命令方法（校验 + 生成意图）
    → AxisCommand 放入 m_pending_intent
    → Infrastructure 层取出命令下发 PLC
    → PLC 执行 → AxisFeedback 返回新状态
    → Axis::applyFeedback() 闭环判定 → 清理 pending_intent
```

**关键设计决策：**
- 设置操作在 `Axis` 层进行**前置校验**（状态准入、参数合法性），校验失败立即拒绝并设置 `RejectionReason`。
- 校验通过后生成 `AxisCommand` variant 放入 `m_pending_intent`，Infrastructure 层周期性轮询并下发。
- 闭环消费在 `applyFeedback()` 中完成：当 PLC 反馈的值与命令目标一致时，自动清理意图。
- 简单设置操作（如速度设置）**不经过 Orchestrator**，直接由 `AxisViewModelCore` 调用 `Axis` 方法即可。

### 4.2 点动速度设置（已实现，作为参考模板）

```
QML: viewModel.setJogVelocity(v)
        │
        ▼
QtAxisViewModel::setJogVelocity(v)                    // Q_INVOKABLE
  m_core->setJogVelocity(v);
  emit velocityChanged();
        │
        ▼
AxisViewModelCore::setJogVelocity(v)                  // 直接委托
  m_axis.setJogVelocity(v);
        │
        ▼
Axis::setJogVelocity(v)
  1. 校验 v > 0，否则返回 InvalidArgument
  2. 校验 state == Idle || Disabled，否则返回 InvalidState
  3. m_jog_velocity = v;                             // ✅ 乐观更新本地值
  4. m_pending_intent = SetJogVelocityCommand{v};    // ✅ 生成意图
        │
        ▼
Infrastructure 层轮询 getPendingCommand() → 下发到 PLC
        │
        ▼
PLC 执行后，下一轮 AxisFeedback::getjogVelocity 返回新值
        │
        ▼
Axis::applyFeedback() 中的闭环判定：
  if (auto* cmd = std::get_if<SetJogVelocityCommand>(&m_pending_intent)) {
      if (m_jog_velocity == cmd->velocity) {          // PLC 返回值与命令一致
          m_pending_intent = std::monostate{};         // ✅ 消费意图
      }
  }
```

### 4.3 设置相对零点（需要补充的链路）

**业务语义：** 用户点击"设置相对零点"按钮 → 系统将当前位置标记为相对坐标原点 → 后续 `relPos` 从此原点开始计算。

**Axis 层已有的实现（`Axis::setRelativeZero()`）：**

```cpp
bool Axis::setRelativeZero() {
    if (m_state != AxisState::Idle) {
        m_last_rejection = RejectionReason::InvalidState;
        return false;
    }
    m_expected_zero_base = m_current_abs_pos;          // 记录发起时的绝对位置
    m_pending_intent = SetRelativeZeroCommand{};       // 生成意图
    m_last_rejection = RejectionReason::None;
    return true;
}
```

**闭环消费逻辑（`applyFeedback()` 中）：**
```cpp
if (std::holds_alternative<SetRelativeZeroCommand>(m_pending_intent)) {
    bool isRelPosZero = std::abs(m_current_rel_pos) < POSITION_EPSILON;
    bool isBaseMatched = std::abs(m_rel_zero_abs_pos - m_expected_zero_base) < POSITION_EPSILON;
    if (isRelPosZero && isBaseMatched) {
        m_pending_intent = std::monostate{};           // ✅ 消费意图
    }
}
```

**需要补充的调用链路：**

```
QML 按钮点击
        │
        ▼
QtAxisViewModel::setRelativeZero()             // ❌ 需新增 Q_INVOKABLE
  bool ok = m_core->setRelativeZero();
  // 可选的：根据 ok 给用户反馈（如 Toast 提示）
        │
        ▼
AxisViewModelCore::setRelativeZero()           // ❌ 需新增
  return m_axis.setRelativeZero();
        │
        ▼
Axis::setRelativeZero()                        // ✅ 已有
  → 校验 state == Idle
  → 记录 m_expected_zero_base
  → 生成 SetRelativeZeroCommand 意图
        │
        ▼
Infrastructure 下发到 PLC → PLC 执行 → 反馈确认
        │
        ▼
applyFeedback() 闭环 → 意图消费
        │
        ▼
QML 通过 relZeroAbsPos 属性自动更新显示（因为 Axis 成员变量已更新）
```

### 4.4 清除相对零点（需要补充的链路）

**Axis 层已有的实现（`Axis::clearRelativeZero()`）：**

```cpp
bool Axis::clearRelativeZero() {
    if (m_state != AxisState::Idle) {
        m_last_rejection = RejectionReason::InvalidState;
        return false;
    }
    m_pending_intent = ClearRelativeZeroCommand{};     // 生成意图
    m_last_rejection = RejectionReason::None;
    return true;
}
```

**闭环消费逻辑：**
```cpp
if (std::holds_alternative<ClearRelativeZeroCommand>(m_pending_intent)) {
    bool isRelPosRestored = std::abs(m_current_rel_pos - m_current_abs_pos) < POSITION_EPSILON;
    bool isBaseReset = std::abs(m_rel_zero_abs_pos) < POSITION_EPSILON;
    if (isRelPosRestored && isBaseReset) {
        m_pending_intent = std::monostate{};           // ✅ 消费意图
    }
}
```

**需要补充的调用链路：** 与"设置相对零点"完全对称，只是调用 `m_axis.clearRelativeZero()`。

## 5. 需要新增的具体内容

### 5.1 AxisViewModelCore 需要新增

```cpp
// === 状态投影（只读） ===

/// 相对位置 = 绝对位置 - 相对零点
double relPos() const { return m_axis.currentRelativePosition(); }

/// 相对零点对应的绝对位置（PLC 寄存器值）
double relZeroAbsPos() const { return m_axis.relativeZeroAbsolutePosition(); }

// === 控制指令 ===

/// 设置当前位置为相对零点。返回 true 表示校验通过、意图已生成；false 表示被拒绝。
bool setRelativeZero() { return m_axis.setRelativeZero(); }

/// 清除相对零点，恢复相对坐标 = 绝对坐标。返回 true 表示校验通过。
bool clearRelativeZero() { return m_axis.clearRelativeZero(); }

/// 查询是否有待处理的相对零点操作
bool isRelativeZeroPending() const { return m_axis.hasPendingCommand(); }
```

## 6. 数据刷新机制

### 6.1 周期性刷新循环

```
Timer (e.g., 50ms)
    │
    ▼
Infrastructure: 读取 PLC 寄存器 → 构造 AxisFeedback
    │
    ▼
Axis::applyFeedback(feedback)           // 更新成员变量 + 闭环消费意图
    │
    ▼
QtAxisViewModel::tick()                 // 比较缓存值，按需 emit 信号
    │
    ▼
QML 属性绑定自动更新 UI
```

### 6.2 QtAxisViewModel 的信号节流策略

`QtAxisViewModel::tick()` 中采用缓存-比较-按需发射策略，避免高频信号导致 QML 过度重绘：

```cpp
void QtAxisViewModel::tick() {
    // 现有：state, absPos 的节流
    // 需新增：
    double newRelPos = m_core->relPos();
    if (std::abs(newRelPos - m_lastRelPos) > EPSILON) {
        m_lastRelPos = newRelPos;
        emit relPosChanged();
    }
    
    double newRelZero = m_core->relZeroAbsPos();
    if (std::abs(newRelZero - m_lastRelZeroAbsPos) > EPSILON) {
        m_lastRelZeroAbsPos = newRelZero;
        emit relZeroChanged();
    }
}
```

## 7. 设计决策总结

| 决策点 | 做法 | 原因 |
|--------|------|------|
| 数据真相源 | Axis 实体 | 单一真相源原则，避免 ViewModel 层缓存导致数据不一致 |
| 设置操作路由 | ViewModel → Axis 直接调用 | 简单设置操作不需要 Orchestrator 编排，Orchestrator 只负责多步骤运动流程（如 Jog 的按下-释放、Move 的启停判定） |
| 闭环确认 | applyFeedback() 中自动消费意图 | 命令-反馈确认模式，确保 UI 展示的状态与 PLC 真实状态一致 |
| 前置校验在 Axis 层 | Axis::setRelativeZero() 中校验 state == Idle | 领域逻辑集中在 Domain 层，ViewModel 层保持薄透传 |
| UI 反馈 | QML 属性绑定自动更新 | 数据变化通过信号-槽/QML 绑定自动反映到 UI，无需手动刷新 |
| relPos 来源 | PLC 计算后填入 AxisFeedback | 避免 UI/ViewModel 层自行计算（absPos - relZeroAbsPos），防止浮点精度不一致 |

## 8. 扩展性考虑

如果未来需要为相对零点操作增加更复杂的 UI 反馈（如操作进行中的加载动画、成功/失败 Toast），可以考虑：

1. **在 Axis 中增加 `isRelativeZeroOperationPending()` / `isRelativeZeroOperationSuccess()`** 查询接口
2. **在 QtAxisViewModel 中暴露对应的 Q_PROPERTY**，QML 根据状态显示不同的 UI
3. **或者利用 `RejectionReason`**：在设置操作返回 false 时，通过 `Axis::lastRejection()` 获取拒绝原因，映射为用户可读的提示信息

当前架构已为此预留了足够的扩展点（`RejectionReason` 枚举、`hasPendingCommand()` 查询），无需架构层面的改动。

# 统一命令总线与反馈分发 -- 架构重构思考

---

## 一、问题起点：编译错误暴露的深层架构问题

修改 `SystemContext.h` 将 `GantryGroup` 替换为 `GantryCouplingController` 后，编译暴露出：

```
fatal error: gantry/GantryGroup.h: No such file or directory
```

根因：`ISystemDriver.h` 中仍 `#include "gantry/GantryGroup.h"` 以获取 `GantryCommand` 类型。

这暴露了比文件缺失更深的架构问题：**Infrastructure 层直接依赖 Domain 层的具体聚合结构**。

---

## 二、命令方向：接口补丁化（Interface Patchwork）

### 2.1 当前结构

```
IAxisDriver
    send(AxisId, AxisCommand)

↓ 继承

ISystemDriver
    sendGantry(GantryCommand)         ← 第一块补丁
    sendGantryMotor(GantryPowerCommand)  ← 缺失，即将成为第二块补丁
```

### 2.2 问题本质

接口是按"命令来源的**聚合对象**"分类的，而不是按"命令的**抽象语义**"分类的。

| 当前分类依据 | 实际应分类依据 |
|------------|------------|
| Axis 发什么命令 | 系统需要发送什么硬件控制意图 |
| Gantry 发什么命令 | 使能、联动、解耦、定位、停止... |

### 2.3 违反的原则

**单一职责破坏（SRP）**

`ISystemDriver` 开始承担多种不相关的通信职责：
```
Axis + Gantry + 未来 Safety + 未来 IO + 未来 Alarm...
```

继续按聚合扩展，最终会演变成：
```cpp
class ISystemDriver {
    virtual void sendAxis(...) = 0;
    virtual void sendGantry(...) = 0;
    virtual void sendGantryMotor(...) = 0;
    virtual void sendIO(...) = 0;
    virtual void sendAlarm(...) = 0;
    virtual void sendSafety(...) = 0;
};
```
-- 典型的 Fat Interface 反模式。

**依赖方向违反 Clean Architecture**

```cpp
#include "gantry/GantryGroup.h"   // Infrastructure 引用 Domain 具体结构
```

Clean Architecture 要求：**外层依赖内层**，内层不应知道外层的存在。但这里的依赖是：

```
Domain (GantryCommand)
    ↑
Infrastructure (ISystemDriver)
```

这是**正确方向**----Domain 定义类型，Infrastructure 消费它。

**真正的问题不是方向，而是粒度。**

Infrastructure 不应该感知"这个命令是 Gantry 聚合产生的"----它只需要知道"这是一个需要发给 PLC 的控制命令"。

当前结构强制 Infrastructure 理解：
- 存在一个叫 Gantry 的领域聚合
- Gantry 有自己独特的命令协议
- 每种聚合都需要单独的 `sendXxx()` 通道

### 2.4 命令碎片化

当前领域层产生的命令类型：

| 领域对象 | 命令类型 |
|--------|--------|
| Axis | `AxisCommand` |
| GantryCouplingController | `GantryCouplingCommand` |
| GantryPowerController | `GantryPowerCommand` |
| 未来：EmergencyStop | `EmergencyStopCommand` |
| 未来：Safety | `SafetyCommand` |

每个领域对象各自定义自己的命令，Driver 接口随着命令类型线性膨胀。

## 三、反馈方向：applyFeedback 的隐式问题

### 3.1 当前结构

PLC 轮询线程直接调用多个 Domain 对象的 `applyFeedback`：

```cpp
// PLC 轮询线程伪代码
GantryFeedback snapshot = plc.getGantryFeedback();

powerController.applyFeedback(snapshot);    // 消费 .enable
couplingController.applyFeedback(snapshot); // 消费 .isCoupled / .errorCode
// 未来：safety.applyFeedback(snapshot);
// 未来：alarm.applyFeedback(snapshot);
```

### 3.2 当前做对的地方

- `GantryFeedback` 已经是独立 DTO，不依附于任何特定的领域对象
- 多个消费者共享同一份数据快照----正确的数据一致性策略
- 反馈方向是 **PLC -> Domain**，这是从外到内的正确依赖方向

### 3.3 当前存在的隐性问题

**（a）分发责任散落在调用方**

轮询线程需要知道：
- 哪些对象需要接收反馈
- 每个对象的 `applyFeedback` 签名
- 调用顺序

如果未来增加 SafetyController、AlarmController 等反馈消费者，轮询线程需要逐个添加调用。这使轮询线程成为"分发协调者"----一个本不该属于它的职责。

**（b）缺少统一的反馈分发点**

当前没有"反馈中心"这一概念。反馈是点对点推送到各个 Controller 的。

对比命令方向：命令至少（还）通过 Driver 接口统一发出。但反馈方向没有任何统一抽象----每个消费者各自被直接调用。

**（c）调用顺序不确定**

`powerController.applyFeedback()` 和 `couplingController.applyFeedback()` 谁先调用？如果未来两个消费者之间存在依赖关系（例如 SafetyController 需要在耦合状态变化时重置某些计数器），调用顺序就变得关键，但目前没有明确的编排。

**（d）GantryFeedback 的字段消费是隐式的**

```
powerController.applyFeedback()   -> 只读取 .enable
couplingController.applyFeedback() -> 只读取 .isCoupled / .errorCode
```

这种"按需消费"是合理的（ISP），但消费关系的文档散布在两个 Controller 的实现注释中，没有集中管理。新人难以一眼看出"哪些字段被谁消费"。

### 3.4 反馈方向是否需要重构？

**结论：反馈方向的问题严重性远低于命令方向，当前阶段不需要强制重构。**

理由：

| 维度 | 命令方向 | 反馈方向 |
|-----|--------|--------|
| 接口膨胀 | 紧迫，需要频繁新增 sendXxx | 不紧迫，消费者数量有限（2~3 个） |
| 依赖关系 | Driver 直接引用领域类型 | 调用方直接调用 Controller 方法 |
| 架构影响 | 阻碍扩展，违反 OCP | 可接受的内聚式设计 |
| 优先级 | 高 | 中低 |

如果未来消费者超过 3 个，可以考虑引入 `FeedbackDispatcher`：

```cpp
class FeedbackDispatcher {
    std::vector<IFeedbackConsumer*> consumers;
public:
    void dispatch(const GantryFeedback& snapshot) {
        for (auto* c : consumers) c->onFeedback(snapshot);
    }
};
```

但当前生产环境中这不是瓶颈。更值得优先解决的是命令方向的统一抽象。

## 四、推荐解决方案：统一命令总线

### 4.1 核心思想

```text
所有领域意图 -> 统一为 SystemCommand -> Infrastructure 统一处理
```

不是按"谁发的"分类，而是按"这是什么控制意图"抽象。

### 4.2 具体设计

**Step 1：定义统一命令 variant**

```cpp
// domain/command/SystemCommand.h（注：层次定位在 domain 层----Domain 定义意图）

#include "command/AxisCommand.h"
#include "gantry/GantryCouplingController.h"   // GantryCouplingCommand
#include "gantry/GantryPowerController.h"       // GantryPowerCommand

using SystemCommand = std::variant<
    AxisCommand,
    GantryCouplingCommand,
    GantryPowerCommand
>;
```

选择放在 `domain/command/` 的理由：
- 命令是领域意图的表达，属于 Domain 层
- `std::variant` 的模式与项目现有的 `UseCaseError` / `ContextRejection` 一致
- Infrastructure 只需依赖此统一类型，不再感知具体聚合结构

**Step 2：统一 Driver 接口**

```cpp
// infrastructure/ISystemDriver.h

class ISystemDriver : public IAxisDriver {
public:
    virtual ~ISystemDriver() = default;

    // 统一命令发送入口
    virtual void send(const SystemCommand& cmd) = 0;
};
```

关键变化：
- 不再有 `sendGantry()`、`sendGantryMotor()` 等按聚合分类的方法
- 新增任何一种命令类型，只需要：
  1. 在 `SystemCommand` variant 中添加该类型
  2. 在 Driver 实现中添加对应的 `std::visit` handler
- 接口本身保持稳定（开闭原则）

**Step 3：Driver 实现侧使用 std::visit 分发**

```cpp
void FakeAxisDriver::send(const SystemCommand& cmd) {
    std::visit([this](auto&& c) { handle(c); }, cmd);
}

void FakeAxisDriver::handle(const AxisCommand& cmd) {
    // 现有 Axis 命令处理逻辑
    plc.onCommand(..., cmd);
}

void FakeAxisDriver::handle(const GantryCouplingCommand& cmd) {
    // 联动/解耦命令处理
    plc.onGantryCommand(cmd);
}

void FakeAxisDriver::handle(const GantryPowerCommand& cmd) {
    // 龙门电机使能
    plc.onCommand(AxisId::X1, EnableCommand{cmd.enable});
    plc.onCommand(AxisId::X2, EnableCommand{cmd.enable});
}
```

**Step 4：聚合层如何连接**

```
GantryOrchestrator::tick()
  ├── motor.requestEnable(true)          -> motor.hasPendingCommand() -> GantryPowerCommand
  ├── coupling.requestCouple(true)       -> coupling.hasPendingCommand() -> GantryCouplingCommand
  └── // Orchestrator 收集所有 Pending Commands
      for each Pending Command:
          driver->send(toSystemCommand(cmd))
```

`toSystemCommand()` 负责将具体的 `GantryPowerCommand` / `GantryCouplingCommand` / `AxisCommand` 包装为 `SystemCommand` variant。

### 4.3 架构对比

| 维度 | 修复前 | 修复后 |
|-----|-------|-------|
| 新增命令类型 | 修改 ISystemDriver 接口 + 所有实现类 | 只需修改 SystemCommand variant |
| Driver 对 Domain 的感知 | 知道 Gantry 聚合存在 | 只知道"有命令要发" |
| 接口方法数量 | 线性增长 | 始终 1 个 send() |
| 符合 OCP | ❌ 接口必须修改 | ✅ 扩展 variant 即可 |
| 与项目风格一致性 | -- | ✅ 与 UseCaseError variant 一致 |

### 4.4 需要注意的问题

**`IAxisDriver` 与 `ISystemDriver` 的关系**

如果保留 `IAxisDriver`，`ISystemDriver` 不应继承它。因为：
- `IAxisDriver::send(AxisId, const AxisCommand&)` 的签名与统一 `send(const SystemCommand&)` 语义重叠
- 更好的方案：`ISystemDriver` 独立存在，不继承任何接口，只暴露统一入口

或者如果 `IAxisDriver` 在 application 层被广泛使用：
- 保留 `IAxisDriver`，但不再使其继承进 `ISystemDriver`
- `IAxisDriver` 仅用于单轴 UseCase 的测试 double
- `ISystemDriver` 作为系统的唯一真实出口

## 五、为什么这个问题现在必须正视

### 5.1 当前处于架构演化节点

项目已经从"只有单轴控制"演化到"龙门联动 + 电机使能"阶段。命令类型从 1 种增长到 3 种，且增长趋势明确。

如果现在不统一抽象，每增加一种领域能力，就需要在 Driver 接口上打一个补丁。后期重构成本会指数级上升。

### 5.2 工业控制架构的关键分岔

```
┌─── 路径 A：按聚合扩展接口 ───-> Fat Interface -> 越来越难维护
│
└─── 路径 B：统一命令总线 ─────-> 稳定的 Driver -> 开闭原则
```

现在正是选择路径 B 的最佳时机。

---

## 六、实施建议与顺序

### 优先级排序

| 序号 | 事项 | 紧迫性 | 说明 |
|-----|------|-------|-----|
| 1 | 定义 `SystemCommand` variant | 高 | 这是后续所有变更的基础 |
| 2 | 统一 `ISystemDriver` 接口为单一 `send()` | 高 | 消除当前的编译错误 + 架构债务 |
| 3 | 更新 `FakeAxisDriver` 实现 | 高 | 适配统一接口 |
| 4 | 更新 `GantryOrchestrator` 的 send 路径 | 中 | 改用统一 `send()` |
| 5 | 移除 `ISystemDriver` 对 `IAxisDriver` 的继承 | 中 | 消除接口层次的语义冲突 |

### 反馈方向暂不纳入重构范围

`applyFeedback` 的消费者当前仅有 2 个，且 `GantryFeedback` DTO 已经提供了合理的抽象层。反馈分发问题在当前规模下可接受，待到消费者超过 3 个再考虑引入 `FeedbackDispatcher`。

---

## 七、总结

| 问题 | 根因 | 解决方案 |
|-----|-----|--------|
| 编译错误 | `ISystemDriver.h` 引用已废弃的 `GantryGroup.h` | 引入统一命令接口 |
| 接口膨胀 | 按领域聚合分类接口方法 | 统一 `SystemCommand` variant |
| Driver 感知 Domain 结构 | `sendGantry(GantryCommand)` | `send(SystemCommand)`，消息对 Driver 透明 |
| applyFeedback 分发 | 轮询线程直接调用多个 Controller | 当前可接受，未来可选 FeedbackDispatcher |
| Fat Interface 趋势 | 缺乏抽象命令层 | 命令层统一为 variant + visitor |

核心结论：**Domain 输出抽象命令，Infrastructure 负责物理化，中间不应让 Infrastructure 感知命令的具体聚合来源。**

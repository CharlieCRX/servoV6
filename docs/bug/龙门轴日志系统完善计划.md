# 龙门轴日志系统完善计划

> **版本**: 1.0
> **背景**: 龙门轴（双轴龙门联动）的日志系统与单轴控制系统相比存在严重缺失，导致使用诊断时无法通过日志定位故障源自哪一层。
> **对比基准**: 单轴控制系统（`AxisViewModelCore` + `Axis` + `FakePLC` + `FakeAxisDriver`）已具备较为完整的命令生命周期日志。

---

## 一、当前日志覆盖对比总览

| 模块 | 单轴日志覆盖 | 龙门日志覆盖 | 差距 |
|------|------------|------------|------|
| ViewModel 层 | ✅ 全面 | ❌ 几乎为零 | 致命 |
| Application/Policy 层 | ✅ Orchestrator 有日志 | ❌ 完全空白 | 致命 |
| Domain 层 (PowerController) | ✅ Axis 有决策日志 | ❌ 完全空白 | 严重 |
| Domain 层 (CouplingController) | ✅ Axis 有决策日志 | ❌ 完全空白 | 严重 |
| Domain 层 (CouplingState) | ✅ Axis 有 state 日志 | ❌ 完全空白 | 中等 |
| Infrastructure/Driver 层 | ✅ 有 TRACE/DEBUG | ❌ 龙门反馈无日志 | 中等 |
| Error 收集机制 | ✅ `pushError` + `m_errorHistory` | ❌ 无错误收集 | 严重 |
| TraceScope 贯穿 | ✅ 每个控制方法 | ❌ 完全未使用 | 严重 |
| 命令格式化日志 | ✅ `CommandFormatter` | ❌ 未使用 | 中等 |

## 二、逐层日志缺失详细分析

### 2.1 ViewModel 层 — `GantryViewModel.cpp`

**当前状态**: 完全无日志。与单轴 `AxisViewModelCore.cpp` 形成鲜明对比。

**缺失清单**:

| 位置 | 单轴对照 | 龙门缺失 | 影响 |
|------|---------|---------|------|
| `#include` | `#include "infrastructure/logger/Logger.h"` | **无此 include** | 无法编译任何日志宏 |
| `#include` | `#include "infrastructure/logger/TraceScope.h"` | **无此 include** | 无法创建 TraceScope |
| 构造函数 | `LOG_INFO(LogLayer::UI, "AxisVM", "created")` | **无日志** | 无法确认 ViewModel 是否创建 |
| `startCoupling()` | `LOG_INFO(LogLayer::UI, "AxisVM", "...")` | **无日志** | 无法追踪用户操作入口 |
| `stopCouplingAndDisable()` | 同上 | **无日志** | 同上 |
| `enableAndDecouple()` | 同上 | **无日志** | 同上 |
| `enable()` | `LOG_INFO(LogLayer::UI, "AxisVM", "...")` | **无日志** | 同上 |
| `disable()` | `LOG_INFO(LogLayer::UI, "AxisVM", "...")` | **无日志** | 同上 |
| `refreshGantryState()` | 无直接对应 | **无日志** | 无法追踪状态缓存变化 |
| `refreshOrchestratorState()` | 无直接对应 | **无日志** | 无法追踪编排器状态变化 |
| `advanceOrchestrator()` | 无直接对应 | **无日志** | 无法追踪编排器推进过程 |
| `tick()` | `LOG_TRACE_EVERY_N(100, ...)` | **无日志** | 无法确认 tick 循环是否运行 |
| 错误收集 | `pushError()` + `LOG_WARN` | **无任何错误收集** | 错误信息丢失，无法向 UI 传播 |

**根因**：
- `GantryViewModel` 的头文件未包含日志系统
- ViewModel 设计时忽略了可观测性需求
- 所有方法都是 `void` 返回值（正确），但缺少副作用日志

## 三、与单轴日志系统的详细对比

### 3.1 AxisViewModelCore 的日志模式（龙门应采用的参考）

```
结构模板：
┌──────────────────────────────────────────────────────┐
│ void AxisViewModelCore::enable(bool active)          │
│ {                                                    │
│     TraceScope scope(...);        ← 贯穿追踪          │
│     LOG_INFO("enable requested"); ← 用户操作入口      │
│     auto result = uc->execute();                      │
│     if (!ok) {                                        │
│         pushError(vmError, "EnableUC");               │
│         LOG_ERROR("enable failed: " + code); ← 错误   │
│     } else {                                          │
│         LOG_DEBUG("enable accepted");  ← 接受确认    │
│     }                                                 │
│ }                                                     │
└──────────────────────────────────────────────────────┘
```

### 3.2 当前 GantryViewModel 的实际代码

```
void GantryViewModel::enable() {
    auto* ctx = getContext();
    if (!ctx) return;       // ❌ 静默失败

    auto& power = ctx->gantryPowerController();
    power.requestEnable(true);  // ❌ 结果被丢弃
}
```

**核心差异**：
1. **无 TraceScope** — 丢失 `[group][axis][traceId]` 三要素
2. **无 LOG_INFO** — 用户操作不可追踪
3. **无返回值检查** — `requestEnable()` 的 `GantryRejection` 被丢弃
4. **无 pushError** — 错误信息不传播到 UI
5. **无日志 include** — 即使想加也无法编译

#### 第三阶段（P2 — 中优先级）：Domain 实体日志

**目标**：解决"power/coupling 内部状态机决策细节"的盲区。

**GantryPowerController.h 改造**：

```cpp
#include "infrastructure/logger/Logger.h"

GantryRejection requestEnable(bool active) {
    if (m_status == Status::NotSynchronized) {
        LOG_WARN(LogLayer::DOM, "GantryPower",
            "requestEnable(" + std::to_string(active)
            + ") rejected: NotSynchronized");
        return GantryRejection::NotSynchronized;
    }
    if (m_status == Status::Enabling || m_status == Status::Disabling) {
        LOG_WARN(LogLayer::DOM, "GantryPower",
            "requestEnable(" + std::to_string(active)
            + ") rejected: StateConflict, current status=" + statusToString(m_status));
        return GantryRejection::StateConflict;
    }

    // 幂等检查
    if (active && m_status == Status::Enabled) {
        LOG_DEBUG(LogLayer::DOM, "GantryPower",
            "requestEnable(true) idempotent: already Enabled");
        return GantryRejection::None;
    }
    if (!active && m_status == Status::Disabled) {
        LOG_DEBUG(LogLayer::DOM, "GantryPower",
            "requestEnable(false) idempotent: already Disabled");
        return GantryRejection::None;
    }

    m_pending_command = GantryPowerCommand{active};
    m_status = active ? Status::Enabling : Status::Disabling;
    LOG_DEBUG(LogLayer::DOM, "GantryPower",
        "requestEnable(" + std::to_string(active)
        + ") accepted: status " + statusToString(m_status)
        + ", pending command generated");
    return GantryRejection::None;
}

void applyFeedback(const GantryFeedback& feedback) {
    Status oldStatus = m_status;
    m_status = feedback.enable ? Status::Enabled : Status::Disabled;
    if (oldStatus != m_status) {
        LOG_DEBUG(LogLayer::DOM, "GantryPower",
            "applyFeedback: " + statusToString(oldStatus)
            + " -> " + statusToString(m_status)
            + " (PLC enable=" + std::to_string(feedback.enable) + ")");
    } else {
        LOG_TRACE(LogLayer::DOM, "GantryPower",
            "applyFeedback: no change, status=" + statusToString(m_status));
    }
}

// 新增辅助方法
static std::string statusToString(Status s) {
    switch (s) {
        case Status::NotSynchronized: return "NotSynchronized";
        case Status::Disabled: return "Disabled";
        case Status::Enabling: return "Enabling";
        case Status::Enabled: return "Enabled";
        case Status::Disabling: return "Disabling";
        default: return "Unknown";
    }
}
```

### 4.2 架构决策

#### 4.2.1 header-only vs .cpp 分离

**现状**：
- `GantryOrchestrator` — header-only（~180行）
- `GantryPowerController` — header-only（~100行）
- `GantryCouplingController` — header-only（~180行）
- `GantryCouplingState` — header-only（~80行）

**问题**：添加日志后，所有头文件体积将膨胀 30%~60%。

**建议**：

| 文件 | 当前 | 改造后体积 | 建议方案 |
|------|------|-----------|---------|
| `GantryOrchestrator.h` | ~180行 | ~350行 | **分离**：`tick()` 移入 `.cpp` |

### 4.4 TraceScope 贯穿设计

**当前 TraceScope 使用情况**：

```
单轴: AxisVM::enable() → TraceScope → Axis::setRelativeZero() → Driver::send() → PLC::processCommand()
                                       ↓
                                  同一调用栈，scope 自动贯穿 ✅

龙门: GantryVM::startCoupling() → 创建 GantryOrchestrator → tick() 中调用
                                  GantryPowerController / GantryCouplingController
                                       ↓
                                  同一调用栈，scope 可以贯穿
```

**设计**：在 `GantryViewModel` 的每个操作入口创建 TraceScope：

```cpp
void GantryViewModel::startCoupling() {
    TraceScope scope(m_groupName, "Gantry", generateTraceId());
    // ...
    // tick() 中的 GantryOrchestrator::tick()
    // → GantryPowerController::requestEnable()
    // → GantryPowerController::applyFeedback()
    // 都在同一线程的同一调用栈中，scope 自动贯穿所有日志
}
```

**好处**：添加 TraceScope 后，所有日志行会自动携带 `[group][axis][traceId]` 标签，例如：
```
[17:23:45.110][INFO][UI][GantryVM][Grp1][Gantry][Gantry_a3f2] startCoupling requested
[17:23:45.120][DEBUG][APP][GantryOrch][Grp1][Gantry][Gantry_a3f2] EnsuringEnabled -> WaitingEnabled
[17:23:45.121][DEBUG][DOM][GantryPower][Grp1][Gantry][Gantry_a3f2] requestEnable(true) accepted: Enabling
```

## 七、风险与注意事项

### 7.1 头文件膨胀

`GantryOrchestrator.h` 当前是 header-only，添加日志后将从 180 行膨胀到 350 行。建议 Phase 3 分离到 `.cpp`。

### 7.2 测试隔离

所有日志宏仅输出到日志系统，不改变控制流。测试用例不需要因日志而修改断言。唯一需要的改动是测试 fixture 的 `SetUp`/`TearDown` 中添加 Logger 初始化/关闭。

### 7.3 性能考虑

| 高频路径 | 频率 | 日志策略 |
|---------|------|---------|
| `GantryViewModel::tick()` | 每 10ms | 使用 `LOG_TRACE_EVERY_N(100)` 节流 |
| `GantryOrchestrator::tick()` | 每 10ms | 稳态等待 case 用 TRACE，转换 case 用 DEBUG |
| `GantryPowerController::applyFeedback()` | 每 10ms | 状态变化时 DEBUG，无变化时 TRACE |
| `GantryCouplingController::applyFeedback()` | 每 10ms | 同上 |

### 7.4 宏名称冲突

当前已有 `LOG_DEBUG` / `LOG_INFO` / `LOG_WARN` 等宏。注意头文件包含顺序和宏保护，不要与 QML 或 Qt 的日志宏冲突。

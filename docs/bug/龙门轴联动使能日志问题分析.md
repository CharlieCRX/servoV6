## 二、问题 1：GantryOrch 层日志全是 N/A，TraceScope 生命周期不足

### 2.1 现象

| 行号 | 日志行 | group | axis | traceId |
|------|--------|-------|------|---------|
| L03-L04 | `[GantryVM]` startCoupling | ✅ Machine_A | ✅ Gantry | ✅ Gantry_xxx |
| L05-L08 | `[GantryOrch]` EnsuringEnabled | ❌ N/A | ❌ N/A | ❌ N/A |
| L09-L10 | `[GantryVM]` orchBusy/orchStep | ❌ N/A | ❌ N/A | ❌ N/A |
| L13-L21 | `[GantryOrch]` Coupling... | ❌ N/A | ❌ N/A | ❌ N/A |

### 2.2 根因分析

**当前 TraceScope 生命周期：**

```cpp
// GantryViewModel.cpp - startCoupling()
void GantryViewModel::startCoupling() {
    TraceScope scope(m_groupName, "Gantry", generateTraceId());  // ← scope 创建
    LOG_INFO(LogLayer::UI, "GantryVM", m_groupName + " startCoupling requested");
    // ... 创建 orchestrator ...
    m_orchestrator = std::make_unique<GantryOrchestrator>(...);
    m_orchestrator->startCoupling();
    LOG_DEBUG(...);
}  // ← scope 销毁！此后 tick() 中所有日志丢失上下文
```

之后在主循环中每个 tick 调用：

```cpp
// main.cpp tick loop
gantryVM_A.tick();                              // ❌ 无 TraceScope
  → advanceOrchestrator()
    → m_orchestrator->tick()                    // ❌ 无上下文 → 所有日志 N/A
  → refreshGantryState()                        // ❌ 无上下文 → 日志 N/A
  → refreshOrchestratorState()                  // ❌ 无上下文 → N/A
```

**核心矛盾：**
- `TraceScope` 是 RAII 对象，绑定到线程局部栈，创建的方法退出即销毁
- 联动编排是**跨帧异步过程**：`startCoupling()` 在第 1 帧创建 orchestrator，后续 tick 在后续帧逐帧推进
- GantryOrchestrator::tick() 在 **N 帧之后**才执行，此时 `startCoupling()` 的 TraceScope 早已销毁

### 2.3 影响

1. **无法关联日志到具体分组和操作**：15 条 GantryOrch 日志全部丢失 Machine_A/Gantry 上下文
2. **并发问题未暴露**：若未来 Machine_A 与 Machine_B 同时联动，日志将无法区分
3. **故障排查困难**：error 日志无 group 信息，无法定位问题分组

### 2.4 解决方案

**方案：在 tick() 入口统一创建 TraceScope**

在 `GantryViewModel::tick()` 和 `advanceOrchestrator()` 中创建 TraceScope，确保整个编排流程都在上下文中运行。

```cpp
// GantryViewModel::tick() 改造
void GantryViewModel::tick() {
    // 当 orchestrator 活跃时，为整个 tick 创建上下文
    if (m_orchestrator && !m_orchestrator->isDone() && !m_orchestrator->hasError()) {
        // 使用 orchestrator 的 traceId（在 startCoupling 时生成并保存）
        TraceScope scope(m_groupName, "Gantry", m_activeTraceId);
    }
    
    advanceOrchestrator();
    refreshGantryState();
    refreshOrchestratorState();
}
```

或者更简洁的方式 —— 在 `advanceOrchestrator()` 内部：

```cpp
void GantryViewModel::advanceOrchestrator() {
    if (!m_orchestrator) return;
    if (m_orchestrator->isDone() || m_orchestrator->hasError()) {
        LOG_TRACE_EVERY_N(50, ...);
        return;
    }
    
    // 为整个 tick 创建 TraceScope，确保 GantryOrchestrator 内所有日志有上下文
    TraceScope scope(m_groupName, "Gantry", m_activeTraceId);
    m_orchestrator->tick();
}
```

**需要配套修改：**
- `GantryViewModel` 新增成员 `std::string m_activeTraceId;`
- 在 `startCoupling()` / `stopCouplingAndDisable()` / `enableAndDecouple()` 中生成 traceId 并保存到 `m_activeTraceId`
- `GantryOrchestrator` 无需修改，因为它本身不持有 Logger 上下文

## 三、问题 2：applyFeedback 日志不知道是哪台机器的哪根轴

### 3.1 现象

```
L11 [14:10:57.363][DEBUG][DOM][Axis][N/A][N/A][N/A] applyFeedback: state Disabled(1) -> Idle(2)
L12 [14:10:57.363][DEBUG][DOM][Axis][N/A][N/A][N/A] applyFeedback: state Disabled(1) -> Idle(2)
L22 [14:10:57.472][DEBUG][DOM][Axis][N/A][N/A][N/A] applyFeedback: state Unknown(0) -> Disabled(1)
```

- 两条 `Disabled → Idle` 的日志，**无法区分**是 X1 还是 X2（或两者）
- `Unknown → Disabled` 也不知道是哪个轴
- Domain 层日志的 `group`/`axis` 字段全部是 `N/A`

### 3.2 根因分析

`Axis::applyFeedback()` 由 `FakeAxisDriver::pollFeedback()` 在 tick loop 中调用：

```cpp
// FakeAxisDriver.h - pollFeedback 伪代码
void pollFeedback(SystemContext& ctx) {
    for (auto id : ALL_AXES) {             // 循环遍历所有轴
        AxisFeedback fb = m_plc.getFeedback(id);
        ctx.axis(id).applyFeedback(fb);    // ← Axis::applyFeedback 内直接 LOG，无 TraceScope
    }
}
```

`main.cpp` 的 tick loop：
```cpp
// main.cpp
drv->pollFeedback(*ctx);  // ← 无 TraceScope wrapper
```

### 3.3 解决方案

**方案：在 FakeAxisDriver::pollFeedback 中为每个轴创建 TraceScope**

```cpp
// FakeAxisDriver.h
void pollFeedback(SystemContext& context) {
    for (const auto& [name, axisCtx] : context.allAxes()) {  // 假设有 allAxes()
        TraceScope scope(m_groupName, axisIdToString(axisCtx.id), "");
        auto fb = m_plc.getFeedback(axisCtx.id);
        axisCtx.axis.applyFeedback(fb);
    }
}
```

但 FakeAxisDriver 目前不持有 `groupName`。需要两种方式之一：

**子方案 A**：从 SystemContext 获取 groupName（如果 SystemContext 存储了它）
**子方案 B**：在 main.cpp tick loop 中包裹 TraceScope：

```cpp
// main.cpp tick loop
for (const auto& groupName : manager.groupNames()) {
    SystemContext* ctx = nullptr;
    if (manager.tryGetGroup(groupName, ctx, r) && ctx && ctx->driver()) {
        // ✅ 在分组级别创建 TraceScope
        TraceScope groupScope(groupName, "", "");
        ctx->driver()->pollFeedback(*ctx);
    }
}
```

但这只能解决 group 问题，无法区分具体 axis。更彻底的方案：

**推荐方案：在 Axis::applyFeedback 内部使用所在 Axis 实体的名称**

这需要 `Axis` 实体持有自己的 `AxisId` 和所属 `groupName`，或从上层传入。当前架构中 Axis 实体不持有这些信息，因此需要重构。

**低侵入方案**（推荐立即实施）：

1. **main.cpp 层**：在 `drv->pollFeedback()` 外包装 group 级别的 TraceScope
2. **Domain 层**：在 `AxisEntity` 中保存其 `AxisId`（当前已有 `AxisId.h`），在 `applyFeedback` 时将 `axisId` 写入日志

```cpp
// domain/entity/Axis.h 添加
class Axis {
    // ...
    void setIdentity(AxisId id, const std::string& groupName) {
        m_id = id;
        m_group = groupName;
    }
private:
    AxisId m_id = AxisId::Unknown;
    std::string m_group;
};

// domain/entity/Axis.cpp
void Axis::applyFeedback(const AxisFeedback &feedback) {
    TraceScope scope(m_group, axisIdToString(m_id), "");
    // ... 现有逻辑 ...
}
```

这样 Domain 层的 `applyFeedback` 日志将带 `[Machine_A][X1][]` 等上下文。

## 四、问题 3：联动成功后 X 轴状态仍为 Disabled

### 4.1 现象

联动流程完成后（L23-L28 确认 `Done`），telemetry 显示：

```
L01 [14:10:56.202] X: pos=+0.0 Unknown   X1: pos=+0.0 Disabled  X2: pos=+0.0 Disabled
    ...
L29 [14:10:58.203] X: pos=+0.0 Disabled  X1: pos=+0.0 Unknown   X2: pos=+0.0 Unknown
```

- X1、X2 进入 `Unknown` → `Idle` 过渡态（龙门使能生效）
- **X 轴保持 `Disabled`**（甚至从 `Unknown` 退化到 `Disabled`）

### 4.2 根因分析

**关键理解：当前架构中 X、X1、X2 是三个独立的 Axis 实体**

```
Machine_A: {
    X   → 单轴 Axis 实体（不参与龙门联动）
    X1  → 龙门左侧轴
    X2  → 龙门右侧轴
}
```

联动流程中的使能逻辑：

```
GantryOrchestrator::tick()
  → GantryPowerController::requestEnable(true)
    → 生成 GantryPowerCommand{enable=true}
      → ISystemDriver::send() → FakePLC::onGantryCommand()
        → FakePLC::tickGantryPower()
          → 直接修改 X1.feedback.state = Idle    ← ✅ X1 被使能
          → 直接修改 X2.feedback.state = Idle    ← ✅ X2 被使能
          → X 轴不参与此流程                        ← ❌ X 未被修改
```

**回顾 FakePLC::tickGantryPower()：**

```cpp
void tickGantryPower(int ms) {
    // ...
    if (m_gantryPowerTarget) {
        // 使能：只修改 X1 和 X2
        if (x1.feedback.state == AxisState::Disabled) {
            x1.feedback.state = AxisState::Idle;
        }
        if (x2.feedback.state == AxisState::Disabled) {
            x2.feedback.state = AxisState::Idle;
        }
    }
}
```

**结论：X 单轴实体从未被龙门使能流程触及，其状态变化完全依赖独立的 EnableCommand**

### 4.3 这是 Bug 还是设计决策？

#### 论证

| 角度 | X 单轴的意义 | 结论 |
|------|-------------|------|
| **龙门模式下** | X 代表整个龙门的逻辑位置（X1 + X2 同步运动的虚拟轴） | X 应反映龙门综合状态 |
| **非龙门模式下** | X 是一个独立可用的单轴 | X 自己的 enable/disable 独立管理 |
| **当前实现** | X 是独立 Axis 实体，与 X1/X2 无关系 | X 的 enable 和 gantry enable 是两个通道 |

**当前行为在"龙门 X 单轴"视角下是正确的**：X 单轴从未被使能，所以 It's Disabled。

#### 用户体验问题

用户期望：龙门联动 = 龙门 X 轴被使能并联动。UI 上 X 轴显示 `Disabled` 会让用户困惑——明明已经联动成功了，为什么 X 还是 Disabled？

这个困惑的根源是 **"X 轴"在 UI 上的语义模糊**：
- 在龙门模式下，"X" 是逻辑龙门轴
- 在单轴模式下，"X" 是一个独立的物理轴

### 4.4 解决方案

根据架构选择，有三种方案：

#### 方案 A：去除 X 单轴（推荐）

既然 Machine_A 是龙门架构（X1/X2 成对），**X 单轴不应同时存在**。

```
Machine_A（龙门组）:
    Y, Z, R     ← 独立单轴
    X1, X2      ← 龙门轴对

Machine_B（如果需要独立 X 单轴）:
    X           ← 独立单轴
```

- 从 Machine_A 移除 X 单轴的 ViewModel
- QML TelemetryBlock 中只显示 X1/X2 龙门轴对
- 龙门联动时通过 GantryViewModel 显示龙门状态

#### 方案 B：X 轴状态投影为龙门综合状态

在 `QtAxisViewModel` 的 `state()` 方法中，对 X 轴做特殊处理：

```cpp
// QtAxisViewModel.cpp 或 AxisViewModelCore.cpp
AxisState AxisViewModelCore::state() const {
    if (m_axisId == AxisId::X) {
        // X 是龙门逻辑轴：返回龙门综合状态
        auto& power = ctx->gantryPowerController();
        auto& coupling = ctx->gantryCouplingController();
        if (power.isEnabled()) {
            return coupling.isCoupled() ? AxisState::GantryCoupled : AxisState::Idle;
        }
        return AxisState::Disabled;
    }
    return m_axis->state();
}
```

此方案在 ViewModel 层做投影，不改变 Domain 模型。但这种特殊逻辑让代码耦合度增加。

#### 方案 C：在 FakePLC 层级联动使能同时更新 X 轴状态

```cpp
void FakePLC::tickGantryPower(int ms) {
    // ... 现有逻辑 ...
    if (m_gantryPowerTarget) {
        // 同时更新 X 单轴（龙门逻辑轴）
        auto& x = m_axes.at(AxisId::X);
        if (x.feedback.state == AxisState::Disabled) {
            x.feedback.state = AxisState::Idle;  // ← 新增
        }
    }
}
```

但此方案混淆了"龙门使能"和"单轴使能"的语义，不推荐。

### 4.5 推荐方案

**短期**（方案 A）：从 Machine_A 的轴列表中移除 X 单轴，只保留 X1/X2 龙门对。
- UI 的 TelemetryBlock 中让龙门轴对在联动时高亮显示
- 联动状态通过 `GantryViewModel.isCoupled()` 来表示

**长期**（方案 B）：如果业务确实需要一个"逻辑 X 轴"来代表龙门整体，在 ViewModel 层做状态投影。

### 7.1 TraceScope 生命周期示意

```
Frame N:   GantryViewModel::startCoupling()
           ├── TraceScope(Machine_A, Gantry, Gantry_xxx)   ← 创建
           │   ├── LOG "[Machine_A][Gantry][Gantry_xxx]"   ← ✅
           │   └── 创建 orchestrator
           └── }                                           ← 销毁

Frame N+1: GantryViewModel::tick()
           ├── advanceOrchestrator()
           │   └── orchestrator->tick()
           │       └── LOG "[N/A][N/A][N/A]"               ← ❌ 无 context
           └── refreshGantryState()
               └── LOG "[N/A][N/A][N/A]"                   ← ❌ 无 context
```

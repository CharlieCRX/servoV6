# 日志系统缺失分析——命令端到端可观测性整改方案

> **版本**: 2.0（基于完整源码审查的精确修订）
> **背景**: 排查"点动完成后设置相对零点无效"问题时，发现无法通过日志定位故障源自哪一层，只能通过阅读源码逐层追踪。

---

## 一、现有日志基础设施评估

项目已具备较完善的日志基础设施，问题在于**使用不充分、不均衡**：

| 能力 | 状态 | 说明 |
|------|------|------|
| 分层日志宏 `LOG_TRACE/DEBUG/INFO/WARN/ERROR/SUMMARY` | ✅ 已有 | `Logger.h` 定义了完整宏体系 |
| 分层标签 `LogLayer::UI / APP / DOM / HAL` | ✅ 已有 | 但 DOM 层**从未使用** |
| `TraceScope` 贯穿追踪 | ✅ 已有 | 仅 ViewModel 层使用，未向下传递 |
| `LogContext` 携带 group/axis/traceId | ✅ 已有 | `Logger::log()` 自动读取当前 scope |
| 节流宏 `LOG_TRACE_EVERY_N` | ✅ 已有 | 仅 FakePLC tick 使用 |
| `CommandFormatter` 命令序列化 | ✅ 已有 | 仅 Driver 和 ViewModel 使用 |
| 异步日志队列 + 文件/控制台独立级别 | ✅ 已有 | 默认控制台 `minConsoleLevel=INFO` |
| 按级别过滤 | ✅ 已有 | `minConsoleLevel` 已可运行时配置 |

**核心结论**：不需要新建日志基础设施。只需要在**正确的层级用正确的宏补全日志调用**。

---

## 二、配置层面调整（前置工作）

当前默认 `minConsoleLevel = INFO` 意味着所有 `LOG_TRACE` 和 `LOG_DEBUG` 在生产环境默认不输出。为了支持调试：

1. 保持生产默认 `INFO`
2. 运行时可通过修改 `Logger::init()` 参数切换到 `DEBUG` 或 `TRACE`
3. 或将 `minConsoleLevel` 改为支持运行时动态调整（可选增强）

**约定**：本方案中所有新增日志遵循以下级别规则：
- `INFO`：用户可见的操作（如按钮点击、命令请求）
- `DEBUG`：命令流转的关键分支、闭环消费、拒绝原因
- `TRACE`：高频细粒度（每帧反馈注入、运动学推演）
- `WARN`：静默拦截但不算错误的情况
- `ERROR`：硬件/通讯异常、不可恢复的错误

### 节点 [C/D/E]：`consumePendingCommands()` 分类判断

**源码位置**：`presentation/viewmodel/AxisViewModelCore.cpp:consumePendingCommands()`

**当前状态**：
- `hasPendingCommand()` 检查 → 无日志 ❌
- `isZeroOrVelocity` 分类判断 → 无日志 ❌
- 获取 group → 失败有 ERROR ✅，成功无日志 ❌
- 获取 driver → 失败有 ERROR ✅，成功无日志 ❌

**改造**：在关键分支添加 DEBUG 日志。

```cpp
void AxisViewModelCore::consumePendingCommands()
{
    auto* axis = tryGetAxis(m_manager, m_groupName, m_axisId);
    if (!axis) return;

    if (!axis->hasPendingCommand()) return;

    const AxisCommand& cmd = axis->getPendingCommand();

    bool isZeroOrVelocity =
        std::holds_alternative<ZeroAbsoluteCommand>(cmd) ||
        std::holds_alternative<SetRelativeZeroCommand>(cmd) ||
        std::holds_alternative<ClearRelativeZeroCommand>(cmd) ||
        std::holds_alternative<SetJogVelocityCommand>(cmd) ||
        std::holds_alternative<SetMoveVelocityCommand>(cmd);

    if (!isZeroOrVelocity) {
        // 运动类命令由 Orchestrator 处理，此处不需要额外日志
        return;
    }

    // ⭐ 新增：记录即将下发的命令
    LOG_DEBUG(LogLayer::UI, "AxisVM",
        logPrefix() + " consumePending: " + utils::format(cmd));

    // ... 后续 group/driver 获取和 send
```

**注意**：不记录"分类判断结果是零位类"这种冗余日志。在 send 前记录命令内容即可同时覆盖"有 pending"+"是零位类"+"是什么命令"三个信息点。

### 节点 [H]：`FakePLC::processCommand()` 各命令处理

**源码位置**：`infrastructure/FakePLC.h`

**当前状态**：这是**最严重的日志盲区**。

- `processCommand(StopCommand)` → ❌ 无日志
- `processCommand(EnableCommand)` → ❌ 无日志（急停拦截无日志，状态更新无日志）
- `processCommand(JogCommand)` → ❌ 无日志（急停/龙门拦截无日志）
- `processCommand(MoveCommand)` → ❌ 无日志（急停/龙门拦截无日志）
- `processCommand(ZeroAbsoluteCommand)` → ❌ 无日志
- `processCommand(SetRelativeZeroCommand)` → ❌ 无日志
- `processCommand(ClearRelativeZeroCommand)` → ❌ 无日志
- `processCommand(SetJogVelocityCommand)` → ❌ 无日志
- `processCommand(SetMoveVelocityCommand)` → ❌ 无日志

**改造**：为每个 `processCommand` 重载添加统一的入口日志 + 状态过滤拒绝日志。

**改造模板**（以 `SetRelativeZeroCommand` 为例）：

```cpp
void processCommand(AxisId id, const SetRelativeZeroCommand&) {
    auto& axis = m_axes.at(id);

    LOG_DEBUG(LogLayer::HAL, "PLC",
        "process: axis=" + axisIdToString(id)
        + " cmd=SetRelativeZero"
        + " state=" + std::to_string(static_cast<int>(axis.feedback.state))
        + " absPos=" + std::to_string(axis.feedback.absPos)
        + " currentBase=" + std::to_string(axis.feedback.relZeroAbsPos));

    if (axis.feedback.state == AxisState::Idle ||
        axis.feedback.state == AxisState::Disabled) {
        double oldBase = axis.feedback.relZeroAbsPos;
        axis.feedback.relZeroAbsPos = axis.feedback.absPos;
        LOG_DEBUG(LogLayer::HAL, "PLC",
            "SetRelativeZero executed: axis=" + axisIdToString(id)
            + " base " + std::to_string(oldBase) + " → " + std::to_string(axis.feedback.relZeroAbsPos));
    } else {
        LOG_WARN(LogLayer::HAL, "PLC",
            "SetRelativeZero REJECTED: axis=" + axisIdToString(id)
            + " state=" + std::to_string(static_cast<int>(axis.feedback.state))
            + " (requires Idle or Disabled)");
    }
}
```

**其他 processCommand 的内容差异**：

| 命令 | 入口日志额外字段 | 拒绝条件 | WARN 内容 |
|------|-----------------|---------|----------|
| `EnableCommand` | `active=true/false` | 急停激活时 | `"EMERGENCY STOP ACTIVE — ignoring Enable"` |
| `JogCommand` | `dir=Forward/Backward, active=true/false` | 急停 / 龙门联动 / 状态≠Idle | `"EMERGENCY STOP ACTIVE"` / `"GANTRY COUPLED"` / `"state=Moving (requires Idle)"` |
| `MoveCommand` | `type=Absolute/Relative, target=xxx` | 急停 / 龙门联动 / 状态≠Idle | 同上 |
| `StopCommand` | 当前状态 | 状态不在运动中 | `"Stop ignored: axis not in motion, state=Idle"` |
| `ZeroAbsoluteCommand` | `absPos` | 状态≠Idle且≠Disabled | `"ZeroAbsolute rejected: state=Jogging (requires Idle or Disabled)"` |
| `ClearRelativeZeroCommand` | `absPos, relZeroAbsPos` | 同上 | `"ClearRelativeZero rejected: state=..."` |
| `SetJogVelocityCommand` | `velocity` | 无拒绝 | 无需 WARN |
| `SetMoveVelocityCommand` | `velocity` | 无拒绝 | 无需 WARN |

**关键原则**：
- 命令到达 PLC 时 DEGUB 记录 → 解决了"命令是否到达硬件"的盲区
- 状态不符时 WARN 记录 → 解决了"命令为什么没生效"的盲区
- 执行成功时 DEBUG 记录变化前后的值 → 解决了"PLC 到底改了哪个寄存器"的盲区

### 节点 [I]：`FakePLC::tick()` 反馈生成

**源码位置**：`infrastructure/FakePLC.h::tick()`

**当前状态**：
- 有 `LOG_TRACE_EVERY_N(50, ...)` 记录轴位置 ✅
- `relPos = absPos - relZeroAbsPos` 更新无日志 ❌

**改造**：在 `tick()` 中 `relPos` 更新后补充 TRACE（低频节流）。

```cpp
// 在 relPos 更新后添加：
axis.feedback.relPos = axis.feedback.absPos - axis.feedback.relZeroAbsPos;

// 当 relZeroAbsPos 非零时（相对零点已设置），记录增量变化
// 使用节流避免高频输出
static Throttle relTraceThrottle(50);
if (relTraceThrottle.should() && axis.feedback.relZeroAbsPos != 0.0) {
    LOG_TRACE(LogLayer::HAL, "PLC",
        "tick axis=" + axisIdToString(id)
        + " abs=" + std::to_string(axis.feedback.absPos)
        + " rel=" + std::to_string(axis.feedback.relPos)
        + " base=" + std::to_string(axis.feedback.relZeroAbsPos));
}
```

**注意**：由于 `relPos` 每 tick 都更新，直接用 `LOG_TRACE_EVERY_N` 或内联 throttle 避免日志爆炸。保留每 50 tick（~500ms）一个轴的完整状态快照级别即可。

### 节点 [J]：`FakeAxisDriver::pollFeedback()` 反馈注入

**源码位置**：`infrastructure/FakeAxisDriver.h::pollFeedback()`

**当前状态**：
- 入口无日志 ❌
- 各轴反馈注入无日志 ❌
- 急停反馈注入无日志 ❌
- 龙门反馈注入无日志 ❌

**改造**：添加节流摘要日志（每 50 帧输出一次）。

```cpp
void pollFeedback(SystemContext& ctx) override {
    m_plc.tick(10);

    // ⭐ 新增：节流摘要
    static int frameCount = 0;
    frameCount++;
    if (frameCount % 50 == 0) {
        LOG_TRACE(LogLayer::HAL, "Driver",
            "pollFeedback frame " + std::to_string(frameCount)
            + " estop=" + (m_plc.getEmergencyStopFeedback() ? "ACTIVE" : "off"));
    }

    // 急停注入
    ctx.emergencyStopController().applyFeedback(m_plc.getEmergencyStopFeedback());

    // 龙门注入
    GantryFeedback gf = m_plc.getGantryFeedback();
    ctx.gantryPowerController().applyFeedback(gf);
    ctx.gantryCouplingController().applyFeedback(gf);

    // 轴注入
    constexpr std::array<AxisId, 6> ALL_AXIS_IDS = {
        AxisId::X, AxisId::X1, AxisId::X2, AxisId::Y, AxisId::Z, AxisId::R
    };
    for (auto axisId : ALL_AXIS_IDS) {
        Axis* axis = nullptr;
        ContextRejection r;
        if (ctx.tryGetAxis(axisId, axis, r) && axis) {
            auto fb = m_plc.getFeedback(axisId);
            axis->applyFeedback(fb);

            // ⭐ 新增：每 50 帧输出每个轴的反馈摘要
            if (frameCount % 50 == 0) {
                LOG_TRACE(LogLayer::HAL, "Driver",
                    "inject axis=" + axisIdToString(axisId)
                    + " state=" + std::to_string(static_cast<int>(fb.state))
                    + " abs=" + std::to_string(fb.absPos)
                    + " rel=" + std::to_string(fb.relPos));
            }
        }
    }
}
```

**注意**：这里是每 10ms 调用一次的高频路径，必须使用节流。

### 节点 [K]：`Axis::applyFeedback()` 闭环消费

**源码位置**：`domain/entity/Axis.cpp`

**当前状态**：❌ 完全无日志。这是**排查闭环逻辑故障最关键**的盲区。

**改造**：在每个闭环分支添加 DEBUG 日志，记录条件评估结果和消费行为。

**关键设计决策**：`applyFeedback()` 每 10ms 被调用一次。不应对每个 tick 都打日志。只在**意图被实际消费时**输出（消费是稀疏事件），以及在**条件接近满足但尚未满足**时每 N 帧输出一次（用于诊断"为什么不消费"）。

**具体改造**（按 9 个闭环分支列）：

```cpp
void Axis::applyFeedback(const AxisFeedback& feedback)
{
    // 先更新镜像状态
    m_state = feedback.state;
    m_current_abs_pos = feedback.absPos;
    m_current_rel_pos = feedback.relPos;
    m_rel_zero_abs_pos = feedback.relZeroAbsPos;
    m_pos_limit_active = feedback.posLimit;
    m_neg_limit_active = feedback.negLimit;
    m_pos_limit_value = feedback.posLimitValue;
    m_neg_limit_value = feedback.negLimitValue;
    m_jog_velocity = feedback.getjogVelocity;
    m_move_velocity = feedback.getMoveVelocity;

    // --- 分支 1：限位熔断 ---
    if (m_pos_limit_active || m_neg_limit_active) {
        if (std::holds_alternative<MoveCommand>(m_pending_intent) ||
            std::holds_alternative<JogCommand>(m_pending_intent)) {
            LOG_WARN(LogLayer::DOM, "Axis",
                "applyFeedback: LIMIT MELTDOWN — clearing pending "
                + utils::format(m_pending_intent)
                + " posLimit=" + (m_pos_limit_active ? "ACTIVE" : "off")
                + " negLimit=" + (m_neg_limit_active ? "ACTIVE" : "off"));
            m_pending_intent = std::monostate{};
        }
    }

    // --- 分支 2：运动状态 → 清理 Jog 意图 ---
    if (m_state == AxisState::Jogging ||
        m_state == AxisState::MovingAbsolute ||
        m_state == AxisState::MovingRelative)
    {
        if (std::holds_alternative<JogCommand>(m_pending_intent)) {
            LOG_DEBUG(LogLayer::DOM, "Axis",
                "applyFeedback: consumed JogCommand (PLC now in motion, state="
                + std::to_string(static_cast<int>(m_state)) + ")");
            m_pending_intent = std::monostate{};
        }
    }

    // --- 分支 3：静止状态 → 清理 Stop 意图 ---
    if (m_state == AxisState::Idle ||
        m_state == AxisState::Disabled ||
        m_state == AxisState::Error)
    {
        if (std::holds_alternative<StopCommand>(m_pending_intent)) {
            LOG_DEBUG(LogLayer::DOM, "Axis",
                "applyFeedback: consumed StopCommand (PLC now idle, state="
                + std::to_string(static_cast<int>(m_state)) + ")");
            m_pending_intent = std::monostate{};
        }
    }

    // --- 分支 4：绝对位置清零闭环 ---
    if (std::holds_alternative<ZeroAbsoluteCommand>(m_pending_intent)) {
        if (std::abs(m_current_abs_pos) < POSITION_EPSILON) {
            LOG_DEBUG(LogLayer::DOM, "Axis",
                "applyFeedback: consumed ZeroAbsoluteCommand — absPos="
                + std::to_string(m_current_abs_pos) + " ≈ 0");
            m_pending_intent = std::monostate{};
        }
    }

    // --- 分支 5：设置相对零点闭环 ---
    if (std::holds_alternative<SetRelativeZeroCommand>(m_pending_intent)) {
        bool isRelPosZero = std::abs(m_current_rel_pos) < POSITION_EPSILON;
        bool isBaseMatched = std::abs(m_rel_zero_abs_pos - m_expected_zero_base) < POSITION_EPSILON;

        if (isRelPosZero && isBaseMatched) {
            LOG_DEBUG(LogLayer::DOM, "Axis",
                "applyFeedback: consumed SetRelativeZeroCommand — "
                "relPos=" + std::to_string(m_current_rel_pos)
                + " ≈ 0, base=" + std::to_string(m_rel_zero_abs_pos)
                + " ≈ expected=" + std::to_string(m_expected_zero_base));
            m_pending_intent = std::monostate{};
        }
        // ⭐ 每 100 帧输出一次诊断（帮助排查"为什么不消费"）
        else {
            static int relZeroDiagCounter = 0;
            if (++relZeroDiagCounter % 100 == 0) {
                LOG_TRACE(LogLayer::DOM, "Axis",
                    "applyFeedback: SetRelativeZero NOT consumed — "
                    "relPos=" + std::to_string(m_current_rel_pos)
                    + " (≈0=" + (isRelPosZero ? "T" : "F") + ")"
                    + " base=" + std::to_string(m_rel_zero_abs_pos)
                    + " expected=" + std::to_string(m_expected_zero_base)
                    + " (match=" + (isBaseMatched ? "T" : "F") + ")");
            }
        }
    }

    // --- 分支 6：清除相对零点闭环 ---
    if (std::holds_alternative<ClearRelativeZeroCommand>(m_pending_intent)) {
        bool isRelPosRestored = std::abs(m_current_rel_pos - m_current_abs_pos) < POSITION_EPSILON;
        bool isBaseReset = std::abs(m_rel_zero_abs_pos) < POSITION_EPSILON;

        if (isRelPosRestored && isBaseReset) {
            LOG_DEBUG(LogLayer::DOM, "Axis",
                "applyFeedback: consumed ClearRelativeZeroCommand — "
                "relPos=" + std::to_string(m_current_rel_pos)
                + " absPos=" + std::to_string(m_current_abs_pos)
                + " base=" + std::to_string(m_rel_zero_abs_pos));
            m_pending_intent = std::monostate{};
        }
    }

    // --- 分支 7：Move 定位指令数值收敛 ---
    if (auto* moveCmd = std::get_if<MoveCommand>(&m_pending_intent)) {
        if (m_state == AxisState::Idle) {
            double physicalTarget = (moveCmd->type == MoveType::Absolute)
                ? moveCmd->target
                : (moveCmd->startAbs + moveCmd->target);

            if (std::abs(m_current_abs_pos - physicalTarget) < POSITION_EPSILON) {
                LOG_DEBUG(LogLayer::DOM, "Axis",
                    "applyFeedback: consumed MoveCommand — "
                    "absPos=" + std::to_string(m_current_abs_pos)
                    + " ≈ target=" + std::to_string(physicalTarget));
                m_pending_intent = std::monostate{};
            }
        }
    }

    // --- 分支 8：使能/掉电闭环 ---
    if (auto* cmd = std::get_if<EnableCommand>(&m_pending_intent)) {
        if (cmd->active) {
            if (m_state != AxisState::Disabled && m_state != AxisState::Unknown) {
                LOG_DEBUG(LogLayer::DOM, "Axis",
                    "applyFeedback: consumed EnableCommand(active=true) — "
                    "state=" + std::to_string(static_cast<int>(m_state)));
                m_pending_intent = std::monostate{};
            }
        } else {
            if (m_state == AxisState::Disabled) {
                LOG_DEBUG(LogLayer::DOM, "Axis",
                    "applyFeedback: consumed EnableCommand(active=false) — state=Disabled");
                m_pending_intent = std::monostate{};
            }
        }
    }

    // --- 分支 9：速度设置闭环 ---
    if (auto* cmd = std::get_if<SetJogVelocityCommand>(&m_pending_intent)) {
        if (m_jog_velocity == cmd->velocity) {
            LOG_DEBUG(LogLayer::DOM, "Axis",
                "applyFeedback: consumed SetJogVelocityCommand — v="
                + std::to_string(m_jog_velocity));
            m_pending_intent = std::monostate{};
        }
    }

    if (auto* cmd = std::get_if<SetMoveVelocityCommand>(&m_pending_intent)) {
        if (m_move_velocity == cmd->velocity) {
            LOG_DEBUG(LogLayer::DOM, "Axis",
                "applyFeedback: consumed SetMoveVelocityCommand — v="
                + std::to_string(m_move_velocity));
            m_pending_intent = std::monostate{};
        }
    }
}
```

**注意**：需要在 `Axis.cpp` 中 include `CommandFormatter.h` 以使用 `utils::format()`。

**为什么在 Domain 层引入 `utils::format`（基础设施工具）？**
这是实践中常见的权衡——`Axis.cpp` 属于 domain 实体，理论上不依赖 infrastructure 工具。但日志记录（特别是 `utils::format` 格式化 variant）是一个**横切关注点**，在所有层都需要的调试辅助。两种方案：

- **方案 A（推荐）**：直接在 `Axis.cpp` include `infrastructure/utils/CommandFormatter.h`。`utils::format` 只依赖 Axis.h 中的类型定义，不引入反向依赖。
- **方案 B**：仅为 `Axis` 添加一个不依赖基础设施的 `AxisCommand` 到字符串的简单转换函数。但会产生重复代码。

选择方案 A，因为日志本身已跨越分层边界（Logger 是基础设施）。

### 全局：TraceScope 贯穿

**当前状态**：
- ViewModel 层每个控制方法入口创建 `TraceScope` ✅
- Logger 会自动读取当前 `TraceScope::current()` 并携带 `group/axis/traceId` ✅
- 但 scope 仅在函数作用域内有效，不会自动传递到 Domain/Driver/PLC 的后续调用 ❌

**分析**：TraceScope 的 thread-local 设计意味着：
- 如果 ViewModel → Domain → Driver → PLC 的调用发生在**同一线程同步调用栈**内，TraceScope 自然贯穿
- 但本项目架构中：
  - ViewModel `setRelativeZero()` → 立即调用 `axis->setRelativeZero()` → ← **同步**，scope 有效
  - `tick()` → `consumePendingCommands()` → `drv->send()` → **同步**，但此时的 scope 是 `tick()` 的，不是原始 `setRelativeZero` 的 ❌
  - `drv->send()` → FakePLC 的调用发生在 Driver 线程 → scope 丢失 ❌

**结论**：由于项目的**命令挂起-消费异步模型**（用户操作产生 pending command，tick 中消费），TraceScope 无法自动贯穿。需要在 pending command 中携带 traceId。

**改造方案**：

在 `AxisCommand` variant 中不直接加 traceId（会污染领域模型）。取而代之的是，在 `consumePendingCommands()` 中手动创建新的 TraceScope：

```cpp
void AxisViewModelCore::consumePendingCommands()
{
    // ...
    // 在下发命令前创建 scope：
    TraceScope scope(m_groupName, axisIdToString(m_axisId), generateTraceId());

    auto commResult = drv->send(AxisCommandWithId{m_axisId, cmd});
    // logger 在此 scope 内会携带 traceId
}
```

这样，`consumePendingCommands` → `drv->send` → `FakeAxisDriver::handle` → `FakePLC::processCommand` 在同一调用栈内，TraceScope 自然贯穿。

**对于多线程场景**（如果将来 Driver 异步），traceId 需要编码到命令数据结构中，但这不在本次整改范围内。

## 五、日志级别使用规范

| 级别 | 适用场景 | 示例 |
|------|---------|------|
| `SUMMARY` | 每 N 帧输出的系统整体健康状态（尚未使用，建议引入） | `"heartbeat: all axes OK, estop=off, position=..."` |
| `ERROR` | 不可恢复的错误：通讯失败、硬件异常、命令下发失败 | `"command delivery failed: NetworkError"` |
| `WARN` | 可恢复的异常：命令被静默拒绝、限位触发、状态不符 | `"SetRelativeZero REJECTED: state=Jogging"` |
| `INFO` | 用户可见操作：按钮点击、命令请求、模式切换 | `"jog Forward pressed"` |
| `DEBUG` | 关键分支决策：准入通过/拒绝原因、闭环消费、值变化 | `"SetRelativeZero executed: base 0.0 → 64.4"` |
| `TRACE` | 高频细节：每帧反馈注入、运动学推演、位置增量 | `"tick axis=Y abs=64.4 rel=0.0 base=64.4"` |

**生产环境**（`minConsoleLevel=INFO`）：
- 可见：SUMMARY / ERROR / WARN / INFO
- 隐藏：DEBUG / TRACE

**调试环境**（`minConsoleLevel=DEBUG` 或 `TRACE`）：
- 全部可见

## 六、改造注意事项

### 6.1 不要破坏现有测试

所有日志宏仅输出到日志系统，不改变任何控制流。不需要修改测试用例。

### 6.2 性能考虑

| 高频路径 | 频率 | 日志策略 |
|---------|------|---------|
| `FakePLC::tick()` | 每 10ms | 使用 `LOG_TRACE_EVERY_N(50)` 节流 |
| `FakeAxisDriver::pollFeedback()` | 每 10ms | 使用 `frameCount % 50` 节流 |
| `Axis::applyFeedback()` | 每 10ms | 仅在**意图消费时**输出（稀疏事件），诊断用每 100 帧节流 |

异步日志队列确保日志输出不会阻塞主循环，即使 TRACE 全开也不会影响系统实时性（仅增加 CPU 和内存开销）。

### 6.3 不引入新的依赖

所有新增日志使用现有的 `Logger.h` 宏体系和 `CommandFormatter.h`，不新增任何头文件依赖。唯一需要的是在 `Axis.cpp` 顶部增加一行：

```cpp
#include "infrastructure/utils/CommandFormatter.h"
```

### 6.4 FakePLC 头文件规模控制

`FakePLC.h` 目前约 350 行。新增 processCommand 日志后预计增加 60~80 行（每个重载 5~8 行），仍可维护。如果未来日志行数膨胀，可考虑：

- 将 `processCommand` 实现从 `.h` 分离到 `FakePLC.cpp`（可选重构，非本次必需）

## 七、总结

### 当前问题根因

日志系统的所有基础设施（Logger、LogContext、TraceScope、CommandFormatter、节流宏）都已就位，但**使用严重不充分**：

| 层 | 当前日志覆盖 | 缺失 |
|----|------------|------|
| ViewModel | ✅ 入口 INFO + 拒绝 WARN | consumePending 分类和下发 |
| Domain | ❌ 零日志 | 准入决策 + applyFeedback 闭环消费（核心盲区） |
| Driver | 🟡 仅 TRACE | handle 入口 + pollFeedback 反馈注入 |
| PLC | 🟡 仅节流 TRACE | **全部** processCommand（最严重盲区） |

### 改造后的效果

将日志级别切换到 `DEBUG` 后，一条命令的完整生命周期可被**一行不漏地追踪**：

```
[17:23:45.110][INFO][UI][AxisVM][Grp1][Y][a3f2] setRelativeZero requested
[17:23:45.110][DEBUG][DOM][Axis][Grp1][Y][a3f2] setRelativeZero accepted: state=Disabled absPos=64.4 expectedZeroBase=64.4
[17:23:45.120][DEBUG][UI][AxisVM][Grp1][Y][b7e1] consumePending: SetRelativeZeroCommand
[17:23:45.121][DEBUG][HAL][Driver][Grp1][Y][b7e1] send to PLC: axis=Y cmd=SetRelativeZeroCommand
[17:23:45.121][DEBUG][HAL][PLC][Grp1][Y][b7e1] process: axis=Y cmd=SetRelativeZero state=2(Disabled) absPos=64.4 currentBase=0.0
```

**核心优势**：
1. **不再需要阅读源码来排查问题** — 日志自带完整的决策路径

3. **一键切换调试** — 将 `minConsoleLevel` 改为 `DEBUG` 即可获得完整追踪
4. **分层清晰** — UI/DOM/HAL 标签可快速过滤到问题所在的层
5. **零破坏性** — 不改变任何控制流，不影响测试，不降低性能

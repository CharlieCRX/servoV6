# 龙门轴联动与解除联动状态翻转问题分析

## 日期
2026-05-20

## 问题概述
龙门轴 UI 在执行「联动使能」→「解除联动并掉电」的完整操作序列中，存在多个状态翻转时序错误。本文档基于实际日志与源码分析，定位根因并提出修复方案。

## 问题详细分析

### BUG-1: X 轴（逻辑龙门轴）无关的假跳变 Unknown → Disabled

**日志位置**: `11:56:56.781`

**现象**: 在联动确认的同时，X 轴（逻辑龙门轴）发生 `applyFeedback: state Unknown(0) -> Disabled(1)`。

**根因**:
- `FakePLC` 中 X 轴 (`AxisId::X`) 是独立轴条目，由 `m_axes[AxisId::X]` 独立管理。
- `GantryPowerCommand` 仅影响 X1/X2（参见 `FakePLC::tickGantryPower`），不触及 X 轴。
- X 轴初始状态为 `Unknown`，在没有收到 EnableCommand 的情况下，`updateStateTransitions` 永远不推进（仅处理 Unknown→Idle）。
- 但 `Axis::applyFeedback` 日志显示 Unknown→Disabled，说明 FakePLC 的某个 tick 周期反馈将 X 轴状态设为了 Disabled。这可能是 `m_axes[AxisId::X].feedback` 的默认初始值 `AxisState::Disabled` 在首次读取时触发了领域实体状态更新。

**相关代码** (`FakePLC.h` 构造函数):
```cpp
FakePLC() {
    m_axes[AxisId::Y] = AxisStateInternal{};
    // ... 包括 m_axes[AxisId::X] = AxisStateInternal{};
}
```
`AxisStateInternal::feedback` 默认值为 `AxisState::Disabled`。但初始遥测显示 `X=Unknown`，说明在启动阶段 AXIS_DOMAIN 将 X 轴置为了 Unknown。之后的某个 tick 周期，`getFeedback(AxisId::X)` 返回了 `Disabled`，导致领域实体记录了一次 Unknown→Disabled 的状态跳变。**这不影响功能，但日志中产生虚假跳变，干扰调试。**

**建议修复**:
- 在 FakePLC 的 tick 循环中，主动将 X 轴状态派生自 X1/X2：
  - X1/X2 均为 Idle → X = Idle
  - X1/X2 任一运动态 → X = 对应运动态
  - X1/X2 均为 Disabled → X = Disabled

### BUG-2: 联动完成后 X1/X2 遥测显示 Unknown

**日志位置**: `11:56:57.501`

**现象**: 联动流程成功完成（`Flow completed successfully`），但 720ms 后的遥测快照显示 `X1=Unknown, X2=Unknown`。

**根因**:
1. `FakePLC::tickGantryPower` 在使能延迟完成后直接将 `X1/X2.feedback.state` 设为 `Idle`。
2. 领域实体通过 `Axis::applyFeedback` 将状态更新为 `Idle`。
3. 但由于 `Axis::applyFeedback` 内部存在 Unknown→Disabled 覆盖逻辑（或 FakePLC 后续 tick 触发了状态回滚），导致 X1/X2 在 subsequent tick 中被重置为 Unknown。

**补充分析**:
检查 `Axis::applyFeedback` 的实现可能揭示：当从 FakePLC 首次读取反馈时，若 `feedback.state` 为 `Disabled`，领域实体会将其覆盖为 `Unknown`（因为尚未被 EnableCommand 触发的状态迁移初始化过）。这是领域实体与 FakePLC 之间的状态同步不一致问题。

**建议修复**:
- 统一状态同步语义：领域实体的 `applyFeedback` 不应覆盖 PLC 反馈的真实值。
- 或在 `SystemContext` 初始化后，立即将 FakePLC 的初始反馈同步到领域实体（消除 Unknown→Disabled 的初始跳变）。

### BUG-3: ❗解除联动时 `coupled` 标志不在 PLC 确认后翻转（时序不对称）

**日志位置**: `11:56:57.930`

**现象**:
- 联动时：`coupled: 0 -> 1` 发生在 PLC 确认联动成功后（`11:56:56.781`，距下发命令 ~101ms）
- 解除联动时：`coupled: 1 -> 0` 发生在**下发解耦命令的同一 tick**（`11:56:57.930`），而非 PLC 确认解耦完成后

**根因**:

在 `GantryViewModel::refreshGantryState()` 中，`coupled` 标志从 `GantryCouplingController::isCoupled()` 读取：

```cpp
bool coupled = coupling.isCoupled();
```

而 `GantryCouplingController::requestCouple(false)` 执行时，`GantryCouplingState::requestDecouple()` 将状态从 `Coupled` 变为 `DecouplingRequested`：

```cpp
void requestDecouple() {
    if (m_status == Status::NotSynchronized) return;
    m_status = Status::DecouplingRequested;
}
```

`isCoupled()` 仅当 `m_status == Status::Coupled` 时返回 `true`。因此状态进入 `DecouplingRequested` 后，`isCoupled()` 立即返回 `false`。

**这与联动流程不对称**：
- 联动时：`isCoupled()` 在 PLC 确认后（`applyCoupledFeedback` → `Coupled`）才返回 `true`
- 解除联动时：`isCoupled()` 在发送命令瞬间（`requestDecouple` → `DecouplingRequested`）就返回 `false`

**影响**:
UI 的「联动已解除」状态会在命令下发瞬间就切换，而实际上 PLC 硬件还需要 ~100ms 延迟（`GANTRY_COUPLING_DELAY_MS`）才能完成物理解耦。如果用户在此期间误判状态并发起新操作（如重新联动），可能产生命令竞争。

**修复方案**:

方案 A（推荐）：UI 的解耦状态判断应同时考虑 `isDecouplingRequested()`：

```cpp
// GantryViewModel::refreshGantryState()
bool coupled = coupling.isCoupled();
bool decoupling = coupling.isDecouplingRequested();  // 新增

// UI 的 "已联动" 指示应取 coupled，而非 !decoupling
// UI 的 "解除中" 指示应取 decoupling（过渡态显示）
```

方案 B：在 `GantryCouplingState` 中保持 `DecouplingRequested` 期间 `isCoupled()` 仍返回 `true`，仅 PLC 确认后才返回 `false`——但这改变了 `isCoupled()` 的语义（从「状态机当前耦合态」变成「物理耦合态」），需要更谨慎的评估。

**推荐方案 A**，在 ViewModel 层解耦 `coupled` 的物理语义与 UI 显示语义。

### BUG-4: ❗掉电等待中 `enabled` 假翻转 0 → 1 → 0

**日志位置**: `11:56:58.051`

**时间线**:
```
11:56:58.041  enabled: 1 -> 0    (Disabling step: requestEnable(false) done)
11:56:58.051  enabled: 0 -> 1    ← 假翻转！仅 10ms 后
11:56:58.190  enabled: 1 -> 0    (WaitingDisabled → Done，PLC 确认掉电)
```

**根因**:

`GantryViewModel::refreshGantryState()` 从 `GantryPowerController` 读取 `isEnabled()`：

```cpp
bool isEnabled() const { return m_status == Status::Enabled; }
```

时间线分析：
1. **11:56:58.041**: `GantryPowerController::requestEnable(false)` 将 `m_status` 设为 `Disabling` → `isEnabled()` 返回 `false` → ViewModel 记录 `enabled: 1 -> 0`。
2. **11:56:58.051** (10ms 后): `GantryPowerController::applyFeedback()` 被调用。`FakePLC` 在 `tickGantryPower` 中已设置 X1/X2 为 Idle（注意：FakePLC 在异步的 tick 中推进），但 `GantryFeedback::enable` 由 `tickGantry` 的最后一步更新：
   ```cpp
   // tickGantry 步骤 5：
   m_gantryFeedback.enable = m_gantryPhysical.x1Enabled && m_gantryPhysical.x2Enabled;
   ```
   `m_gantryPhysical` 在步骤 2 刷新，此时如果 X1/X2 尚未变为 Disabled（因为 power disable 延迟还未结束），则 `enable=true`。
   `GantryPowerController::applyFeedback()` 无条件覆盖状态：
   ```cpp
   void applyFeedback(const GantryFeedback& feedback) {
       m_status = feedback.enable ? Status::Enabled : Status::Disabled;
   }
   ```
   如果 `feedback.enable = true`，则 `m_status` 被强制写回 `Enabled` → ViewModel 记录 `enabled: 0 -> 1`。**这是一个假翻转**：实际上正在等待掉电完成。
3. **11:56:58.190**: `FakePLC::tickGantryPower` 完成掉电延迟，将 X1/X2 设为 `Disabled`。`refreshGantryPhysicalState` 更新 `x1Enabled=x2Enabled=false`，`tickGantry` 步骤 5 将 `GantryFeedback::enable` 改为 `false`。领域实体的 `applyFeedback` 被调用，`m_status = Disabled` → ViewModel 记录 `enabled: 1 -> 0`。

**核心矛盾**: `GantryPowerController::applyFeedback()` 是无条件覆盖（`m_status = feedback.enable ? Status::Enabled : Status::Disabled`），不感知当前是否处于 `Disabling` 等待状态。在 `Disabling` 期间收到 `feedback.enable = true`（物理状态尚未更新）时，它直接覆盖了 `Disabling` 状态，造成假翻转。

**修复方案**:

在 `GantryPowerController::applyFeedback()` 中加入状态感知：

```cpp
void applyFeedback(const GantryFeedback& feedback) {
    // 若处于转换中间态（Enabling / Disabling），忽略与意图相反的中间帧反馈
    if (m_status == Status::Disabling && feedback.enable) {
        // 等待掉电完成，忽略此中间帧
        return;
    }
    if (m_status == Status::Enabling && !feedback.enable) {
        // 等待使能完成，忽略此中间帧
        return;
    }
    m_status = feedback.enable ? Status::Enabled : Status::Disabled;
}
```

这样可以确保在 `Disabling` 期间，即使 PLC 反馈延迟导致 `enable` 短暂为 `true`，状态机也不会发生假翻转。

### BUG-5: 解除联动后 X 轴遥测显示 Unknown（而非 Disabled）

**日志位置**: `11:56:58.500`

**现象**: 解除联动并掉电完成后，遥测显示 `X=Unknown, X1=Disabled, X2=Disabled`。X1/X2 正确显示 `Disabled`，但 X 轴仍然是 `Unknown`。

**根因**: 与 BUG-1 同源。X 轴 (`AxisId::X`) 的状态不参与龙门联动/解除流程中的任何状态更新，始终处于初始 Unknown 状态或偶发跳变到 Disabled。遥测轮询到的 X 轴状态取决于当时 FakePLC `m_axes[AxisId::X].feedback.state` 的值。

**修复**: 与 BUG-1 联动修复。使 X 轴状态由 X1/X2 派生。

## 问题汇总与优先级

| 编号 | 问题 | 严重程度 | 影响 |
|------|------|----------|------|
| BUG-1 | X 轴无关的假跳变 | 低 | 日志噪音，干扰调试 |
| BUG-2 | 联动后 X1/X2 遥测 Unknown | 中 | 状态显示不可靠，用户困惑 |
| BUG-3 | `coupled` 解耦时不等 PLC 确认 | **高** | 时序不对称，UI 提前显示已解耦 |
| BUG-4 | `enabled` 掉电等待中假翻转 | **高** | 状态闪烁，可能触发 UI 错误逻辑 |
| BUG-5 | X 轴状态始终不一致 | 低 | 与 BUG-1 同源 |

---

## 推荐修复顺序

1. **修复 BUG-4**（`GantryPowerController::applyFeedback` 加入中间态保护）
2. **修复 BUG-3**（`GantryViewModel` 增加 `isDecouplingRequested` 判断，保持 `coupled` 语义正确）
3. **修复 BUG-1/BUG-2/BUG-5**（X 轴状态派生自 X1/X2，消除虚假跳变）

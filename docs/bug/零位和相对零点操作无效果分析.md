# 零位操作（zeroAbsolute/setRelativeZero/clearRelativeZero）无效果分析

## 问题现象

在点动完成（轴处于 `Disabled` 状态）后：
1. `zeroAbsolutePosition()` 请求 → 无错误日志，但绝对位置未清零
2. `setRelativeZero()` 请求 → 无错误日志，但相对位置不变
3. `clearRelativeZero()` 请求 → 无错误日志，但相对位置不变

日志确认轴始终处于 `Disabled` 状态，位置无任何变化：

```
[17:02:36.103][INFO][UI][AxisVM][Machine_A][Y][...] Machine_A/Y zeroAbsolutePosition requested
[17:02:37.183][INFO][UI][AxisVM][Machine_A][Y][...] Machine_A/Y setRelativeZero requested
[17:02:38.307][INFO][UI][AxisVM][Machine_A][Y][...] Machine_A/Y clearRelativeZero requested
[17:02:38.659][SUMMARY][UI][Telemetry][N/A][N/A][N/A] === Machine_A === Y: pos=+64.4 Disabled
```

## 完整调用链分析

### 成功经过的关卡

```
用户点击按钮
  → AxisViewModelCore::setRelativeZero()      ← 正常调用
    → tryGetAxis() → 轴查找成功
    → axis->setRelativeZero()                 ← ✅ 已通过（Domain 已允许 Disabled 状态）
    → hasPendingCommand() = true
    → pushError() 未触发（因为 setRelativeZero 返回 true）
```

### 关键路径：consumePendingCommands() + FakePLC

命令的最终下发路径在 `AxisViewModelCore::tick()` 中：

```
void AxisViewModelCore::tick()
{
    m_jogOrch->tick();
    m_absOrch->tick();
    m_relOrch->tick();
    collectOrchError(...);

    consumePendingCommands();  // ← 核心方法：下发零位/速度类命令
}
```

`consumePendingCommands()`：
1. 检查 `axis->hasPendingCommand()` → true（domain 已经接受了 SetRelativeZeroCommand）
2. 检查命令类型是零位/速度类 → 是
3. 获取 driver → 成功
4. 调用 `drv->send(AxisCommandWithId{m_axisId, cmd})` → 成功（无 CommError）
5. FakeAxisDriver 转发到 FakePLC::onCommand()
6. FakePLC 内部调用 `processCommand(id, SetRelativeZeroCommand{})` → ❌ **静默忽略！**

### 根因：FakePLC 层的 Idle-only 状态限制

`FakePLC::processCommand()` 中的零位/相对零点相关方法都有一个 `Idle` 状态检查：

```cpp
void processCommand(AxisId id, const ZeroAbsoluteCommand&) {
    auto& axis = m_axes.at(id);
    if (axis.feedback.state == AxisState::Idle) axis.feedback.absPos = 0.0;
    //   ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ 问题在这里！
    //   轴当前为 Disabled，条件不满足，命令被静默忽略
}

void processCommand(AxisId id, const SetRelativeZeroCommand&) {
    auto& axis = m_axes.at(id);
    if (axis.feedback.state == AxisState::Idle) axis.feedback.relZeroAbsPos = axis.feedback.absPos;
    //   ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ 同样问题
}

void processCommand(AxisId id, const ClearRelativeZeroCommand&) {
    auto& axis = m_axes.at(id);
    if (axis.feedback.state == AxisState::Idle) axis.feedback.relZeroAbsPos = 0.0;
    //   ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ 同样问题
}
```

Domain 层已经放宽了限制，但 FakePLC 仿真层未同步修改。

│  → tick() 中 applyFeedback → pending_command 始终不消费     │

└──────────────────────────────────────────────────────────────┘

## ViewModel tick 驱动详解

`consumePendingCommands()` 方法向 FakePLC 发送命令后，FakePLC 内部进行了操作但条件不满足被跳过。随后下一个 `pollFeedback() → feedback.applyFeedback()` 会刷新 Axis domain 的状态。

`applyFeedback()` 中的闭环消费逻辑：

```cpp
// 4. 设置相对原点闭环
if (std::holds_alternative<SetRelativeZeroCommand>(m_pending_intent)) {
    bool isRelPosZero = std::abs(m_current_rel_pos) < POSITION_EPSILON;
    bool isBaseMatched = std::abs(m_rel_zero_abs_pos - m_expected_zero_base) < POSITION_EPSILON;
    if (isRelPosZero && isBaseMatched) {
        m_pending_intent = std::monostate{}; // 双重达标，消费意图
    }
}
```

由于 FakePLC 未更新 `relPos` 和 `relZeroAbsPos`，这两项条件永远不满足，**pending_command 永远不会被消费**。

## 修复方案

### 方案 A（推荐）：同步放宽 FakePLC 状态限制

将 FakePLC 的三个零位命令处理方法放宽，允许 `Idle` 和 `Disabled` 状态：

```cpp
void processCommand(AxisId id, const ZeroAbsoluteCommand&) {
    auto& axis = m_axes.at(id);
    if (axis.feedback.state == AxisState::Idle || 
        axis.feedback.state == AxisState::Disabled)
        axis.feedback.absPos = 0.0;
}

void processCommand(AxisId id, const SetRelativeZeroCommand&) {
    auto& axis = m_axes.at(id);
    if (axis.feedback.state == AxisState::Idle ||
        axis.feedback.state == AxisState::Disabled)
        axis.feedback.relZeroAbsPos = axis.feedback.absPos;
}

void processCommand(AxisId id, const ClearRelativeZeroCommand&) {
    auto& axis = m_axes.at(id);
    if (axis.feedback.state == AxisState::Idle ||
        axis.feedback.state == AxisState::Disabled)
        axis.feedback.relZeroAbsPos = 0.0;
}
```

**影响和风险：**
- 只影响 `FakePLC.h`，Domain 层代码不动
- 只影响零位/速度类纯坐标操作，不影响运动逻辑
- 使得 FakePLC 仿真行为与 Domain 层策略一致
- 需要同步更新 FakePLC 的单元测试

### 方案 B：仅在 Domain 层闭环，不依赖 FakePLC 写回

修改 `Axis::applyFeedback()` 中的闭环逻辑，对于零位/相对零点指令，在 Axis 自身闭环时不依赖 FakePLC 的状态变化，而是直接在 Domain 层本地闭环。

**优点：** 不依赖 FakePLC 的配合
**缺点：** 绕过了物理仿真，Domain 与 FakePLC 的反馈路径不一致

### 推荐：方案 A

零位操作是纯坐标操作，不影响物理运动。在真实 PLC/伺服系统中，即使轴掉电（`Disabled`），操作面板上依然可以设置/清除相对零点。FakePLC 的状态拦截纯粹是仿真层面的保守设计，与 Domain 一致放宽是合理的。
